Modern systems & why this question exists

FROM LECTURES — PA165 Intro

Modern information systems are typically large software systems based on multiple components that need to co-evolve. The complexity of modern info systems makes them similar to complex systems in other domains (e.g. biology, social networks). Can a single developer understand the whole system?

Challenges of modern system design:

  • Need an advanced orchestration platform (Kubernetes / OpenShift) to schedule resources, deploy containers, auto-scale. Manual would be impossible.
  • Single components are now distributed and must find each other and communicate → need additional pieces like service proxies and meshes.
  • The more connected components, the less reliable the system — cascading failures can happen. Need retries / circuit breakers.
  • Latency may grow with many-service communication.
  • Observability / monitoring is required to understand call flow.
  • Need ways to represent and track changes to the software architecture.
  • Management of the whole development process (DevOps, CI/CD).
FROM LECTURES — Lehman's Laws of Software Evolution
LawWhat it says
Continuing changeSystems must continuously change or become less useful.
Increasing complexityChanges lead to more complex structure; activities are needed to reduce complexity over time.
Large program evolutionAttributes such as size, errors are invariant for each system release.
Organisational stabilityRate of development is approximately constant over the lifecycle.
Conservation of familiarityIncremental changes in each release are constant over the lifecycle.
Continuing growthFunctionality must increase to maintain user satisfaction.
Declining qualityQuality declines over time if the system is not adapted to operational changes.
Feedback systemEvolution should be considered as a multi-loop feedback system.

These laws are the academic underpinning for why we need CI/CD, monitoring, modular architectures, and lifecycle management at all.

INTUITION

Everything in this question (tools, architectures, CI/CD, ORM) is engineering's answer to Lehman's laws. Each tool exists to fight at least one law: refactoring fights increasing complexity; CI/CD fights declining quality; orchestration fights organisational instability; ORM fights the impedance mismatch that adds complexity on every schema change.

Architectures from an implementation & maintenance perspective

FROM REPO — Layered (clean)

Splits a monolithic system into layers; each layer is responsible for some part of the application; a layer uses the services of the layer below. Layers can be open or closed; open layers can be skipped.

  • Presentation — closed, UI / client.
  • Business — closed, business logic.
  • Service — open, shared components of the business layer (auth, logging).
  • Persistence — closed, DB access; the Repository pattern lives here.
  • Database — closed, pure DB layer.

Pros: easy to develop (still monolith), cheap, fits short-budget projects, good when you're not sure what to use. Cons: extra overhead passing data through layers; not crash-tolerant (any part down = whole down); relatively slow startup. A layer can be deployed separately to improve scalability. e.g. e-shop.

FROM REPO — Other architectures
  • MVC / MVP / MVVM — separate model, view, and the mediating component for clearer flow. MVC is common for multi-page web; MVP suits SPAs; MVVM is typical for Android.
  • Client-Server — client is the UI; server processes requests, runs app logic, accesses DB. Communication always initiated by client. Can be tiered (2-tier, 3-tier with DB, more service tiers).
  • Peer-to-Peer — every client is also a server, communicating directly. BitTorrent.
  • Pipeline (Pipes & Filters) — monolithic; works like MapReduce. Pipe = point-to-point comms; Filter = a stateless transformer (can still write to DB). Reusable; suits batch processing (bad for fast response). Unix terminal, compilers.
  • Microkernel / Hexagonal / Component-based — monolithic core + plug-in components. Easy to extend, adapt, customise. Core has minimal functionality. Plug-ins should be independent, runtime-pluggable, possibly with their own DB. Discovery via registry. VS Code (core = text editor, plug-ins = extensions).
  • SOA — hybrid of monolith and microservices. Separate UI(s), separately deployed services (typically 4–12), each focused on one task. Often shares DB. Facade layer for unified API. ACID transactions (microservices use BASE). Good for DDD without too much complexity; good when you need ACID.
  • Microservices — high independence; each service in its own language is possible; each service does the minimum; no shared DB; data duplication is acceptable for lower coupling.
FROM LECTURES — Monolith vs Microservices (PA165 Lect06)

Monolith

Common development model (past 20+ years). Single or small number of deployments. Application server. Single process. Components tightly coupled.

Advantages: Development model matches app requirements (CRUD/MVC: Presentation — Business — Database). Single deployment archive. Easy horizontal scaling (run N copies).

Problems: Adding functionality is hard. General maintenance / CVE patching. Every change requires rebuild + redeploy of the whole app. Replicated server instances take more resources. Slow startup. Often extremely large codebases — hard to understand/maintain, long onboarding, experts per part of the system, often a commitment to a particular technology (or version).

Microservices

A system as a collection of small, isolated services. Each service: owns its data, independently isolated, scalable, resilient, self-maintained. An evolution of SOA. Intuitive approach.

Interaction with other microservices is only through predefined communication interfaces (API).

Organisation: teams segregated according to the architecture; each team responsible for one or a small set of services; structured around business goals; teams should be self-maintained (dev, test, FE…); one team should not have access/knowledge of other teams' services.

Isolation: ownership of resources; data only via the API; the service must act as an external component; loosely coupled; virtual addresses; scaling, load-balancing; resiliency.

Underlying technology may differ — different languages, runtimes, frameworks, versions per service.

Law of Demeter: microservice must communicate with others; principle of least knowledge — each unit knows only "close" units; only talk to your friends.

Single responsibility: "micro" doesn't necessarily mean physically small — it means scope of responsibility. SRP — a microservice should have only one reason to change. Unix philosophy.

API options: HTTP & REST, Apache Kafka, gRPC, GraphQL, AMQP, MQTT, JMS…

FROM LECTURES — Spring Framework / IoC / DI (PA165 Lect02)

Spring is an application framework on top of the Java language (and not only) that provides a dependency injection container for the Java platform. Originally by Rod Johnson (2002). Provides a comprehensive programming and configuration model for modern Java-based enterprise applications.

Inversion of Control (IoC) / Dependency Injection (DI)

Spring container uses the IoC design pattern (a.k.a. Dependency Injection):

  • Objects define their dependencies (e.g. via constructor arguments).
  • The container injects those dependencies when it creates beans.

Three ways to configure metadata:

  • XML — pure declarative; can change config without code; but verbose, hard to maintain, refactoring problems.
  • Annotations — @Component, @Service, @Repository, @Controller, @RestController, @Autowired, @Required; also JSR-330 @Named, @Inject, @Qualifier; @Resource, @PersistenceContext, @PersistenceUnit.
  • Java Config — @Configuration, @Bean, @ComponentScan, @PropertySource, @Import; each @Bean method is evaluated just once.

Aspect-Oriented Programming (AOP)

Spring AOP provides cross-cutting concerns (logging, transactions, security) declaratively. You define an aspect once (e.g. log entry/exit of all @Service methods) without modifying each method.

ADDED — broader architectural patterns to know
Hexagonal / Ports & Adapters (Alistair Cockburn)
Domain core has no dependencies on infrastructure. The core exposes ports (interfaces); concrete infrastructure (DB, REST, queue) is connected via adapters that implement the ports. Lets you swap the DB or the HTTP framework without touching the domain. Same principle as Clean Architecture (Uncle Bob), Onion Architecture — different names, same idea.
CQRS (Command Query Responsibility Segregation)
Split the model used for reads from the model used for writes. Reads can be denormalised and fast; writes can be validated and consistent. Often paired with event sourcing.
Event-driven architecture
Components communicate by publishing/subscribing to events on a broker (Kafka, RabbitMQ, EventBridge). Producers don't know consumers; the broker decouples. Very flexible but harder to reason about.
Serverless / FaaS
Code runs in short-lived functions (AWS Lambda, Azure Functions). Scaling and infrastructure managed by the platform. Best for spiky workloads and glue code.
Modular monolith
Single deploy, but internally organised as decoupled modules with clear interfaces. Easier than microservices, retains most architectural benefits. A pragmatic choice when team size doesn't justify microservice overhead.

Choosing: monolith for <10 devs and simple domain; modular monolith / SOA for medium org; microservices when team count, deploy frequency, or per-service scaling demand it. Don't start with microservices.

CI/CD technologies

FROM LECTURES — Why CI/CD (PA165 Lect09)
  • Releasing more frequently turns out to produce more stable software.
  • Solves the "it works on my machine" problem.
  • Automates testing and deployment of the software.

Responsibility for CI/CD: DevOps — "DevOps combines development (Dev) and operations (Ops) to increase the efficiency, speed, and security of software development and delivery compared to traditional processes." (GitLab)

FROM LECTURES — DevOps activities
  1. Manage — manage and configure the infrastructure required to support your applications.
  2. Plan — organize the work that needs to be done, prioritize it, track completion.
  3. Create — write, design, develop and securely manage code and project data.
  4. Verify — ensure that code works correctly and adheres to quality standards — ideally with automated testing.
  5. Package — package applications and dependencies, manage containers, build artifacts.
  6. Release — deploy the software to end users.
  7. Monitor — track performance metrics and errors to reduce severity and frequency of incidents.
  8. Secure — check for vulnerabilities through static and dynamic tests, fuzz testing, and dependency scanning.
  9. Govern — manage security vulnerabilities, policies, and compliance across the organization.

DevOps vs CI/CD: CI/CD is a set of development practices that enable rapid and reliable delivery of code changes. DevOps is a collection of ideas, practices, processes, and technologies that allow development and operations to work together to streamline product development.

FROM LECTURES — The three "CD"s
Continuous Integration (CI)
A process that provides rapid feedback on the consistency and quality of code to all team members. Integrate code in the main branch as fast as possible. Automate code integration and unit testing. Unit tests play a key role (state testing & interaction testing).
Continuous Delivery (CD)
Automation of the software-delivery process. Deployment pipelines automate testing of deployed artefacts. Integration tests play a key role (verify end-to-end behaviour). Every action up to deployment to production.
Continuous Deployment (CD)
Automation of the process that deploys the application after tests and CI/CD have passed. Manages the process of releasing the system to different users. Very strict checks about what can be released — it goes into production. Companies may still prefer "manual" steps for the final deploy step.
FROM REPO — short definitions
  • Continuous integration — we regularly build the resulting system (with small, simple changes), and automatically test the result.
  • Continuous delivery — we regularly build the system so it could be released immediately, and automatically test the result.
  • Continuous deployment — we regularly deliver, automatically test the result.
  • Regression tests — run after changes to detect unwanted effects.
FROM LECTURES — Pull Requests workflow
  1. Developer creates a task / feature branch.
  2. Once done with coding, developer pushes the dedicated local branch to the public repository.
  3. Developer creates a pull request.
  4. Team members review the code and provide feedback.
  5. Once approved, the project maintainer merges the changes to the target branch.
FROM LECTURES — Branching strategies
Centralized workflow
Main branch only. Easier transition from SVN. Users don't need to change their workflow.
Feature Branch workflow
Feature development isolated in feature branches; pull request for discussing the changes. Master branch always holds stable code.
Forking workflow
Common in open source. Forks of the repo are created. Each user has a personal public repo. Maintainer integrates via pull requests.
Gitflow workflow
Strict branching model around releases. master stores official release history; develop serves as the integration branch for features. Plus release / hotfix branches.
FROM LECTURES — Pipeline anatomy

"A pipeline drives software development through building, testing and deploying code in stages. Pipelines are comprised of jobs, which define what will be done, and stages that spell out when to run the jobs."

  • Runner — an agent/server that executes each job individually.
  • Stages — keyword that defines stages of a job (build, test, deploy). Jobs in the same stage are executed in parallel.
  • Job — instruction that a runner has to execute.

GitLab pipelines are defined in .gitlab-ci.yml (YAML). Scripts grouped into jobs → forming a pipeline. Stages allow parallel execution lines.

EXAMPLE — GitLab pipeline (from PA165 slides) 
stages:
  - build
  - test
  - deploy

image: alpine

build_a:
  stage: build
  script:
    - echo "Build A"

build_b:
  stage: build
  script:
    - echo "Build B"

test_a:
  stage: test
  script:
    - echo "Test A"

test_b:
  stage: test
  script:
    - echo "Test B"

deploy_a:
  stage: deploy
  script:
    - echo "Deploy A"

deploy_b:
  stage: deploy
  script:
    - echo "Deploy B"

Variants: DAG pipeline uses needs: [build_a] on each job to bypass the strict stage ordering; Parent-child pipelines trigger nested .gitlab-ci.yml files from subdirectories using trigger: include: a/.gitlab-ci.yml.

FROM LECTURES — Runners & common properties

GitLab Runner — separate project running on a different machine than GitLab CI. Picks up jobs and reports back. Executors: shell, docker, ssh… Types: shared (per group) and dedicated (per project).

Common job properties:

  • when — condition for running (e.g. when: manual).
  • tags — run only on runners with that tag.
  • cache — keep files between jobs (e.g. .m2 Maven cache).
  • artifacts — expose built artifacts (e.g. JAR).
  • environment — tag the env (review / staging / prod).
  • variables — access branch names, commit SHA, CI/CD variables; also user-defined.
FROM LECTURES — CI/CD environments
  • Review environment — new env for each merge request.
  • Staging environment — near-exact replica of production for testing.
  • Production environment — the real thing.

GitLab pipelines run in their own Docker instance. To use a Docker image, run the Docker-in-Docker service (Docker containers within a Docker container).

FROM LECTURES — CI Best Practices
  • Commit early, commit often.
  • Optimize pipeline stages to detect failures as early as possible.
  • Make builds fast and simple.
  • Run tests locally before adding them to the CI/CD pipeline.
  • Use failures to improve processes.
  • Build in a clean environment.
  • Test environment should mirror production.
FROM LECTURES — OWASP Top 10 CI/CD Security Risks
  1. CICD-SEC-1: Insufficient Flow Control Mechanisms — attacker can push malicious code due to lack of approval/review.
  2. CICD-SEC-2: Inadequate Identity and Access Management — many identities + weak IAM = entry point. e.g. allowing self-registration.
  3. CICD-SEC-3: Dependency Chain Abuse — dependency confusion (publish malicious package with same name as internal one), dependency hijacking (compromise maintainer's account), typosquatting, brandjacking.
  4. CICD-SEC-4: Poisoned Pipeline Execution (PPE) — Direct: attacker updates pipeline config to access auth vars. Indirect: attacker changes artifacts used by the pipeline (inject malicious scripts).
  5. CICD-SEC-5: Insufficient PBAC (Pipeline-Based Access Controls) — malicious code abuses overly-permissive pipeline permissions to move laterally.
  6. CICD-SEC-6: Insufficient Credential Hygiene — credentials printed to console; tokens spread across the pipeline.
  7. CICD-SEC-7: Insecure System Configuration — unpatched OS running CI/CD management.
  8. CICD-SEC-8: Ungoverned Usage of 3rd Party Services — a 3rd-party service granted access to CI/CD resources widens attack surface.
  9. CICD-SEC-9: Improper Artifact Integrity Validation — unverified/unsigned artifacts flow through to deployment.
  10. CICD-SEC-10: Insufficient Logging and Visibility — no logs about the deployment process means attacks go undetected.

EXAMPLE — same pipeline in Jenkins (Declarative) 
pipeline {
    agent { docker { image 'maven:3.9-eclipse-temurin-17' } }
    stages {
        stage('Build') {
            steps { sh 'mvn -B -DskipTests package' }
        }
        stage('Test') {
            steps { sh 'mvn -B test' }
            post { always { junit '**/target/surefire-reports/*.xml' } }
        }
        stage('Image') {
            steps {
                sh 'docker build -t $REGISTRY/myapp:$GIT_COMMIT .'
                withCredentials([usernamePassword(credentialsId: 'ecr',
                    usernameVariable: 'U', passwordVariable: 'P')]) {
                    sh 'echo $P | docker login -u $U --password-stdin $REGISTRY'
                    sh 'docker push $REGISTRY/myapp:$GIT_COMMIT'
                }
            }
        }
        stage('Deploy to staging') {
            steps {
                sh 'terraform -chdir=infra/staging apply -auto-approve ' +
                   '-var image_tag=$GIT_COMMIT'
            }
        }
        stage('Deploy to prod?') {
            steps {
                input message: 'Promote to production?', ok: 'Deploy'
                sh 'terraform -chdir=infra/prod apply -auto-approve ' +
                   '-var image_tag=$GIT_COMMIT'
            }
        }
    }
    post {
        failure { slackSend channel: '#deploys', message: "Build ${env.BUILD_NUMBER} failed" }
    }
}

Same shape as the GitLab YAML above: stages run in order; a manual input gates production; credentials injected at runtime; Terraform handles the actual deploy.

Choosing the appropriate programming language

NOTE

The repo doesn't have a dedicated section on language choice, and PA165 (by definition) advocates for Java. The exam likely expects you to reason about trade-offs, not name a winner.

ADDED — language-choice criteria
Problem fit
What is the language designed for? Python for data + glue. Java for business logic, large teams. Go for network services and CLIs. Rust for systems code with high perf + safety. JS/TS for frontend (and increasingly backend with Node/Deno/Bun). C# in Microsoft ecosystems. C/C++ when you need every cycle.
Team familiarity
The "best" language your team doesn't know is worse than the "OK" language they know. Onboarding new hires also matters.
Ecosystem / libraries
Mature libraries for your domain? Active community? Long-term security maintenance?
Performance requirements
JVM/CLR/Go are fast enough for almost anything. Native (Rust/C++) only when measurements demand it. Python/Ruby are productive but slow per-CPU.
Operational characteristics
Startup time (Lambda cold-starts hurt Java without GraalVM Native Image); memory footprint; runtime dependencies; observability tooling.
Type system
Static typing catches bugs at compile time and aids large-team refactoring (Java, TypeScript, Go, Rust, Kotlin). Dynamic typing is faster to prototype (Python, Ruby).
Tooling
Build tools (Maven/Gradle vs npm vs Cargo), IDE support, profilers, debuggers, formatters.
Hiring market
You will lose people. Esoteric languages limit your hiring pool.
Existing infra / company standards
If everything else in the org is Java/Spring, a Haskell service is going to be lonely at 3am during incident response.
INTUITION

The decision matrix usually has three winners depending on context:

  1. "Boring enterprise" backends — Java/Kotlin/C#. Mature, stable, huge talent pool, great DB ecosystem.
  2. Cloud-native / containers / CLIs — Go. Small binaries, fast startup, good concurrency.
  3. System-level / performance-critical — Rust (modern choice) or C++ (legacy).

Then a special case: data / ML — Python. And frontend — TypeScript by default.

Persistence & ORM

FROM REPO (1_programovani)

Persistence deals with retaining data between runs / restarts / client requests.

  • Relational databases — the most common way to store data. Data is structured into tables of columns; a row = a record. Relations via FKs. SQL for manipulation. (Postgres, MySQL, SQLite.) Widely supported in languages. Beware SQL injection (e.g. Rust sqlx). When using SQL in apps, prefer Repository / DAO pattern, or CQRS.
  • Object-Relational Mapping (ORM) — maps your object model to the relational DB. Pro: no direct SQL, code is (should be) DB-agnostic. Con: less control over the query, complex queries become similarly complex in ORM, performance can suffer. (JS/Prisma, Rust/Diesel, Java/Hibernate.)
  • Filesystem — good for blobs (images, PDFs). Make sure permissions are correct.
  • NoSQL — alternative to relational; for unstructured data; better scalability and flexibility; consistency can be an issue (duplication for speed). (MongoDB, Cassandra.)
  • Caching — temporarily storing frequently-used data / op results. RAM-level, in front of DB, in front of server. (Redis, nginx.)

DAO = Data Access Object — abstracts access to DB/persistence; lets you swap the persistence layer without touching other layers; makes business logic easier to test; usually one per entity; CRUD.

DTO a.k.a. Value Object — Data Transfer Object — encapsulates data for transport between layers.

FROM LECTURES — RDBMs in Java (PA165 Lect07-08)

We need to store/retrieve data. Options:

  • JDBC — runs pure SQL via a JDBC driver.
  • Object-Relational Mapping (ORM) — JPA / Hibernate.
  • iBATIS / MyBatis — SQL Map.
FROM LECTURES — ORM concept

Object Relational Mapping = mapping classes (OOP) to relational DBs.

  • Data type mapping (e.g. on Postgres: Long → int8, Integer → int4, String → varchar(255)).
  • Query syntax.
  • Simplifies joins.
  • JBoss Hibernate is the de-facto standard ORM library in Java.

JPA — Java Persistence API

Based on Hibernate. Hibernate implements JPA. JPA 2.2 was the last in Java EE 8; Jakarta Persistence 3.x (renamed packages from javax.persistence.* to jakarta.persistence.*) is current. Spring 6 / Spring Boot 3 adopts JPA 3.1.

EXAMPLE — Native SQL vs JPA Hello World 
-- Native SQL (JDBC)
CREATE TABLE employee (id int PRIMARY KEY, name varchar,
                       gender varchar, monthlySalary int);
INSERT INTO employee(id, name, gender, monthlySalary)
       VALUES (1, 'LukeSkywalker', 'male', 18000);
SELECT * FROM employee WHERE id = 1;
DELETE FROM employee WHERE id = 1;
DROP TABLE employee;
// JPA equivalent
@Entity
public class Employee {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;
    @Enumerated(EnumType.STRING)
    private Gender gender;
    private Integer monthlySalary;
    // non-parametric constructor required
    public Employee() {}
    // getters/setters...
}

EntityManager em = emf.createEntityManager();
em.getTransaction().begin();
em.persist(new Employee("LukeSkywalker", Gender.MALE, 18000));
em.getTransaction().commit();
Employee luke = em.find(Employee.class, 1L);
FROM LECTURES — Entity essentials
  • @Entity, @Id; non-parametric constructor required.
  • EntityManager#persist — insert executed as late as possible; should run inside a transaction.
  • EntityManager#find — always one instance of entity identity per EntityManager / PersistenceContext.
  • EntityManager vs EntityManagerFactory (EMF) — EntityManager is lightweight, not thread-safe; EMF is heavyweight, thread-safe, usually one per application.
  • Auto-generated keys — IDENTITY column or SEQUENCE. IDENTITY requires INSERT when persist is called.
  • Enum mapping — ORDINAL (default; brittle if you reorder the enum) or STRING (recommended).
  • Temporal types — old java.util.Date / java.sql.Date with @Temporal(DATE/TIME/TIMESTAMP); modern java.time (LocalDate, LocalDateTime, Instant) preferred.
  • equals / hashCode — Java collections require them; choose ID equivalence, instance equivalence, or business equivalence. Use instanceof instead of getClass(); use getters on the "other" object, not direct field access.
  • Validation — Bean Validation: @NotNull, @Min, @Past…
FROM LECTURES — Persistence Context

Set of managed entity instances in memory. Annotation: jakarta.persistence.PersistenceContext.

"A persistence context is a set of managed entity instances in which for any persistent entity identity there is a unique entity instance. Within the persistence context, the entity instances and their lifecycle are managed by the entity manager."

Entity states

  • New / transient — just created, not in DB.
  • Managed — in persistence context; changes are tracked and flushed.
  • Detached — was managed; persistence context closed.
  • Removed — marked for deletion.

For detached entities, reattach with find() or merge().

Database synchronization

  • On flush().
  • On transaction commit().
  • Whenever Hibernate decides it's important (e.g. SELECT on a modified table).
  • Flush mode: AUTO vs COMMIT — performance trade-off.
FROM LECTURES — Relationships

Cardinalities: @OneToOne, @OneToMany, @ManyToOne, @ManyToMany. Each can be unidirectional or bidirectional.

Owning side vs Inverse side

  • Unidirectional has only an owning side.
  • Bidirectional has both.
  • Owning side dictates propagation to the database.
  • The owning side is the one with the FK in DB.
  • The inverse side must use mappedBy.

Cascading

"Use of the cascade annotation element may be used to propagate the effect of an operation to associated entities. The cascade functionality is most typically used in parent-child relationships."

Fetch profiles

  • LAZY (default for @OneToMany, @ManyToMany) — load on access.
  • EAGER (default for @ManyToOne, @OneToOne) — load with parent.
  • N+1 problem — lazy loading inside a loop fires one query per element. Mitigate with JOIN FETCH, fetch profiles, or DTO projections. In production monitor your SQL query count.
FROM LECTURES — Java Persistence Query Language (JPQL)

Similar to SQL but operates on entities, not tables.

// Find employees with salary between two values
SELECT e FROM Employee e WHERE e.monthlySalary BETWEEN :min AND :max

// Multiple scalars — returns Object[]
SELECT e.name, d.name FROM Employee e JOIN e.department d

// Constructor expression — build a DTO directly in the query
SELECT NEW com.acme.EmployeeDto(e.name, e.monthlySalary) FROM Employee e

// JOIN FETCH — eager-load a collection in one query
SELECT e FROM Employee e JOIN FETCH e.skills WHERE e.id = :id

Other constructs: BETWEEN, IS EMPTY, LIKE, IN, DISTINCT, GROUP BY, aggregations (SUM, COUNT…).

Caveat: JOIN FETCH can't be used multiple times with Lists in a single query.

Criteria API

Safer than JPQL with regard to type checking. Useful for dynamic queries (e.g. searching by optional fields).

FROM LECTURES — DAO & schema

Data Access Object

  • One DAO per Entity.
  • CRUD operations.
  • No transaction demarcation (do that one layer up).
  • @Repository annotation in Spring — enables exception translation (SQLException → Spring's DataAccessException hierarchy).

Schema creation

hibernate.hbm2ddl.auto values:

  • create — drop and recreate on each startup.
  • create-drop — create on start, drop on shutdown.
  • update — add missing things; never drop.
  • validate — check schema matches but don't modify.
  • none — do nothing.

In production: use validate (or none) and manage schema via Flyway/Liquibase migrations.

FROM LECTURES — Pros & cons of ORM

Advantages

  • "Clean code" — easy to store/retrieve data in Java.
  • Less code & less duplication.
  • Easier to switch DB engines.

Disadvantages

  • Hard to master all aspects (cascading, concurrency, performance tuning, the N+1 problem).
  • Performance.
  • Introduces complicated bugs:
    • equals() / hashCode().
    • Cascading and related data removal/creation.
    • Concurrent bugs (many operations aren't atomic).

Topics not covered in the lectures (but worth knowing)

Bean validation, locking and concurrency, caching, inheritance, legacy mapping strategies (shared keys), DB refactoring strategies.

EXAM TIP — common ORM gotchas
  • Never use hbm2ddl.auto=update in production — migrations should be explicit.
  • Default FetchType.EAGER on @ManyToOne is a perf trap; switch to LAZY.
  • Open Session In View anti-pattern lets the persistence context leak into the controller → surprise SQL during JSON serialization.
  • Always set spring.jpa.open-in-view=false in Spring Boot.
  • Don't return JPA entities from REST endpoints — use DTOs.

Tools and environments for the development and deployment of large-scale software systems
A typical stack covers the full SDLC:
  • Editor/IDE — IntelliJ, VS Code (Language Server Protocol = consistent tooling across editors).
  • Version control — Git on GitLab/GitHub (also issue tracking + CI/CD on the same platform).
  • Build — Maven / Gradle (JVM), npm / pnpm (JS), Cargo (Rust).
  • Testing — JUnit + Mockito + Testcontainers; Playwright for UI; Postman/Insomnia for ad-hoc APIs.
  • CI/CD — Jenkins, GitLab CI/CD.
  • Containers — Docker / Podman; orchestration via Kubernetes (EKS) or ECS.
  • IaC — Terraform.
  • Observability — Grafana + Prometheus, ELK / OpenSearch, OpenTelemetry → Jaeger / X-Ray.
  • Quality — SonarQube, Checkstyle/SpotBugs, ESLint, tfsec/checkov for IaC.
  • Secrets — HashiCorp Vault / AWS Secrets Manager.
The underlying motivation is to fight Lehman's laws (rising complexity, declining quality, continuing change) with automation.
Basic concepts of software architectures in terms of implementation and long-term maintenance
The set of architectural choices, ordered by independence/distribution:
  • Monolith — one deploy; simple, fast to start, but every change rebuilds everything; ageing monoliths become unmaintainable.
  • Layered — Presentation / Business / Persistence / Database; standard monolith refinement.
  • Microkernel / Hexagonal / Ports & Adapters — core + plug-ins through stable interfaces; great for extensibility (VS Code, Jakarta Beans).
  • Pipeline — sequential stateless filters (compilers, unix pipes).
  • MVC/MVP/MVVM — UI patterns separating model, view, and the mediating piece.
  • SOA — 4–12 services, often shared DB, ACID possible.
  • Microservices — many small services, no shared DB, polyglot allowed, eventual consistency.
  • Modular monolith — pragmatic middle ground; one deploy, internal module boundaries.
For long-term maintenance the key principles are IoC / DI (Spring's hallmark) for swappable parts, AOP for cross-cutting concerns, SOLID for cohesion + coupling, and a clear API contract per component.
Technologies supporting continuous integration and deployment (CI/CD)
CI = integrate code in main as fast as possible with automated unit tests on every push. CD = automated delivery up to (or including) production. Pipeline anatomy: runner executes jobs grouped into stages. Build a Docker image, run tests in containers (Docker-in-Docker), push to a registry, deploy via Terraform/CodeDeploy.
  • GitLab CI — .gitlab-ci.yml in repo; jobs in stages run in parallel; runners are shared or dedicated; secrets as CI variables; deploy to review/staging/production environments.
  • Jenkins — Jenkinsfile in repo (declarative or scripted Groovy); stages with parallel blocks; agents in cloud or self-hosted; Credentials plugin for secrets; input step for manual approvals.
Branching: feature-branch + PR review is the de-facto default; Gitflow for release-based products. OWASP CI/CD Top 10 security risks: insufficient flow control, weak IAM, dependency chain abuse, poisoned pipeline execution, weak PBAC, credential hygiene, insecure config, ungoverned 3rd-party services, artifact integrity, insufficient logging.
Choosing the appropriate programming language
Trade-offs to balance:
  • Problem fit — what is the language designed for? (Python for data, Java/Kotlin/C# for enterprise, Go for cloud-native services, Rust for systems perf+safety, JS/TS for frontend.)
  • Team familiarity — the best language your team doesn't know is worse than the OK one they do.
  • Ecosystem — libraries, frameworks, long-term security maintenance.
  • Performance / footprint — Lambda cold starts, RAM costs, throughput.
  • Type system — static catches bugs at compile time and helps large refactors; dynamic is faster to prototype.
  • Tooling — build, IDE, profilers, debuggers.
  • Hiring & org standards — exotic languages limit your hiring pool; consistency with the rest of the org is operationally valuable.
Default picks in 2026: Java/Kotlin/C# for "boring enterprise" backends; Go for cloud-native services and CLIs; Rust for performance-critical / system code; Python for data & ML; TypeScript for frontend.
Persistence
Retaining data between application runs / restarts / requests. Options:
  • Relational DBs (Postgres, MySQL) — structured, ACID, mature.
  • ORM — maps your object model to relational tables (see next).
  • Filesystem — blobs (PDFs, images).
  • NoSQL (Mongo, Cassandra) — flexible schema, horizontal scale.
  • Caching (Redis) — speed at the cost of staleness.
Patterns: DAO/Repository (abstract DB access — swap implementations, mock for tests), DTO (data transfer between layers — don't return entities from APIs), CQRS (separate read & write models). Schema changes via Flyway/Liquibase with expand/contract for zero downtime; never hbm2ddl.auto=update in production.
ORM
Object-Relational Mapping — maps OO classes to RDBMS tables. Lets you write Java/Kotlin/C# instead of raw SQL; code becomes DB-agnostic; simplifies joins.
  • Java standard: JPA spec (Jakarta Persistence), Hibernate is the de-facto implementation.
  • Key concepts: Entity (@Entity, @Id), EntityManager (per-request, not thread-safe) vs EntityManagerFactory (app-scoped), Persistence Context (set of managed entities, dirty-tracking, identity), entity states (transient, managed, detached, removed), cascading, fetch types LAZY/EAGER.
  • Query language: JPQL (entity-aware) or Criteria API (type-safe).
  • Pros: clean code, less duplication, easier DB switches. Cons: opaque generated SQL, complex bugs (equals/hashCode, cascading), the N+1 problem (lazy loading inside a loop fires one query per element — mitigate with JOIN FETCH or DTO projections), performance tuning is non-trivial.
Other languages: Entity Framework Core (.NET), SQLAlchemy (Python), Prisma / TypeORM / Drizzle (Node), GORM (Go), Diesel / SQLx (Rust). Modern trend: compile-time-checked SQL (jOOQ, sqlc, SQLx) as a middle ground.
