Quality in software systems development

FROM REPO

Quality is an important aspect of SW system development. It is commonly defined as the product's ability to meet requirements — so it's crucial to determine what the requirements actually are.

  • Customer requirements a.k.a. external quality: usability, accuracy/correctness, reliability, security, performance, etc.
  • To meet external quality, development must go smoothly, the product must be easy to maintain long-term (easy to modify/extend), and it must not be unnecessarily expensive (in operating costs and modification costs). This is achieved by maintaining internal quality: modularity, simplicity of units, testability, adaptability to change, code readability, reusability, scalability, portability, sustainability, adherence to standards…
  • Poor external quality is often a symptom of poor internal quality (bug fixes take long, system is slow…) — but not always (maybe just the UI is weak).
INTUITION

External quality is what the user sees. Internal quality is what the developer feels. Internal quality is invisible to the customer — until the team is forced to ship a hotfix and it takes 3 weeks. Then it becomes very visible.

ADDED

The industry-standard formalization of this idea is the ISO/IEC 25010 quality model, which decomposes "quality" into product quality (8 characteristics: functional suitability, performance efficiency, compatibility, usability, reliability, security, maintainability, portability) and quality in use (effectiveness, efficiency, satisfaction, freedom from risk, context coverage). You won't usually be quizzed on it, but if a question asks "list quality attributes formally," ISO 25010 is the framework cited in textbooks.

Quality attributes

FROM REPO

The most important quality attributes of code are:

Maintainability
Ease of modification without accumulating technical debt.
Performance
System response time (and efficiency of resource use).
Reliability
Probability of error-free operation over a given period.
Testability
How easily (and how thoroughly) the system can be tested.
Scalability
Ability of the system to handle larger volumes of data/users.
Security
Resistance against attacks.
Usability
Ease of using the system and learning to work with it; correct functionality (usually a separate point).
ADDED

Common acronyms for memorizing the most-asked subset:

  • RAS — Reliability, Availability, Scalability (an SRE-style lens)
  • RASUI — adds Usability, Integrity
  • "-ilities" — in textbooks, non-functional requirements are often called the "-ilities" because most end in -ility.

Note the subtle distinction: availability (uptime, e.g. 99.99%) is what your SLA promises; reliability (mean-time-between-failures) is the underlying property. A system can be reliable but unavailable (correct, but the server is off) or available but unreliable (always responds, often with wrong data).

Software metrics

FROM REPO

Measurable aspects of a SW system (lines of code, test coverage, cyclomatic complexity…) that give us information about the overall picture — but it can be non-trivial to interpret them properly.

  • e.g. 100% test coverage doesn't mean the system is bug-free. A large number of small classes sounds great, but the classes may be structured nonsensically and strongly coupled.
  • Metrics can be direct (measured directly, e.g. number of defects) or derived (calculated from direct metrics, e.g. defect density = defects per product size).

Common metrics:

  • Lines of code (LOC)
  • (Non-)Commented lines of code
  • Number of classes
  • Number of functions/methods
  • Number of packages
  • Number of files
  • Class coupling — how many other classes we call
  • Inheritance depth
  • Number of methods per class weighted by complexity
  • Cyclomatic complexity — number of independent paths through the inspected unit (usually a function) that execution may take.
    CC = E - N + 2P where E = edges, N = nodes, P = connected components (for a function, P = 1).
    The minimum is 1 (no branching).

Often we care more about ratios/derived metrics: e.g. percentage of comments out of total lines, average method size, deviations within the project or between releases…

It is dangerous to use metrics to evaluate a developer's performance.

INTUITION

Goodhart's Law: "When a measure becomes a target, it ceases to be a good measure." If you reward developers by LOC, you'll get verbose code. If you reward by test coverage, you'll get "watermelon tests" (green on the outside, no real assertions inside). Metrics are thermometers, not steering wheels.

EXAMPLE — cyclomatic complexity 
// Java — what's the CC?
int classify(int n) {
    if (n < 0) {                 // branch 1
        return -1;
    }
    if (n == 0) {                // branch 2
        return 0;
    }
    if (n % 2 == 0) {            // branch 3
        return 2;
    }
    return 1;
}

Three ifs ⇒ CC = 4 (the "+1" is the implicit fall-through path). Tools: SonarQube, Checkstyle, PMD; rule of thumb: keep methods ≤ 10. Anything > 15 is a refactoring candidate.

ADDED — metrics you'll meet in DevOps
  • DORA metrics — the four key process metrics from the DORA / "Accelerate" research: deployment frequency, lead time for changes, change failure rate, mean time to restore. These are about delivery quality, not code quality, but they tell you whether your engineering org is healthy. Jenkins / GitLab dashboards can expose all four.
  • Code Climate / SonarQube technical debt ratio — estimated cost (in hours) to fix all maintainability issues, divided by total dev cost.
  • Mutation score (from mutation testing tools like PIT for Java) — more honest than line coverage.

Tactics for quality assurance (per attribute)

FROM REPO
Maintainability
Refactoring into cohesive units so that the location of a needed change is minimal and easy to localize; separation of data from logic (so a unit can be replaced by another); decoupling (depending on interfaces rather than implementations).
Performance
Caching · parallelism · asynchronous communication/processing · detecting and mitigating bottlenecks · removing adapters and simplifying communication · separation of data from logic.
Reliability
Detecting and remedying sources of unreliability · control mechanisms · proper error handling · automatic reporting of unexpected errors · timeout after a request · regular ping (initiated by the calling service) · heartbeat (a service regularly reports its status) · regular snapshots and rollback on crash · transactions · input checking at every level · removing single points of failure.
Testability
Refactoring toward dependencies on abstractions, decoupling · separation of data and logic · removing global state.
Scalability
Refactoring into simpler, independently deployable units · extracting data to allow unit parallelization · extracting and independently deploying a subsystem · distribution and/or replication of data (DB tends to be the bottleneck; other things parallelize more easily).
Security
Bug detection and fixing · encrypted communication.
Usability
UX improvements · applying performance/scalability tactics when things are slow.

Clean Code

FROM REPO

Readable, easy to understand. Code is read much more often than it is written, so it's vital to make it understandable — developer time is expensive. The key is:

  • Clear naming reflecting the problem domain, sufficiently expressive (not too long or too generic — looking at you, Java). Ideally self-explanatory, no comments needed; comments are still fine for explaining the why or summarizing what a function does.
    • Consistency across the codebase is important (ideally built on the standards of the given language).
    • If it returns a bool, name it has*() or is*().
    • Structs/classes are nouns; methods start with a verb (or are a getter).
    • Public APIs should be clear and simple; internally (private), longer method names are fine when they make purpose clearer.
  • Reasonable size of units — ideally short functions, simple classes… single-responsibility principle. The content of a unit should reflect its name.
  • Using language/technology standards.

DRY — Don't Repeat Yourself

FROM REPO
  • Each piece of information should be uniquely defined in exactly one place in the system.
  • e.g. generate documentation from source code so there isn't a second source of truth.
  • e.g. define a schema (Prisma) from which we generate both SQL tables and structs for our language.
  • e.g. pull shared functionality into its own function.
  • Applies to anything that can be duplicated in the system — even in processes: repeated manual test runs ⇒ automate them.

KISS — Keep It Simple, Stupid

FROM REPO
  • Simplicity over cleverness.
  • The systems that work best are the simplest ones.
  • There's no reason to use complex techniques on simple problems.

YAGNI — You Ain't Gonna Need It

FROM REPO
  • Don't build something you won't need (e.g. don't over-abstract for hypothetical future functionality unless it's truly necessary).
  • It's better to do a thing simply and then easily change it, than to do it universally only to find an edge case forced a full rewrite. Development proceeds in small steps.
INTUITION

The three are three angles on the same idea: delete, don't predict. DRY says "delete duplication." KISS says "delete cleverness." YAGNI says "delete speculation." Every line of code is a liability you have to maintain forever; don't write any you don't need today.

SOLID

FROM REPO
S — Single Responsibility
Each class should have only one responsibility, i.e. there should be only one reason for a class to change. (e.g. FileReader should only handle reading from a file, not processing the data read. Only a change to how we read should cause FileReader to change.) ⇒ lower coupling, higher cohesion, more flexible code.
O — Open/Closed
Open for extension, closed for modification: we prefer adding new functionality to changing existing source/binary. ⇒ smaller chance of breaking things; nothing depends on the new classes. Implemented via interfaces / abstract classes. OCP increases complexity (and slightly hurts performance in languages needing dynamic dispatch), so use it judiciously where functionality changes/adds often.
L — Liskov Substitution
Instances of classes should be replaceable by their subclasses without breaking system behaviour — all subclasses should honour the contracts of their parents and not remove parent behaviour. A problem sign: having to explicitly check which subtype an instance is — that's what polymorphism should solve. If two classes share much functionality but aren't substitutable, refactor them to subclass a new class containing the shared behaviour.
I — Interface Segregation
Clients shouldn't be forced to depend on methods they don't use, i.e. prefer small, simple interfaces over large ones. (e.g. in Rust, to convert a struct to a string you just implement the Display trait — nothing else.)
D — Dependency Inversion
Modules should depend on abstractions (interfaces), not concrete implementations. Lowers module coupling; allows custom or mock implementations. Constructors should accept everything the struct depends on rather than creating resources themselves (e.g. a repo shouldn't create the DB connection itself; it should be passed via the constructor) = constructor dependency injection.
EXAMPLE — SRP violation in Java 
// BAD — class does parsing AND persistence
class UserService {
    User parse(String csvLine) { ... }
    void save(User u, Connection c) { ... }
    void sendWelcomeEmail(User u) { ... }
}

// GOOD — three responsibilities, three classes
class UserParser    { User parse(String csvLine); }
class UserRepository{ void save(User u); }
class UserNotifier  { void sendWelcomeEmail(User u); }

Now a change to the email template doesn't risk breaking the CSV parser.

EXAMPLE — OCP via Strategy 
// Adding a new payment provider should NOT require editing PaymentService.
interface PaymentProvider { Receipt charge(Money m); }

class StripeProvider implements PaymentProvider { ... }
class PayPalProvider implements PaymentProvider { ... }
// Add ApplePayProvider later — no change to PaymentService.

class PaymentService {
    private final PaymentProvider provider;     // dependency injection (D too!)
    PaymentService(PaymentProvider p) { this.provider = p; }
    Receipt pay(Money m) { return provider.charge(m); }
}
EXAMPLE — LSP violation 
class Rectangle { void setWidth(int w); void setHeight(int h); int area(); }

class Square extends Rectangle {
    void setWidth(int w)  { super.setWidth(w); super.setHeight(w); }
    void setHeight(int h) { super.setWidth(h); super.setHeight(h); }
}
// Any code expecting Rectangle and calling setWidth(5); setHeight(10); assert area()==50;
// will fail when handed a Square. The "is-a" relationship in math doesn't hold for behaviour.
INTUITION

Think of SOLID not as five rules but as two themes: cohesion (SRP, ISP — "keep things narrow and focused") and coupling/extensibility (OCP, LSP, DIP — "depend on contracts, not concretions, so you can swap parts").

Refactoring

FROM REPO
  • Modifying a module such that its external behaviour does not change — only its internal structure / modifiability is improved.
  • Before refactoring, it's important to have behaviour solidly covered by tests, so we don't accidentally change anything.
  • During refactoring we do nothing else (no new functionality).
  • Techniques (many editors support these, which makes the work easier and probably more reliable):
    • Extract function — a piece of code acts as a unit / would need a comment ⇒ extract it into a function, give it a fitting name; can be reused elsewhere.
    • Inline function — the opposite; good for trivial cases like isMoreThanFiveEven(x).
    • Replace function with method on struct — useful when a function uses a bundle of variables ⇒ they become fields of the struct.
    • Move method/field from one struct to another if it makes sense (e.g. domain-wise).
    • Extract/inline class — from a class containing a set of related fields, extract a new object the original class will contain (or inverse for inline).
    • Early return — the function should describe the correct/error-free flow. If we detect a bad input, just return. Don't use if-else; use if return.
    • Rename anything.
    • Group many parameters into a struct (parameter object).

Code that reads and maintains well is not always the fastest/most efficient (abstractions cost). Usually a small performance drop in exchange for readability is fine, but not always.

EXAMPLE — Early return refactoring 
// BAD — pyramid of doom
String greet(User u) {
    if (u != null) {
        if (u.isActive()) {
            if (u.hasName()) {
                return "Hello, " + u.getName();
            } else { return "Hello, friend"; }
        } else { return "Account inactive"; }
    } else { return "Unknown user"; }
}

// GOOD — early returns
String greet(User u) {
    if (u == null)        return "Unknown user";
    if (!u.isActive())    return "Account inactive";
    if (!u.hasName())     return "Hello, friend";
    return "Hello, " + u.getName();
}

Testing

FROM REPO

Testing = the process of evaluating whether the system meets specified requirements… easier said than done.

  • In practice testing is usually incomplete. Testing reveals defects; it does not prove correctness.
  • Each test should test only one thing/property/feature; ideally lots of small tests so the source of a problem is easy to find.
  • Ideally testing is done by someone other than the author of the code being tested.

Categories by knowledge of internals:

  • Whitebox (structural) — we can see source and direct inputs at critical points (off-by-one, zero division…). e.g. unit, integration, performance tests.
  • Blackbox (functional) — we don't see what's happening inside, only inputs and outputs. e.g. acceptance, system tests.

If you hit a bug for which there was no test, fix it and add a (preferably automated) test so the bug can't recur.

  • Regression testing — checking that changes haven't broken existing (automated) tests.
  • Smoke testing — checks that selected critical functions work in a new environment. If not, there's no point in deploying or running further tests.
  • Sanity testing — like smoke, but to verify a bug fix / new functionality.
  • A/B testing — two variants in production; we measure which is more successful (typically for UI).

:hahaa: In practice, some "experts" practice watermelon testing — green on the outside, red on the inside :hahaa:

  • Test quality can be verified with mutation testing. We inject defects (mutating source via operator negation, off-by-one, dropped calls…) and see how many are caught by tests. If something passes, it's a candidate for new tests.
  • Some scenarios are riskier (can be assessed during analysis) and should get more testing attention.
  • Inputs should be split into categories (e.g. <0, 0, >0); pick a few representatives from each.
  • Make invalid states unrepresentable (Rust enums <3, builder pattern, state machines…).
INTUITION

Edsger Dijkstra: "Program testing can be used to show the presence of bugs, but never to show their absence." The goal isn't to prove correctness — the goal is to cheaply find bugs early.

Test coverage

FROM REPO
  • Coverage = the share of code that has been exercised by at least one test, usually in %.
    • Line / statement coverage — lines / statements covered.
    • Function coverage — was the function called at least once?
    • Branch coverage — logical branches covered.
    • Condition coverage — each boolean condition evaluated as both true and false.
  • 100% is often not possible (redundant defensive tests, better-safe-than-sorry), and 100% coverage doesn't mean bug-free.
  • Some parts are harder to cover than others.
  • Coverage helps find untested code but says little about test quality.

Unit tests

FROM REPO
  • Validate that an isolated unit of code (function / class) behaves as expected.
  • White box.
  • Automated, fast, simple, readable, deterministic, each tests one thing.
  • Isolated from the rest of the system via test doubles — fake dependencies:
    • Dummy object — never used, but needed as a parameter.
    • Fake — for tests only; simple but not usable in production (e.g. in-memory DB).
    • Stub — always returns the same thing.
    • Spy — remembers how and with what it was called (e.g. "the send-email method was called with this body").
    • Mock — pre-programmed (when called with A, do X, otherwise Y).
  • AAA — Arrange (setup), Act (execute), Assert (verify) — three phases; act should be as short as possible.
  • e.g. cargo test, Jest, JUnit.
  • Advanced techniques: source analysis + input generation (symbolic execution), formal verification with mathematical proofs, model checking…
EXAMPLE — JUnit + Mockito AAA 
@Test
void chargesCardAndRecordsReceipt() {
    // ARRANGE
    PaymentProvider stub = mock(PaymentProvider.class);
    when(stub.charge(any())).thenReturn(new Receipt("R-1"));
    Ledger ledger = mock(Ledger.class);
    PaymentService svc = new PaymentService(stub, ledger);

    // ACT
    Receipt r = svc.pay(Money.eur(42));

    // ASSERT
    assertEquals("R-1", r.id());
    verify(ledger).record(r);          // spy-style verification
    verify(stub).charge(Money.eur(42));
}

Integration tests

FROM REPO
  • Check whether units interact as expected.
  • Slower, larger and more complex than unit tests.
  • Black / white box.
  • For UI testing, use Playwright (previously Selenium).
ADDED — Testcontainers

For backend integration tests in Java, the de-facto standard is Testcontainers — it spins up real Docker containers (Postgres, Kafka, LocalStack for AWS) for the test, then tears them down. You get tests that run against the real database driver, not an H2 imitation. Critical for catching SQL-dialect issues before prod.

System tests

FROM REPO
  • Verify the system meets specified requirements.
  • Test usability, capacity, performance, functional completeness, security…
  • Benchmarking, penetration testing, user testing…
  • Can be automated via a programmatically driven browser (Playwright, formerly Selenium, Puppeteer).
  • Black box.

Acceptance tests

FROM REPO
  • Verify the system meets business requirements and is ready for release.
  • May take the form of ticking off boxes against requirements the customer set in advance.
  • Performed with the customer.
  • Black box.

Test-driven development (TDD)

FROM REPO

Three phases — red, green, blue — iterated. Each phase aims for one thing only:

  • Red / test — write a failing test for the smallest piece of functionality.
  • Green / write — implement it as simply as possible so the test passes (and doesn't break others).
  • Blue / refactor — clean up the implementation to standards.

Behaviour-driven development (BDD)

FROM REPO
  • With the customer, write the system's behaviour as scenarios.
  • Scenarios serve devs and testers alike.
  • e.g. Gherkin, Cucumber — given / when / then (like AAA) define scenarios in an English-like language customers can read; the same scenarios appear in tests.
EXAMPLE — Gherkin 
Feature: Password reset email

  Scenario: User receives reset link
    Given a user with email "alice@acme.com" exists
    When  she submits the password-reset form
    Then  she receives an email containing a one-time reset link
    And   the link expires in 60 minutes

Debugging and performance testing

FROM REPO

The goal is identifying and solving problems related to speed, response time and throughput — finding the system's limits.

  • Load testing — how the system handles longer-term load.
  • Stress testing — how the system handles short-term load spikes (a sudden surge of requests).

A running system should also be monitored long-term to surface further weak spots.

Performance can generally be increased at the expense of other attributes (e.g. maintainability), so picking the right trade-off matters.

Tools: Gatling, Siege, LoadRunner.

ADDED — debugging discipline

"Debugging" wasn't elaborated in the repo. A few canonical techniques the examiner may expect:

  • Reproduce first. A bug you can't reproduce is a bug you can't fix; invest in repro before in fix.
  • Bisect. git bisect — binary search across commits to find which commit introduced a regression.
  • Rubber-duck. Explaining the bug aloud forces you to articulate hidden assumptions.
  • Hypothesis-driven. "If X is the cause, I should see Y. Let's verify." Avoid shotgun debugging.
  • Logging vs. step-through debugger — both have a place. Concurrency / async bugs usually need logging (debugger changes timing). Algorithm bugs usually need a debugger.
  • Post-mortems. Once fixed, ask: how did the bug get past tests? Add the missing test.
ADDED — performance testing types
TypeWhat it answers
Load testDoes the system meet SLA under expected normal load?
Stress testWhat is the breaking point? How does it fail?
Soak / enduranceMemory leaks? Resource exhaustion over hours/days?
Spike testRecovery from a sudden 10× surge.
Capacity testWhat's the largest load that still meets SLA?
Volume / DB testBehaviour with very large data sets.

In your stack: Gatling pipelines triggered by Jenkins, results in Grafana; or AWS Distributed Load Testing solution for synthetic large-scale runs.

Quality management process

FROM REPO

The quality management process in SW development consists of:

  • Defining SW quality requirements and planning — specifying functional and non-functional requirements, setting evaluation criteria, risks, metrics, detailing activities to ensure quality.
  • Assuring SW quality — defining and overseeing processes that prevent defects (including CI/CD setup).
  • Controlling SW quality — checking whether the product or its parts meet requirements (including quality requirements) and that development follows the defined processes; monitoring whether we're sticking to processes and goals.
  • Improving quality — striving to improve processes to improve quality.
FROM REPO — Capability Maturity Model

Defines an organisation's maturity levels for quality assurance:

  1. Initial — chaos; unpredictable cost, schedule.
  2. Repeatable — intuitive; cost and quality vary; plan under conscious control; informal methods.
  3. Defined — quality-oriented; reliable cost & plan; system performance still unpredictable.
  4. Managed — measurement; deliberate, statistically managed product quality.
  5. Optimising — automation, process improvement, defect prevention, tech innovation.

Prevention of quality problems

FROM REPO
  • Following best practices and conventions — general (SOLID, Clean Code) and language-specific (eslint, cargo fmt).
  • Using design-pattern principles.
  • TDD, pair programming, code reviews.
  • Process standards (ITIL); agile techniques (Scrum, Kanban).
  • Automated testing, CI.
  • Communication, ubiquitous language.
  • Fail-fast — detect input problems early rather than blindly accept anything and be surprised later.
  • Design by Contract — methods may require a contract (assertions) so they can guarantee outputs. Use conditional compilation to keep contracts in dev only (but you lose precise problem-location info in prod).

Non-functional quality problems are addressed by architecture. To prevent them, build a model and simulate requirements (e.g. ability to serve N requests in T seconds) and derive per-component constraints (e.g. max processing time per component).

For verification, formal verification is sometimes used (e.g. for algorithm correctness).

Detection of quality problems

FROM REPO
  • Code reviews (peer between devs), inspections (formal — use a form to ensure preparedness, filter trivia; focus on correctness, standards, not style).
  • (Automated) testing (cargo test).
  • Static analysis (cargo clippy, borrow checker, SonarQube) — we don't run the code.

Tracking quality problems

FROM REPO
  • Issue tracking.
  • Technical-debt management (tracking, allocating time to pay down).

Bad code & code smells

FROM REPO — Bad-code symptoms
  • Mixed levels of abstraction.
  • Low cohesion (mega-classes, long functions…).
  • Circular dependencies (implementation coupling, hard to follow, hard to test, maintain, scale).
  • Code duplication.
  • Too many parameters.
  • Bad names.
  • Being too "clever" when it's not needed.
  • Not following standards / suitable language constructs.
  • Magic constants.
FROM REPO — Code smells & cures (per attribute)

Tactics can conflict (e.g. maintainability vs performance) — pick what you're optimising for!

Maintainability
Premature optimisation ⇒ profile first, then optimise. Excessive flexibility ⇒ start simple, extend later. Cleverness ⇒ keep foundations simple. Not using standards / patterns. Low modularization.
Performance
Redundant work ⇒ caching, memoization, bottom-up DP. Sequential search ⇒ binary search, smarter algorithms, sorted collections, parallelization go brrrr. Long critical sections in multithreaded programs ⇒ minimize critical sections, possibly multiple locks. Active waiting ⇒ async processing, get notified when the op completes.
Reliability
Trusting input blindly. Bad error/exception handling. Assuming functions are only called in one order. Hype-train: using tech we don't fully understand. No logging. Single point of failure killing the whole system ⇒ multiple services, restart/recover, auto-failover.
Testability
Global state. Hidden dependencies (prefer DI over init() globals). Cross-unit communication that shouldn't exist ⇒ SOLID. Hacks needed to test ⇒ SOLID. Nondeterminism (time, randomness, global state, DB… watch out for Rust HashMap iteration order). Init logic mixed with app logic.
Scalability
Monolith. Distribution increases capacity/perf but adds overhead — harder testing, deployment, security.
EXAM TIP

The examiner often wants you to name the attribute and then name two or three concrete tactics for it. Don't enumerate everything — pick 2–3 strong ones and give a one-line example. e.g. "For testability: dependency injection (constructor takes a Clock so we can inject a fixed clock in tests), no global state, separation of init from logic."

Practical examples — putting it together

ADDED — end-to-end example

You're shipping a new refund endpoint in a Java/Spring service deployed to ECS via Jenkins + Terraform. Quality concerns and where they live:

  1. Define — acceptance criteria written in Gherkin in the Jira ticket. NFRs reference the existing wiki page ("p99 latency < 300 ms; refund must be idempotent within 24 h").
  2. Assure —
    • Pre-commit: spotless formatter, checkstyle.
    • CI: mvn verify runs unit + integration tests (Testcontainers Postgres), jacoco enforces ≥80% branch coverage on changed files, sonar-scanner blocks merge if new code introduces critical issues.
    • Terraform: tfsec + checkov in pipeline.
  3. Control —
    • Pre-deploy smoke test against staging.
    • Post-deploy: CloudWatch synthetic canary hits /refund every minute.
    • Alarms on p99 latency & error rate route to PagerDuty.
  4. Improve — weekly review of DORA metrics in Grafana; blameless post-mortem template for every Sev-2+ incident.

Each phrase from the exam prompt, with a tight answer you could give aloud in 30–60 seconds. Use this as a final-revision checklist.

Quality in software systems development
Quality = the product's ability to meet requirements. Split into external quality (what the customer sees: correctness, performance, security, usability, reliability) and internal quality (what the developer feels: modularity, testability, readability, low coupling, high cohesion). Poor external quality is usually a symptom of poor internal quality — bugs are slow to fix because the code is tangled.
Quality attributes
The "-ilities". The seven typically asked: maintainability (ease of change without debt), performance (response time + resource efficiency), reliability (prob. of error-free operation), testability (ease of testing), scalability (handles more load/data), security (resistance to attack), usability (ease of use and learning). The industry-standard reference framework is ISO/IEC 25010.
Software metrics
Measurable aspects of code: LOC, number of classes/methods, class coupling, inheritance depth, cyclomatic complexity (independent paths, E - N + 2P), test coverage. Direct metrics are measured, derived ones computed from direct (defect density). Goodhart's Law warning: when a measure becomes a target it stops being a good measure — metrics are thermometers, not steering wheels.
Tactics for quality assurance at the level of individual quality attributes
Per-attribute mitigations. Maintainability → refactor to cohesive units, decouple via interfaces. Performance → caching, parallelism, async, bottleneck mitigation. Reliability → input validation, transactions, retries, heartbeats, removal of single points of failure. Testability → DI, no global state, separate init from logic. Scalability → smaller independently deployable units, sharded data. Security → encryption + bug fixing. Tactics conflict (perf vs maintainability) so you must choose what to optimize.
Clean Code
Robert Martin's discipline of writing code that is read more often than written. Key practices: meaningful names (booleans as has*/is*, classes as nouns, methods as verbs), small functions / classes (SRP), language conventions. Three guiding principles: DRY (don't repeat yourself — one source of truth), KISS (simplicity first), YAGNI (don't build for hypothetical futures).
SOLID principles
Five OO design principles:
  • Single Responsibility — one reason to change per class.
  • Open/Closed — open to extension, closed to modification (extend via inheritance/interfaces, don't edit existing).
  • Liskov Substitution — subclasses must honour the parent's contract.
  • Interface Segregation — many small interfaces over one fat one.
  • Dependency Inversion — depend on abstractions, inject concretes.
Two themes: cohesion (SRP, ISP) and coupling/extensibility (OCP, LSP, DIP).
Code refactoring
Changing the internal structure without changing external behaviour. Requires tests as a safety net. Common techniques: extract function, inline function, move method/field, extract/inline class, early return (replace nested if-else with guard clauses), rename, parameter object (group many params into a struct). Trade-off: readability may slightly hurt performance, usually worth it.
Code testing
The process of evaluating whether the system meets specified requirements. Dijkstra: "testing can show the presence of bugs, never their absence." Practical taxonomy by visibility: white-box (see source — unit, integration) vs black-box (only inputs/outputs — system, acceptance). Other types referenced by name: regression, smoke, sanity, A/B, mutation testing.
Unit tests
Validate an isolated unit (function/class) behaves as expected. White-box. Fast, deterministic, one assertion per test. Dependencies isolated via test doubles: dummy (placeholder), fake (working but unfit for prod, e.g. in-memory DB), stub (returns canned data), spy (records calls), mock (pre-programmed expectations). Structure: AAA — Arrange, Act, Assert.
Integration tests
Verify multiple units interact correctly. Slower, larger, black/white box. Real DB via Testcontainers (spins up actual Postgres/Kafka in Docker), real HTTP via REST Assured / Playwright for UI.
User and acceptance tests
User tests (system tests) — verify the system as a whole meets specified requirements: usability, capacity, performance, security. Black box. Driven via automated browser (Playwright). Acceptance tests — verify business requirements; performed with the customer ticking off pre-agreed criteria. Often expressed in Gherkin (Given/When/Then) as part of BDD.
Debugging
Systematic discipline, not random poking. Steps: reproduce reliably, bisect via git bisect to find the regression commit, form a hypothesis, verify with one experiment at a time. Tools: debuggers for sequential algorithm bugs, logging for concurrency / timing bugs (a debugger changes timing). Always end with a post-mortem: how did this slip past tests? Add the missing test.
Performance testing
Find the system's limits. Load testing — sustained expected traffic. Stress testing — short spikes / overload. Plus soak (endurance, find leaks), spike, capacity, volume. Tools: Gatling, Siege, JMeter, LoadRunner. Outcome: identify bottlenecks; remember performance often trades off against maintainability.
Quality management process in software systems development
Four phases: define (specify functional + non-functional requirements, set metrics & criteria) → assure (define and enforce processes that prevent defects: CI/CD, code review, standards) → control (verify the product and the processes are followed; monitor) → improve (continuously improve processes). The Capability Maturity Model (CMM) rates an org's maturity from 1 (chaos/initial) to 5 (optimising).
Practical examples for all of the above
End-to-end on a Java/Spring service on AWS: define acceptance criteria in Jira (Gherkin) + reference NFR wiki. Assure with pre-commit Spotless + Checkstyle, CI runs Maven verify with JaCoCo coverage gate, SonarQube quality gate, Terraform scanned by tfsec/checkov. Control with CloudWatch synthetic canaries, latency & error alarms → PagerDuty. Improve with weekly DORA-metric review (deploy freq, lead time, change failure rate, MTTR) and blameless post-mortems.
