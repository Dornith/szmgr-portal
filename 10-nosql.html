Why NoSQL exists

FROM LECTURES — Trends driving the change (PA195 L1)
  • Big Data — Volume, Velocity, Variety. Later authors added Veracity and Value. Sources: social networks, server/router logs, sensor networks, IoT, machine-generated data (one transatlantic Boeing flight = 640 TB).
  • Big Users — a web app can have millions of users within months.
  • Cloud computing — flexibility & distribution by default.

Big Data definition (Gartner 2012): "high volume, high velocity, and/or high variety information assets that require new forms of processing to enable enhanced decision making, insight discovery and process optimization."

FROM LECTURES — RDBMS for Big Data?
TrendRequirement
Volume of dataReal DB scalability — massive distribution, dynamic resource management
Cloud computingHorizontally scaling systems
VelocityFrequent updates
Many usersMassive read throughput
VarietyFlexible schema, semi-structured data

Why RDBMS struggles:

  • Relational schema — data in tuples, schema known a priori. But current data is naturally flexible.
  • Schema normalization (3NF) — data split into tables, queries merge them. Inefficient at scale; slow in distributed environment.
  • Full ACID transactions — very inefficient in distributed environments.
FROM LECTURES — What is NoSQL? (Sadalage & Fowler)

The term "NoSQL" was coined in the late 90s by Carlo Strozzi (for something different). Resurrected at an informal meetup in 2009 in San Francisco (Voldemort, Cassandra, Dynomite, HBase, Hypertable, CouchDB, MongoDB).

"NoSQL is an accidental term with no precise definition." — Sadalage & Fowler

NoSQL databases are (mostly):

  • Not using the relational model (nor SQL)
  • Designed to run on large clusters (horizontally scalable)
  • No schema — fields can be freely added to any record
  • Open source
  • Based on the needs of 21st-century web estates

Often also: easy replication, simple API, eventually consistent (not ACID).

FROM LECTURES — NoSQL properties & challenges

Properties

  1. Good scalability — horizontal instead of vertical.
  2. Dynamic schema — different levels of flexibility by DB type.
  3. Efficient reading — spend more time storing data so reads are fast; keep related information together.
  4. Cost saving — designed for commodity HW; typically open source.

Challenges

  1. Maturity — getting better, but RDBMS had decades.
  2. User support — rarely the kind of pro support Oracle provides.
  3. Administration — massive distribution requires advanced admin skill.
  4. Standards for data access — RDBMS has SQL; NoSQL world is wilder.
  5. Lack of experts — not enough DBAs for NoSQL stacks.
INTUITION — relational vs NoSQL in one line

Relational DBs ask "what's the perfect way to model this data?" NoSQL DBs ask "what's the perfect way to model this data for the way my app actually reads and writes it?" That's the whole shift. NoSQL trades the universality of the relational model for tight optimization to specific access patterns.

The Aggregate Data Model (the key concept)

FROM LECTURES — From RDBMS to NoSQL (PA195 L3)

Efficient implementations of table joins and transactional processing require a centralized system. NoSQL Databases:

  • Database schema tailored for a specific application.
  • Keep together data pieces that are often accessed together.
  • Write operations might be slower; read is fast.
  • Weaker consistency guarantees.

⇒ efficiency and horizontal scalability.

FROM LECTURES — Aggregates

An aggregate is:

  • A data unit with a complex structure (not just a row).
  • A collection of related objects treated as a unit.
  • The unit for data manipulation and management of consistency.

The relational model is aggregate-ignorant. That's a feature — it lets you look at the data in many ways. The downside: when there's a clear primary access pattern, you'd benefit from telling the database "this stuff goes together." Aggregates do exactly that.

Key-value, document, and column-family stores are aggregate-oriented. Graph stores are not.

EXAMPLE — aggregate (PA195 slides) 
// Customer collection
{
  "customerID": 1,
  "name": "Jan Novák",
  "address": { "city": "Praha", "street": "Krásná 5", "ZIP": "111 00" }
}

// Order collection — order contains its items as a nested array
{
  "orderNumber": 11,
  "date": "2015-04-01",
  "customerID": 1,
  "orderItems": [
    { "productID": 111, "quantity": 1,  "price": 1300 },
    { "productID": 112, "quantity": 10, "price":  300 }
  ],
  "invoice": { "bankAccount": "...", "address": {...} }
}

// Product collection
{ "productID": 111, "name": "Vysavač ETA E1490" }
{ "productID": 112, "name": "Sáček k ETA E1490" }

An Order aggregate carries everything you need to display or process that order — no joins. A relational schema would split this across orders, order_items, products, invoices, addresses, etc.

INTUITION — why aggregates matter for distribution

Aggregates tell the database what should live on the same node. When you fetch an Order, the system retrieves a single aggregate from a single node — no cross-machine joins. That's the whole reason aggregate-oriented NoSQL scales horizontally so well, and the reason graph databases (which can't define a single aggregate when relationships go everywhere) struggle with distribution.

One other consequence: most aggregate-oriented NoSQL DBs only guarantee atomic writes within a single aggregate. Cross-aggregate updates can race.

Distribution: sharding & replication

FROM LECTURES — Scalability

Scalability = handling growing amounts of data and queries without losing performance.

Vertical (scaling up)

  • Bigger, more powerful machine: more RAM, more cores, disk arrays.
  • Pro: simple, no data distribution to handle; favours strong consistency.
  • Cons: higher costs; data growth limit (even the largest machine has a limit); needs proactive provisioning (upfront budget); vendor lock-in (large machines come from a few vendors).

Horizontal (scaling out)

  • Distribute across many commodity machines.
  • Data partitioned across many disks; can use all machines' main memory; distributed computation.
  • Cons: synchronization, consistency, partial failures.

Watch out for the fallacies of distributed computing: network is reliable, latency is zero, bandwidth is infinite, network is secure, topology doesn't change… all false.

FROM LECTURES — Two ways to distribute

For horizontal scalability:

Replication
The same data is copied over multiple nodes. Master-slave or peer-to-peer.
Sharding
Different chunks of data are placed on different nodes (data partitioning). Different clients access different parts of the dataset.

You can use either or combine. Distribution models = specific ways to do sharding, replication, or both.

Best case is still a single server if you can — spares you a lot of problems. NoSQL on a single server still gives you flexible schema and simplicity. Graph DBs are especially happy single-server (it's hard to partition a dense graph).

FROM LECTURES — Sharding goals
  1. Keep data accessed together, together — so the user gets it from a single server. Aggregates help.
  2. Arrange data on the nodes:
    • Keep load balanced (can change over time).
    • Consider the physical location (data centres near users).

Many NoSQL databases offer auto-sharding. A node failure makes that shard's data unavailable — so sharding is often combined with replication.

FROM LECTURES — Master-slave replication
  • Replicate data across multiple nodes.
  • One node = primary (master); others = secondaries (slaves).
  • Master processes all updates.
  • Reads can come from any node.

Scales read-intensive applications: more reads ⇒ more slaves. If the master fails, slaves can still serve reads, and a slave can be promoted to new master quickly.

Bottleneck: the master's ability to process writes. Masters can be selected manually (user-defined) or automatically (cluster-elected).

FROM LECTURES — Peer-to-peer replication
  • No master — all replicas are equal.
  • Every node can handle a write, then spreads the update.

Problem: consistency. Two users may write simultaneously at two different nodes.

Solutions:

  • Coordinate before writing (avoids conflict at the cost of network traffic).
  • Not all replicas need to agree — just a majority (quorum, see below).
FROM LECTURES — Combinations
  • Sharding + master-slave — each shard has a single master. A node can be master for some data and slave for other.
  • Sharding + peer-to-peer — common for column-family DBs. Typical default replication factor = 3 (each shard on three nodes). ⇒ need to solve consistency issues.

Consistency

FROM LECTURES — Consistency definitions

"Consistency is the lack of contradiction in the DB."

Centralized RDBMS ⇒ strong consistency. Distributed NoSQL DBs typically relax consistency (and/or durability):

  • Strong consistency → eventual consistency.
  • BASE instead of ACID.
  • CAP theorem — tradeoff between consistency and availability.
FROM LECTURES — Three kinds of consistency conflict

1. Write-write conflict

Two users want to update the same record. Risk: lost update, second update based on stale data.

  • Pessimistic — prevent conflicts: acquire write locks before update.
  • Optimistic — let conflicts happen, detect, resolve: conditional update, save both updates and record the conflict, version stamps.

2. Read-write conflict (inconsistent read)

One user reads while another writes — logical inconsistency. Ideal solution: ACID transactions (strong consistency). NoSQL DBs typically support atomic updates only within a single aggregate, so updates affecting multiple aggregates leave an inconsistency window.

3. Replication consistency

Ensuring the same data item has the same value on different replicas. After some time the write propagates everywhere — "eventual consistency". In the meantime: stale data.

Read-your-writes (session consistency) — gets violated if a user writes and reads on different replicas. Solution: sticky sessions (session affinity).

FROM LECTURES — CAP Theorem

CAP = Consistency, Availability, Partition tolerance.

Consistency
After an update, all readers in a distributed system see the same data.
Availability
Every request results in a response.
Partition tolerance
System continues operating even if two sets of servers get isolated — a connection failure shouldn't shut the system down.

CAP Theorem: a shared-data system cannot have all three. Only two of three are possible. First formulated in 2000 by Eric Brewer; proven in 2002 by Gilbert & Lynch.

Reality:

  • A single-server system is always CA (as are all ACID systems).
  • A distributed system practically must be Partition-tolerant (you can't reliably detect all network failures).
  • ⇒ the real choice is C vs A during partitions. It's not a binary decision; it has many shades.
FROM LECTURES — CAP in practice

PC — Partition tolerance + Consistency (strong consistency)

Two users, two masters. Before a write is committed, both nodes must agree on order. If partitioned, writes wait forever — availability lost (reads may still work).

PC with master-slave (some availability)

Master can commit the write during a partition. Slaves have stale data — we've lost some consistency. Better balance.

PA — Partition tolerance + Availability

Peer-to-peer replication, eventual consistency. All requests are answered during partitions — but we risk losing consistency guarantees entirely.

Quorum (the middle road)

For replication consistency.

FROM LECTURES — Quorum (R + W > N)

Peer-to-peer replication with replication factor N (number of replicas of each data object).

  • Write quorum: W — when writing, at least W replicas must agree. W > N/2 ⇒ write consistency (only one of two simultaneous writes can get a majority).
  • Read quorum: R — number of peers contacted for a single read. Assuming each value has a timestamp (so older values can be told from newer), R + W > N ⇒ strong read consistency (the reader can't get stale data).

Example: N = 3, W = 2, R = 2 — classic Dynamo/Cassandra configuration.

  • 2 + 2 > 3 ⇒ strong consistency.
  • 2 > 3/2 ⇒ write consistency.
  • Can survive a single node failure (you only need 2 of 3 in either direction).
FROM LECTURES — Relaxing durability

Durability: when a write is committed, the change is permanent. In some cases (sessions, sensor data) strict durability isn't essential and can be traded for scalability:

  • Store data in memory and flush to disk regularly. If the system shuts down, you lose updates still in memory.
  • Write-behind / write-back — ack immediately, replicate later. Fast writes but if the writing node crashes before spreading, durability fails.
  • Write-through — spread to replicas first, ack only after acknowledgement from others. Slower but durable.
FROM LECTURES — BASE concept

BASE is a vague term used as contrast to ACID:

Basically Available
The system works basically all the time. Partial failures occur but without total system failure.
Soft state
The system is in flux — non-deterministic state. Changes happen all the time (replication, repair).
Eventual consistency
The system will be in some consistent state at some time in the future.
EXAM TIP — the canonical answer

When asked "how do NoSQL databases ensure consistency in a distributed setting?" cover:

  1. They relax consistency to gain availability & scale (CAP theorem says you must).
  2. Three flavors of conflict: write-write, read-write, replication.
  3. Tools to handle them: locking (pessimistic) / versioning (optimistic) / quorum (R + W > N).
  4. Most settle for eventual consistency; explicit ACID can still be implemented on a single aggregate.
  5. BASE replaces ACID as the operating philosophy.

The four NoSQL families

FROM LECTURES — Taxonomy

Four canonical families:

  1. Key-value stores — can store any data; if JSON, often extra functionality.
  2. Document databases — structured hierarchical text data (JSON, XML).
  3. Column-family stores — rows with many columns associated with a row key; can be written as JSON.
  4. Graph databases — nodes and edges as first-class citizens.

The first three are aggregate-oriented. Graph is not.

1. Key-value stores

FROM LECTURES — Basics (PA195 L4)
  • A simple hash table (map). Primary use: all accesses via primary key.
  • RDBMS equivalent: a table with two columns — ID and a BLOB value.
  • Operations: put(key, value), value = get(key), delete(key).

Querying

You query by key only. To query by an attribute of the value, you have to read it first — or use an additional index (full-text), after the data has been indexed. Some systems support this (e.g. Riak Search).

Keys

  • Provided by the user (natural unique key — user ID, session ID, cart ID).
  • Generated by some algorithm.
  • Derived from time stamps.
  • Expiration: useful for caches, sessions, shopping carts.

Namespaces / buckets

Group keys logically. e.g. Riak: location = <namespace, key>. Multiple namespaces (UserProfiles, ShoppingCart, Sessions) implement aggregate-based modelling.

FROM LECTURES — Origins & representatives

2007: Amazon Dynamo paper — the foundational distributed key-value store. Today Amazon offers it as a service: DynamoDB.

Other representatives: Redis, Riak, Voldemort, Memcached.

ADDED — when to use a KV store

Good fit:

  • Cache (Redis sitting in front of a DB)
  • Session storage (login tokens, shopping carts)
  • User preferences keyed by user ID
  • Leaderboards (Redis sorted sets)
  • Feature flags / config

Bad fit:

  • Anything where you need to query by something other than the primary key.
  • Complex relationships between objects.

Pick: Redis for in-memory speed; DynamoDB if you want a managed AWS service that scales infinitely (you'll meet it as a DevOps engineer).

2. Document databases

FROM LECTURES — Fundamentals (PA195 L6)
  • Basic concept: document — self-describing piece of data.
  • Hierarchical tree structures — nested associative arrays, collections, scalars.
  • Formats: XML, JSON, BSON.
  • Documents in a collection should be similar; their schema can differ.
  • Documents often stored as values of a key-value — "examinable values," with indexes on chosen keys/fields.

Why document DBs

  • JSON / XML are popular for data exchange.
  • Data from a document DB can be used directly — no ORM.
  • JSON is much closer to the in-memory object representation than relational rows — Object-Document Mapping (ODM) is faster than ORM.
FROM LECTURES — JSON essentials

Text-based open standard for data interchange. ECMA-404. Derived from JavaScript. MIME type: application/json. Language-independent.

Data types:

  • object — unordered set of name:value pairs (properties).
  • array — ordered collection of values.
  • value — string, number, true/false/null, object, array.

No comments (security reasons). No way to specify number precision — depends on parser. JSON Schema exists as a way to specify the schema (also in JSON).

vs XML — XML is mature (DTD/XSD schema, namespaces, DOM/SAX parsing, XPath, XQuery, XSLT), great for config and metadata. XML databases are not considered NoSQL. JSON wins today: compact, easier to write, has all features typically needed.

FROM LECTURES — MongoDB
  • Initial release: 2009
  • Written in C++; open source; cross-platform
  • JSON documents (stored as BSON internally)
  • High performance — many indexes
  • High availability — replication + eventual consistency + automatic failover
ADDED — MongoDB minimal cheat-sheet 
// Insert
db.users.insertOne({ _id: 1, name: "Andrej", roles: ["admin", "user"] });

// Find
db.users.find({ roles: "admin" });

// Find with projection (only specific fields)
db.users.find({ _id: 1 }, { name: 1, _id: 0 });

// Update
db.users.updateOne({ _id: 1 }, { $set: { lastLogin: new Date() } });

// Push to an array
db.users.updateOne({ _id: 1 }, { $push: { roles: "ops" } });

// Aggregation pipeline
db.orders.aggregate([
  { $match: { status: "paid" } },
  { $group: { _id: "$customerID", total: { $sum: "$amount" } } },
  { $sort: { total: -1 } }
]);

// Index for fast lookups
db.users.createIndex({ email: 1 }, { unique: true });
ADDED — when to use a document DB

Good fit:

  • Variable schema per record (CMS, product catalogs).
  • Records you want to fetch as a whole (an order with all its items).
  • Rapid prototyping (no schema migrations).
  • Mobile / JS-heavy backends where data already moves as JSON.

Bad fit:

  • Heavily relational data (lots of joins).
  • Strict ACID transactions across many entities (MongoDB does support multi-doc transactions since 4.0 but performance is the cost).
  • Aggregations across the entire dataset (slower than warehouses).

Pick: MongoDB if self-hosted or self-managed; AWS DocumentDB if you want Mongo API on managed AWS; Couchbase if you want SQL-like syntax over documents (N1QL).

3. Column-family stores

FROM LECTURES — Basics (PA195 L7)

Alternative terms: wide-column, columnar. Don't confuse with column-oriented RDBMS (they're different things).

Data model: rows that have many columns associated with a row key. Column families are groups of related data (columns) that are often accessed together.

  • e.g. for a customer, you usually access all profile info together — but not the customer's orders. Profile and Orders become separate column families.
FROM LECTURES — Data model

Column = a 3-tuple (name, value, timestamp).

Row = a collection of columns attached to a row key. Columns can be added to any row at any time, without having to add them to other rows.

// row
"martin-fowler" : {
  firstName: "Martin",
  lastName:  "Fowler",
  location:  "Boston"
}

Column family = a set of columns containing related data.

{
  // row 1
  "pramod-sadalage" : {
    firstName: "Pramod", lastName: "Sadalage",
    lastVisit: "2012/12/12"
  },
  // row 2
  "martin-fowler" : {
    firstName: "Martin", lastName: "Fowler",
    location: "Boston",  active: "true"
  }
}

Super column family (Cassandra 1.0) — a column family whose columns each contain a map of sub-columns. Used e.g. to model multiple addresses per user (home_address, work_address) under one row.

FROM LECTURES — Representatives
  • Google BigTable — the original.
  • Apache Cassandra — open source; born at Facebook; used at Netflix, Apple, eBay.
  • Apache HBase — open source clone of BigTable; runs on HDFS.
  • ScyllaDB — Cassandra-compatible, written in C++ for performance.

Apache Cassandra is the canonical study target:

  • Data model 1.0 (super columns) vs 2.0 (CQL3 — table-like).
  • CQL (Cassandra Query Language) — SQL-like.
  • Partitioning + replication (peer-to-peer, replication factor 3 typical).
  • Local data persistence: SSTables, commit log, memtables.
  • Indexes, Lightweight Transactions (Paxos-based).
ADDED — when to use column-family

Good fit:

  • Write-heavy workloads (time-series, IoT, sensor data, event logs).
  • Massive scale where peer-to-peer with tunable consistency suits.
  • Data with many sparse columns per row.

Bad fit:

  • Ad-hoc queries (you must design tables per access pattern).
  • Strict ACID transactions.
  • Small datasets — operational complexity isn't worth it.

Pick: Cassandra (self-managed); AWS Keyspaces (managed Cassandra); BigTable if you're in GCP; DynamoDB on AWS is often used instead even though it's KV in spirit.

4. Graph databases

FROM LECTURES — Mission (PA195 L8)

To store entities and relationships between them.

  • Nodes are instances of objects. Nodes have properties (e.g. name).
  • Edges connect nodes; they're directional. Edges have types (e.g. likes, friend).

Nodes are organized by relationships. Allows finding interesting patterns: "Get all nodes that are 'employee' of 'Big Company' and that 'like' 'NoSQL Distilled'".

FROM LECTURES — Graph theory primer

Graph G = (V, E), nodes V (|V| = n) and edges E (|E| = m). Basic operations:

  • Find neighbours of a node.
  • Check if two nodes are connected.
  • Update the graph structure.

Data structures

Adjacency matrix
n × n booleans, A[i][j] = "i connected to j". Good for: checking connectivity, adding/removing edges. Bad for: O(n²) space (sparse graphs waste); O(n) to list neighbours; expensive add/delete of nodes.
Adjacency list
Vector of n lists, each enumerating neighbours of one node. Good for: compact for sparse graphs; cheap to add nodes; fast neighbour retrieval. Bad for: O(log n) edge-existence check (sorted) or O(n) (unsorted).
Incidence matrix
n rows (nodes) × m columns (edges). Good for hypergraphs (one edge connects many nodes). Bad for storage (n × m bits) and slow listing.
Laplacian matrix
n × n integers; diagonal = degree of node, off-diagonal = -1 if connected, 0 otherwise. Used in spectral graph theory.

Most graph DBs use adjacency lists internally because real-world graphs are sparse.

FROM LECTURES — Representatives
  • Neo4j — transactional, the most well-known graph DB. Cypher query language.
  • JanusGraph — non-transactional, distributed; uses Cassandra/HBase as backend.
  • Amazon Neptune — managed; supports Gremlin and SPARQL.
  • ArangoDB — multi-model (graph + document + KV).
EXAMPLE — Cypher (Neo4j) 
// Create nodes
CREATE (alice:Person {name: "Alice"})
CREATE (bob:Person {name: "Bob"})
CREATE (acme:Company {name: "Acme"})

// Create relationships
CREATE (alice)-[:WORKS_AT {since: 2020}]->(acme)
CREATE (bob)-[:WORKS_AT {since: 2018}]->(acme)
CREATE (alice)-[:KNOWS]->(bob)

// Friends-of-friends who work at the same company
MATCH (me:Person {name: "Alice"})-[:KNOWS*1..3]-(friend)-[:WORKS_AT]->(c:Company)
WHERE c.name = "Acme"
RETURN DISTINCT friend.name

// Shortest path
MATCH p = shortestPath((a:Person {name:"Alice"})-[:KNOWS*]-(b:Person {name:"Eve"}))
RETURN p

What's powerful: [:KNOWS*1..3] — "traverse 1 to 3 hops along KNOWS edges" — would be a triple self-join in SQL with painful performance. Native graph engines handle this in linear time of the result.

ADDED — when to use a graph DB

Good fit:

  • Social networks (friend-of-friend, recommendations).
  • Fraud detection (find rings of transactions).
  • Knowledge graphs (Wikidata, enterprise ontologies).
  • Network/IT topology, supply chains, identity graphs.
  • Any data where the relationships are as important as the entities.

Bad fit:

  • Tabular analytical workloads (warehouses are better).
  • Massive write-heavy throughput (graphs are harder to shard).
  • Simple lookups by key (overkill).

Pick: Neo4j if you can self-host; Amazon Neptune on AWS.

FROM LECTURES — Polyglot persistence

Sadalage & Fowler: "using different data stores under different circumstances." Relational DBs aren't going away — they became one option for data storage. Two trends:

  1. NoSQL DBs implement standard RDBMS features (transactions, joins).
  2. RDBMS adopt NoSQL principles (JSON columns in Postgres, sharding extensions).

The end of relational? No. Real systems often combine: Postgres for orders, Redis for cache, Elasticsearch for search, S3 for blobs, Neo4j for the recommendation engine.

Answering the question — point by point

Each phrase from the exam prompt, with a tight answer you could give aloud in 30–60 seconds.

Principles of NoSQL databases
NoSQL ("Not Only SQL") emerged from late-2000s web-scale needs (Amazon Dynamo paper, 2007). Database technologies that are (mostly): not relational, designed for horizontal scale on commodity clusters, schema-less (fields freely added per record), open source. Four NoSQL properties: good scalability (horizontal not vertical), dynamic schema, efficient reading (spend more on writes to keep related data together), low cost (commodity HW). Challenges: maturity, fewer experts, weaker professional support, less standardized data access language. The single biggest conceptual shift from SQL: aggregate data model — designed around the access pattern, not the universal data structure.
Taxonomy of NoSQL databases
Four canonical families:
  1. Key-value stores — opaque values keyed by ID. Simplest, fastest lookups. Redis, DynamoDB, Riak, Memcached.
  2. Document databases — JSON/BSON documents in collections; queryable by fields, with indexes. MongoDB, Couchbase, Firestore.
  3. Column-family stores — rows have many sparse columns grouped into families; data model tuned for time-series and write-heavy workloads. Cassandra, HBase, BigTable.
  4. Graph databases — first-class nodes and edges with properties; fast traversal of relationships. Neo4j, JanusGraph, Neptune.
First three are aggregate-oriented (good for sharding); graph isn't (hard to partition a dense graph). Adjacent categories: time-series (InfluxDB), search engines (Elasticsearch), vector DBs (Pinecone), multi-model (Cosmos DB, ArangoDB), NewSQL (Spanner, CockroachDB).
Distributed databases
NoSQL DBs are typically distributed for scale + availability + locality. Two basic mechanisms:
  • Sharding (data partitioning) — different chunks on different nodes. Goals: keep co-accessed data together (aggregates help), balance load, consider physical location. Many NoSQL DBs auto-shard.
  • Replication — same data copied to multiple nodes. Patterns:
    • Master-slave — one writer, many read-only replicas. Scales reads; master is SPOF until failover.
    • Peer-to-peer — every node accepts writes and spreads them. No master; needs conflict resolution.
Common combination: sharding + P2P replication with replication factor 3 (Cassandra default). Sharding alone makes each shard a SPOF.
Ensuring consistency
Distributed systems can't have all of Consistency, Availability, Partition tolerance (CAP theorem, Brewer 2000 / proved Gilbert & Lynch 2002). Practically, a distributed system must tolerate partitions, so the real trade-off is C vs A. Three conflict types:
  • Write-write — two simultaneous updates. Fix: pessimistic locks or optimistic versioning (version stamps, vector clocks).
  • Read-write — read in the middle of someone's writes. Ideal fix: ACID transactions. NoSQL typically supports atomic writes only within a single aggregate → inconsistency window across aggregates.
  • Replication — same value differs across replicas. Fixes: eventual consistency, quorum (R + W > N for strong reads; W > N/2 for write consistency — classic N=3, W=2, R=2). Read-your-writes via sticky sessions.
Operating philosophy: BASE instead of ACID — Basically Available, Soft state, Eventually consistent.
Architecture of key-value stores
A simple hash map. All accesses by primary key. Operations: put(k, v), get(k), delete(k). Values are opaque blobs (or JSON, which unlocks extra functionality). Keys can be user-provided (natural), generated, or time-derived; often expire. Keys grouped into buckets / namespaces for logical separation. Distributed design (Amazon Dynamo paper):
  • Consistent hashing for partitioning (auto-shard, minimal data movement on node join/leave).
  • Virtual nodes for balanced load.
  • Vector clocks / version stamps for replica consistency.
  • Gossip protocol for membership/liveness (no central registry).
  • 2PC or MVCC for transactions.
Representatives: Redis (in-memory, blazing fast, cache/sessions/leaderboards), DynamoDB (AWS managed, unlimited scale), Riak, Voldemort, Memcached (pure cache, no persistence). Querying limit: only by key — secondary indexes require extra functionality (Riak Search).
Architecture of document databases
Stores self-describing hierarchical documents (JSON/BSON/XML). Documents in a collection are similar but may differ in schema. Indexes on chosen fields make sub-document querying fast. Why preferred over KV: queryable by field. Why preferred over relational: closer to in-memory objects → Object-Document Mapping is faster & simpler than ORM; no joins needed; schema can evolve per document. Representatives: MongoDB (the canonical), Couchbase, Amazon DocumentDB (Mongo API on AWS), Firestore (Google), Cosmos DB (multi-API). Common features: replication + automatic failover, eventual consistency, indexes (including compound, geospatial, text), aggregation pipelines, since recent versions multi-document ACID transactions (with perf cost).
Architecture of column-family stores
Also called wide-column or columnar (not to be confused with column-oriented RDBMS). Data model: rows have a row key and many columns; columns are grouped into column families (data accessed together — e.g. one CF for profile, another for orders). A column is a 3-tuple (name, value, timestamp). Columns can be added to any row without affecting other rows — perfect for sparse data. Super columns (Cassandra 1.0) allowed nesting one level (modern CQL3 replaced this with table-like syntax). Representatives: Google BigTable (the original), Apache Cassandra (open source, peer-to-peer, RF=3, CQL query language, tunable consistency per query), Apache HBase (on HDFS), ScyllaDB (Cassandra-compatible, C++ for performance), AWS Keyspaces (managed Cassandra). Best fit: write-heavy, time-series, IoT, event logs at massive scale.
Architecture of graph databases
First-class nodes (entities with properties) and edges (directional relationships with types and properties). Designed for relationship-heavy queries that are painful self-joins in SQL. Stored internally with adjacency lists (sparse-graph efficient) rather than adjacency matrices (O(n²)). Graph theory primer used in lectures: G = (V, E), basic ops are neighbour lookup, connectivity check, structure update; data structures include adjacency matrix, adjacency list, incidence matrix, Laplacian matrix. Representatives: Neo4j (transactional, Cypher query language, the canonical), JanusGraph (distributed, backed by Cassandra/HBase), Amazon Neptune (managed, Gremlin + SPARQL), ArangoDB (multi-model). Best fit: social networks, fraud detection (rings of transactions), recommendation, knowledge graphs, supply chains, identity graphs. Worst fit: tabular analytics; massive write throughput (graphs are hard to shard).
Basic characteristics of the main representatives
One-liner per family:
  • Redis (KV) — in-memory, single-threaded core, supports pub/sub + sorted sets + streams; default for cache, sessions, leaderboards, rate limiting.
  • DynamoDB (KV / document) — AWS managed, unlimited scale, single-digit-ms latency, tunable consistency, fully serverless billing.
  • MongoDB (document) — JSON/BSON, replica sets with automatic failover, sharding by shard key, since 4.0 has multi-document ACID transactions, aggregation pipelines.
  • Cassandra (column-family) — masterless peer-to-peer, tunable per-query consistency (ALL/QUORUM/ONE), CQL, time-series & write-heavy workloads, the canonical Dynamo-style store.
  • Neo4j (graph) — ACID, Cypher, transactional, native graph storage, free Community Edition. Best of breed for relationship queries.
Practical examples for all of the above
A realistic polyglot persistence stack: Postgres for the orders write model (ACID required); Redis for sessions, rate-limiting, and a read-through cache in front of Postgres; MongoDB for the product catalog (variable attributes per product); Cassandra for event audit logs and time-series telemetry; Neo4j for the recommendation engine ("customers who bought X also bought Y"); Elasticsearch for full-text search; S3 for blobs. Each chosen because its data model fits the access pattern. Combine them with eventual-consistency event streams over Kafka. Sadalage & Fowler: "different data stores under different circumstances."
