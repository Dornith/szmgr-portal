Software life cycle & development process

FROM REPO

Every model contains in some form the phases analysis, design, implementation, testing, and operation (including deployment). Models differ in whether and how they split the project into more digestible parts, and consequently in how development is managed.

INTUITION

Every lifecycle model is an answer to one question: how do we slice the work to manage risk and feedback? Waterfall does one big slice (you find out everything was wrong at the end). Iterative models do many small slices (you find out something was wrong every few weeks and can change). The cost of late discovery is what these models optimize against.

Waterfall

FROM REPO

Consists of strictly ordered phases:

  • Analysis
    • Collecting customer requirements.
    • Distinguish what the customer says they need from what they actually need. Watching how the end user works with the current solution helps.
    • We care about what and why; customers often mention how. Ask why — might be a legitimate reason, might be ignorance.
    • Deliverables: feasibility study, requirements document.
  • Design — architecture, units, technologies, test plan. Deliverables: UML diagrams, wireframes, prototypes.
  • Implementation — build the system per the design.
  • Testing
  • Operation

i.e. first collect all requirements, then design, implement, test, and deploy the SW as a whole.

Pros — easy to manage; cheapest if everything goes smoothly.
Cons — usually nothing goes smoothly; poor reaction to change (you have to go back to earlier phases); customer can't define what they need up front, completely and accurately; in practice the order is rarely followed (we want to test during dev, show progress to impatient customers…).

Incremental model

FROM REPO
  • Project is split into increments developed and delivered in sequence; each gets a simple high-level analysis.
  • Increments are developed in priority order.
  • After deployment of an increment, we have customer feedback on it.

Pros

  • System delivered in parts ⇒ total cost distributed.
  • No need for a huge team — work arrives in chunks.
  • User sees the system early. Reaction to feedback is fast.
  • Change is detected earlier and is cheaper to introduce (just add a change-increment).

Cons

  • Total dev cost is high because of partial deliveries.
  • Demanding planning to distribute work.
  • Modules developed across phases must be glued together — interfaces must be described thoroughly.

Spiral

FROM REPO
  • Combination of iterations and waterfall, with emphasis on risk analysis.
  • Development proceeds in cycles, each with several phases:
    • Analysis
    • Design
    • Implementation
    • Testing, feedback, plan for the next cycle — feedback feeds into the next cycle.
  • Unlike the incremental model, you don't have to have a deployed piece after each iteration (the increment may be requirements, a design, etc.).
  • Cycles are applied to individual waterfall phases too.
  • Better for handling uncertainty — but slower.

Prototyping

FROM REPO
  • Build a prototype to understand how the customer wants to use the system and what they expect.
  • After analysing the prototype we throw it away and start work on the real system using an appropriate model.

"Researcher" / Code-and-fix model

FROM REPO
  • Design and implement. Works? Great. Doesn't? Back to design/implementation.
  • Hard to manage, no documentation, contributors are hard to replace — it's experimentation.

V-process model

FROM REPO
  • Waterfall-like, but it also depicts the testing levels for each phase (unit, integration, system, user, acceptance…).
ADDED

The V-model pairs each build phase with a corresponding verify phase along the V's two limbs:

Requirements ──────────────── Acceptance test System design ────────────── System test Architecture ──────────── Integration test Module ────────── Unit test ▼ Implementation

The point is that each design level is paired with the test level that validates it — you write the acceptance test in the requirements phase, not the test phase.

FROM REPO

Independent of the model: set up communication, define and use a ubiquitous language. To manage anything we need information about current state, plan adherence, expected changes, problems…

The main SW project management methodologies are predictive (e.g. RUP) and agile (e.g. Scrum).

Rational Unified Process (UP, RUP)

FROM REPO
  • Rigid; emphasis on processes.
  • Suitable when requirements are clear and firm; variable aspects can be time and resources.
  • Requires substantial up-front planning.
  • Iterative and incremental; activities (planning, requirements, modelling, design, dev, testing, deployment) partially overlap.
  • Driven by risks and use-case requirements.
  • Architecture is central — there's an architecture team that other teams consult; it serves as a central communication hub (better than many dev teams talking to each other).
  • Allows tight control over processes and team.
  • Good when you need proper documentation (UML diagrams).
  • Suited for large, heterogeneous products and large teams.

Pros

  • Customer not needed during development; the product is anchored in the contract (they know exactly what they'll get).

Cons

  • Fixed deadlines, budget, and functionality (in reality deadline and budget shift somewhat).
  • Change requests are a problem.
  • More planning time required.
  • Complex contract — everything must be considered (exhaustive acceptance criteria, penalties…).

Iterations are grouped into phases:

Inception (1 iteration)
Feasibility, key requirements, risks. Description of significant requirements affecting architecture. Actor identification. Identifying other systems we must communicate with. By the end we know goals, rough architecture, what's used for similar systems, our experiences, chosen technologies, rough price, schedule, and risks. ⇒ Project brief.
Elaboration (2 iterations)
Requirements, architecture; play with UML. End state: architecture and a design that reflects requirements.
Construction (4 iterations)
Building the system, testing, deploying. End state: beta — relatively stable, tested, ready to be used.
Transition (2 iterations)
Bug hunting, manuals, consultations. User testing (beta → change requests), acceptance tests.

An iteration shouldn't exceed 3 months. The output of an iteration is an increment. Each iteration includes workflows (more or less present), each using particular UML diagrams:

  • Business modelling — activity diagram
  • Requirements — use-case diagram
  • Analysis and design — sequence, collaboration diagrams
  • Implementation — class, object, component diagrams
  • Testing — use case, class, activity diagrams
  • Deployment — deployment diagram

RUP is a concrete methodology built on UP (adding team roles, responsibilities, concrete procedures…). UP is the generic framework.

INTUITION

RUP is "structured agility": you iterate (like agile), but each iteration produces formal artefacts (UML, docs) — the rigor of waterfall in micro-cycles. It survives in regulated industries (medical, defence, aerospace) where you must prove you considered things up front.

Agile methodologies & principles

FROM REPO
  • Flexible; emphasis on people.
  • Prefer reacting to change over sticking rigidly to a plan.
  • Try to fix time and resources; functionality is variable ("We built the house and fence; budget left for a garage or a pool. Which do you want?").
  • Good when requirements change, the final shape isn't clear, or the customer wants something tangible relatively soon ⇒ no precise finish date.
  • Requires minimal upfront planning.
  • Communication and collaboration are key.
  • Automated testing.
  • Variable functionality (when time/money runs out we ask the customer to add or drop things).
  • Face-to-face communication, quick meetings — faster and clearer.
  • Simple documentation — docs have a hard time keeping up with reality, so keep them simple and ideally tied to code.
  • Frequent stakeholder meetings (sprint review), demonstrating new functionality (better than describing it).
  • e.g. Extreme Programming (XP) — takes proven practices to extremes (reviews work? do them as much as possible).
    • Pair programming, emphasis on tests, refactoring, code is the single source of truth (generate docs from code, use schemas to generate other things…).
    • Fast feedback, simplicity, small increments.
ADDED — Agile Manifesto (2001)

The four value pairs:

  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan

"That is, while there is value in the items on the right, we value the items on the left more."

The 12 principles include: welcome changing requirements even late; deliver working software frequently (weeks, not months); sustainable pace; regular reflection and adjustment. If asked "what is agile?", citing the manifesto is the canonical answer.

SCRUM

FROM REPO
  • Most commonly used agile methodology.
  • Iterative, incremental.
  • Simple; expects to be combined with other tools/processes.
  • Good for smaller teams (≤15).
  • Good when you have a relatively self-directed team and need to produce something quickly.

Roles

FROM REPO
Product owner
Represents stakeholders, has the best overview of requirements, manages the product backlog.
Scrum master
Responsible for enforcing Scrum and solving process issues.
Development team
3–9 people; self-sufficient (all skills covered) and self-organising; manages the sprint backlog; responsible for delivering the product.

Artefacts

FROM REPO — Product backlog
  • Contains all remaining required functionality as user stories.
    • A unit of functionality, testable, a logical whole.
    • Each story has:
      • Story points representing time effort estimated by planning poker.
      • Acceptance criteria (testable, formulated as Given…When…Then…).
      • May have a risk list.
      • Priority (MoSCoW) by value, effort, risk, benefit…
        • Must — essential
        • Should — ought to have
        • Could — nice to have
        • Won't / Wish — forget it, maybe later
    • For testing, you can use Gherkin / Cucumber (As a… I can… So that…).
  • Built by the whole Scrum team, managed by the product owner.
  • In practice, a board (physical/virtual) of sticky notes.
FROM REPO — Sprint backlog
  • Subset of the product backlog (a set of user stories) to be done in this sprint.
  • Stories are split into tasks; each has an estimate in hours.
  • Each task has phases Todo, In progress, Done.
  • Devs pick tasks themselves; no stories (or tasks) may be added/removed mid-sprint — you'd have to cancel the entire sprint (the product owner does that).
  • Managed by the dev team.
FROM REPO — Product increment
  • All product-backlog items completed during the sprint (i.e. what gets done in the sprint).
  • Built by the dev team, tested by the customer, may be released by the product owner.
  • Must be usable and meet the team's definition of done.

Events

FROM REPO
Project planning
Create the project charter. Build the product backlog. Pick key strategies (communication, risks, change management, quality…).
Sprint planning
Start of sprint, ~8 hours. Whole Scrum team. Set the sprint goal (what we want to deliver), pick items from the product backlog, assign tasks.
Sprint
Iteration focused on developing the sprint backlog. Goal: produce a usable, potentially releasable increment. Whole team participates. PO handles comms, devs dev, scrum master watches process. Analysis, design, implementation, testing. Max 1 month; all sprints are the same length. After the sprint we track team velocity; better future estimates, possibly resize the sprint backlog.
Daily Scrum (standup)
15 minutes, every day; devs and maybe the scrum master. What did I do yesterday, what will I do today, did I hit any problems?
Sprint review
4 hours; whole team + key stakeholders (customer, user). Demo the increment. Discuss issues, changes, questions. Discuss possible product-backlog changes. Possibly recompute the projected finish date. Discuss what to do next; adjust priorities/order.
Sprint retrospective
3 hours; Scrum team. Address processes (work distribution, goal met, anything to improve?), relationships (was it smooth? need closer collaboration?), tools (good communication? enough transparency?), people (anyone struggle? anyone to praise?). Aim for one process improvement to use next sprint. Possibly adjust key strategies, risks.
Project retrospective
Closing the project with the team. Lessons learned, what worked, what didn't… Thanks all round.

When SCRUM can end

FROM REPO
  • Product backlog is empty (everything done), or its contents are no longer considered important (agreed with stakeholders).
  • Time/money runs out.
  • We've done the last sprint and there's stuff to fix but defects are acceptable.
  • Product owner / stakeholder decides to end the project.

Design and architecture can be done continuously for individual user stories, or fixed in the process as a standard (we use a layered architecture, these technologies, etc.).

Contracts are usually based on how many people (and what skills) will work on the project for a given period — great for flexibility, hard to give a final price/deadline. The customer is dragged into development and can give feedback immediately, but this overhead takes extra time.

Balancing Time, Cost, Scope.

Burndown chart

FROM REPO

Shows how much work remains and how we're tracking against the plan.

Team velocity

FROM REPO

Story points completed per sprint; visible as dy/dx of the burndown chart, or as a separate curve.

Planning poker

FROM REPO

For each story, each team member privately estimates; estimates are revealed simultaneously. Discussion follows until the team converges. Recommended points: Fibonacci-like sequence.

INTUITION

Fibonacci numbers are used because estimation gets less precise as size grows. Discriminating between 1 vs 2 is meaningful; between 21 vs 22 is not. Wider gaps at higher numbers reflect that honesty.

SCRUM Board

FROM REPO

Visible to the whole team; sticky notes also show who's working on each task.

ADDED — common SCRUM gotchas examiners ask
  • "Sprint zero" — informal pre-sprint for setting up infrastructure (CI/CD, repos, dev environments). Officially not part of Scrum, but pragmatically common.
  • Definition of Done vs Acceptance Criteria — DoD applies to all stories (passes tests, code reviewed, deployed to staging…); acceptance criteria are specific to each story.
  • Velocity is not productivity. Velocity is for the team's own forecasting. Comparing velocity between teams is meaningless (different point scales) and creates perverse incentives.
  • Scrumban & Kanban — Kanban has no sprints, just a WIP limit on each column ("max 3 cards in In Progress"). Scrumban is a hybrid: sprint cadence + WIP limits + continuous backlog refinement. Good for ops / support teams where interrupts dominate.
  • SAFe, LeSS, Spotify model — scaled-agile frameworks for big orgs. SAFe is heavyweight (lots of ceremony); Spotify's "tribes & squads" model is famously aspirational and only Spotify actually does Spotify.
EXAM TIP

If the examiner asks "compare RUP and Scrum," frame the difference as:

  • What's fixed, what's variable. RUP fixes scope, varies time/cost (sometimes). Agile fixes time/cost, varies scope.
  • How much upfront design. RUP front-loads architecture; Scrum lets it emerge.
  • Customer involvement. RUP isolates the customer behind a contract; Scrum embeds them.
  • Documentation. RUP heavy; Scrum lean.

Deployment and operation

FROM REPO
  • Before deployment, test the system in an environment as close to production as possible.
  • In operation: logging events (so we know what happened before a failure), monitoring, feedback channels are crucial.
  • Deployment includes environment prep (OS, DBs…) — can be automated/simplified with PaaS or Kubernetes.
  • Before going live, check the documentation — it may be stale due to development.
  • Deployment includes user training — to avoid failure due to user reluctance or ignorance.
  • Deployment may include customization for the specific customer (if the system supports it).
ADDED — deployment strategies
Big-bang
Stop the old, start the new. Simple, but downtime + irreversible.
Rolling
Update instances one at a time. Default in Kubernetes Deployments and ECS rolling updates. Zero downtime if health checks are honest.
Blue/Green
Two production environments. Deploy to "green," switch load balancer target, "blue" is the instant-rollback. Cost = 2× infra during the swap. Implemented in AWS with two target groups + ALB listener rule, or with Route 53 weighted records.
Canary
Route 1–5% of traffic to the new version, watch metrics, ramp up. Argo Rollouts, AWS App Mesh / CodeDeploy.
Feature flags / dark launch
Code deploys but is dormant behind a flag (LaunchDarkly, Unleash); marketing controls activation independently of release. Decouples deploy from release.
Shadow / mirror traffic
Send a copy of production traffic to the new version, compare results. Reveals real-world bugs before they affect users.
INTUITION

"Deploy" and "release" should mean different things. Deploy = the binary is running in production. Release = users can see/use the feature. Feature flags separate the two so deployment risk and release risk are decoupled.

Maintenance & reusability

FROM REPO
  • At the end of a project, do an analysis: what worked, what didn't, what to improve?
    • Achieved productivity and quality.
    • Process used, deviations, why.
    • Plan vs reality and reasons (time, money, bugs, FP/LOC…).
    • Risks (plan vs reality, how we handled risks and problems).
    • Effort (also broken down by stage).
    • Defect summary.
    • Causal analysis — performance deviations of the process used (how and why).
    • Technologies used and their evaluation.
    • Document the team and individuals to contact later (e.g. when troubleshooting in another project).
    • Process assets — what arose that can be reused in other projects (libraries, checklists…).
  • Maintenance can be a standalone project; specialised teams may run it.
  • Maintenance handles bug fixes (incl. security), updates, improvements (ideally automated, but user-confirmed); change management (what changed, how, why).
  • Reusability is usually addressed inside services/programs/components, not by reusing structs across projects (unless it's a specialised library) — you'd have problems on change.
ADDED — Lehman's laws of software evolution

Empirical "laws" formulated by Manny Lehman about long-lived systems:

  • Continuing change — a system in use must change or it becomes progressively less satisfactory.
  • Increasing complexity — complexity rises unless work is done to reduce it.
  • Continuing growth — functional content must grow.
  • Declining quality — quality declines unless rigorously maintained.

The maintenance budget exists to fight these tendencies. If you cut maintenance, you don't save money — you defer it with interest.

ADDED — types of maintenance

The IEEE / SWEBOK taxonomy that often appears in exam questions:

Corrective
Fixing observed bugs.
Adaptive
Adjusting to environmental change (new OS, new dependency major version, new browser).
Perfective
Enhancements to functionality or performance based on user request.
Preventive
Refactoring, paying down technical debt before it bites — making future maintenance cheaper.

Industry studies (Lientz/Swanson) found ~60% of maintenance effort is perfective, only ~20% corrective. People imagine maintenance = bug fixing; it's mostly enhancement.

ADDED — reusability mechanisms
  • Libraries / SDKs — the simplest form (Maven artefacts, npm packages).
  • Frameworks — "inversion of control" libraries; you write code, the framework calls you (Spring, Hibernate).
  • Components — deployable units behind a contract (Jakarta EE beans, microservices).
  • Templates / archetypes — Maven archetypes, Terraform modules, Cookiecutter for Python projects.
  • Infrastructure as code modules — e.g. Terraform AWS modules for VPCs reused across projects.
  • Design patterns & reference architectures — reusing solutions, not code.

The DRY-but-don't-overgeneralize tradeoff: a reusable component built without enough real-world variation tends to fit nobody well. The "rule of three" — only extract a reusable abstraction after you've written essentially the same thing three times.

Practical example — running a project

ADDED — end-to-end

Imagine a 6-month project to ship a new internal expense-management tool, Java/Spring backend on ECS, React frontend on CloudFront/S3:

  1. Approach choice — agile (Scrum), 2-week sprints. Requirements are unclear (finance department doesn't know what they want until they see it). Total budget fixed; scope variable.
  2. Sprint 0 — spin up Terraform skeletons (VPC, RDS, ECS cluster), Jenkins pipeline (build, test, deploy), GitLab project, definition of done.
  3. Backlog — build the product backlog with the finance team as PO surrogate. Stories prioritised MoSCoW.
  4. Sprints 1-12 — each sprint produces a deployable increment behind feature flags. Daily standups. Sprint review with the finance manager. Retrospective drives one improvement per sprint.
  5. Deployment — blue/green via ALB target group switch. Smoke tests after every deploy. Canary by feature flag for high-risk features.
  6. Operation — CloudWatch alarms → PagerDuty; logs centralised in CloudWatch Logs / OpenSearch; runbook published in Confluence.
  7. Maintenance — 20% of team capacity reserved for adaptive + preventive maintenance. Quarterly tech-debt review.
