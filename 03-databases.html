Principles of data storage

FROM REPO

Data is stored either directly in the filesystem, or in a database (relational, document, graph…).

Filesystem

FROM REPO
  • Lower system overhead, simpler.
  • Ensuring consistency is harder; we have to deal with file locking; transactional approach is problematic.
  • Access-rights management is more demanding.
  • Data formats must be handled consistently.
  • Worse readability and documentability of the data model.
  • The OS abstracts different filesystems, providing a unified interface for applications.
  • For applications, the filesystem is suitable for large files (PDF, images, video, static pages — unless using a CDN) that are impractical to keep in a DB. Care must be taken not to expose more than intended.

Database system

FROM REPO
  • Independent of the application; a uniform interface for all.
  • Easy security, consistency, concurrent access.
  • Easy readability and documentability.
  • Relational systems correlate with ERD.
  • Declarative approach.
  • Implementing more complex structures may be hard (depends on the system).
  • Relational databases support ACID transactions.
INTUITION

A database is just structured file management with strong guarantees added on top: schema, concurrent-access locking, transactions, durability, an indexed query layer. If you don't need those guarantees, the filesystem is fine. If you need any one of them, rebuilding it yourself is almost always a mistake.

Encoding & compression

FROM REPO

Techniques to transform information into a format efficient for storage or transmission.

  • Lossless compression — original data is fully recoverable from the compressed form (e.g. PNG, ZIP).
  • Lossy compression — part of the data is lost (e.g. JPEG, MP3), but we can achieve far better ratios.

Selected methods (statistical)

  • Huffman coding — assign each symbol a code based on its frequency; minimum redundancy; variable-length codes.
    1. Sort symbols by frequency into a priority queue as nodes.
    2. Take the two lowest-frequency nodes, merge them into a parent whose frequency = sum. Repeat until one node remains. This builds a tree.
    3. Walk from root; left branches marked 0, right 1. The path from root to a leaf is that symbol's code.
  • Shannon-Fano — similar to Huffman, may not be optimal. Walk from the root; sort symbols by frequency and split sequentially into halves (summing frequencies; when over half, split). At each split, mark 0 and 1. Stop when each half holds one symbol.
EXAMPLE — Huffman

Source: BCAADDDCCACACAC — counts: A=5, B=1, C=6, D=3.

[15] / \ [6] [9] C=6 / \ [4] [5]=A / \ D=3 [1] ← wait — corrected: / \ B=1 ...

Variable-length codes for the most common symbols (C, A) are short (1–2 bits); rare symbols (B) get long codes. The prefix property — no code is a prefix of another — means no separators needed.

ADDED — practical compression
  • GZIP / DEFLATE — Huffman + LZ77 (back-references). Default web compression. Content-Encoding: gzip in nginx/CloudFront.
  • Brotli — Google's successor to GZIP, ~20% better for text. Most browsers + CDNs support it.
  • Zstandard (zstd) — Facebook's lossless compressor; tunable speed/ratio. Used in Kafka, MongoDB, ZFS.
  • Snappy — very fast lossless, lower ratio. Default for Cassandra, HBase column blocks.
  • In DBs, page-level compression (Postgres TOAST, Oracle Advanced Compression) saves storage at the cost of CPU on access.

Architecture of relational databases

FROM REPO

Fun fact: what architecture does an RDBMS use? You won't find out in the course "Architecture of relational databases."

A simple architecture could look like:

  • Database server
    • Receives, processes, and answers requests.
  • Relational DBMS
    • Authentication, authorization.
    • Application running over the actual database.
    • Enables creating tables/indexes… data manipulation, reading…
    • Ensures data integrity.
    • Evaluates and executes SQL queries, performs internal optimisations.
    • May do caching.
  • Database — the actual place where data is stored.

The RDBMS may include techniques for managing access rights (restricting operations, row/column-level visibility).

If the question means what does a relational DB consist of, talk about tables, columns, the SQL language for definition (DDL) and manipulation (DML), indexes, (materialised) views…

ADDED — common RDBMS architecture layers

A typical academic breakdown of an RDBMS's internals (e.g. PostgreSQL):

  1. Network / connection layer — listener, session, auth.
  2. Parser — SQL text → parse tree.
  3. Rewriter — expands views, applies rules.
  4. Planner / optimiser — produces alternative execution plans, costs them, picks the cheapest (cost-based optimisation using table statistics).
  5. Executor — runs the plan against the storage engine.
  6. Buffer manager — caches pages from disk in shared memory.
  7. Storage engine — physical layout (heap files, indexes, WAL).
  8. Transaction manager — locks, MVCC versions, WAL writing.

The ANSI/SPARC three-level schema abstraction the textbooks love:

  • External level (views per user role)
  • Conceptual level (logical schema)
  • Internal level (physical storage)

Independence between layers is the whole point of a DB — changing the physical layout (adding an index) doesn't require changing application code.

SQL query language & its parts

FROM REPO

SQL is based on relational algebra. It contains constructs for schema definition, data manipulation, and transactions (see the next sections).

Some systems include procedural programming (PL/SQL — e.g. Oracle).

SQL can include triggers — extra actions on certain statements (INSERT, UPDATE, DELETE). Used e.g. for maintaining a history table, or updating updated_at if the RDBMS doesn't do it.

When working with SQL we use prepared statements to prevent SQL injection.

DDL — Data Definition Language

FROM REPO

Note: different RDBMSs support different types. TEXT isn't in base SQL, but using fixed-length VARCHAR2 is impractical.

Creating a table:

/* block comment */
-- inline comment
CREATE TABLE Products (
    id          INT PRIMARY KEY, -- in pg you can use SERIAL so you don't manage sequences
    cost        INT NOT NULL,
    ean         INT UNIQUE NOT NULL,
    name        TEXT NOT NULL,
    description TEXT,
    created_by  INT NOT NULL REFERENCES User(id),
    updated_at  DATETIME
);

In practice it's better to always generate primary keys — external "unique" values may not stay unique/unchanging. Use numbers, not strings (one compare operation, faster, especially for PKs). Compound primary keys are possible but again slower.

Use SERIAL or AUTOINCREMENT for ID generation (sequences are the older way).

For date/time use DATETIME. If you used INT + Unix timestamp, in 2038 you'd have a problem.

For foreign keys we can specify ON DELETE:

  • CASCADE — deleting a user deletes their products too.
  • SET NULL — sets created_by to NULL — but our NOT NULL would block that.
  • In the current setup you can't delete such a user.

Modifying a table

ALTER TABLE Products ADD picture TEXT;
ALTER TABLE Products DROP COLUMN description;
DROP TABLE Products;

You can use IF EXISTS / IF NOT EXISTS for idempotent scripts (handy while experimenting).

In production use a migration framework with UP and DOWN scripts so changes can be reverted.

ADDED — production migration tooling
  • Flyway, Liquibase — Java ecosystem standards. Versioned SQL files; Flyway records applied migrations in a flyway_schema_history table.
  • Spring Boot auto-runs Flyway on startup if it's on the classpath.
  • Prisma Migrate, Knex, Alembic (Python/SQLAlchemy) — same pattern elsewhere.
  • Expand/contract migration pattern for zero-downtime:
    1. Expand — add new column, keep old. Code reads both.
    2. Migrate — backfill new column. Code writes both.
    3. Switch — code reads new only.
    4. Contract — drop old column.
    Never do a destructive schema change in a single deploy.

DML — Data Manipulation Language

FROM REPO — INSERT 
INSERT INTO Tabulka(sloupec_a, sloupec_b)
VALUES (hodnota_a, hodnota_b);

Integrity constraints are checked; if there's an autoincrement/serial key, you don't have to provide it. The statement usually returns the inserted row (including generated values).

FROM REPO — UPDATE 
UPDATE Tabulka
SET sloupec_a = hodnota_a
WHERE ...   -- often the key

An UPDATE without WHERE updates everything. Integrity constraints are checked.

FROM REPO — DELETE 
DELETE FROM Tabulka WHERE ...

DELETE without WHERE deletes everything.

FROM REPO — SELECT

A beefier select:

SELECT DISTINCT Tabulka.sloupec, B.sloupec
FROM Tabulka
JOIN TabulkaB AS B ON Tabulka.cizi_id = B.id
WHERE price > 0
ORDER BY sloupec ASC

JOIN can be rewritten as a WHERE.

A SELECT result can be placed in parentheses and used in place of a table — data still has a tabular structure.

Between data sets with the same structure you can use set operations: UNION, INTERSECT, MINUS.

In WHERE you can use set membership IN, ranges BETWEEN…AND, logical operators AND, OR… For strings, LIKE where _ matches one character and % matches several.

EXAM TIP — SQL evaluation order

The written order of a SELECT is not the execution order. The logical evaluation order is:

  1. FROM + JOIN
  2. WHERE
  3. GROUP BY
  4. HAVING
  5. SELECT (projection)
  6. DISTINCT
  7. ORDER BY
  8. LIMIT / OFFSET

This is why you can't use a SELECT-list alias in WHERE — WHERE runs before SELECT — but you can in ORDER BY.

FROM REPO — Views

A stored, named SELECT that runs whenever you query it.

  • Modification through a view is limited (e.g. cannot if it contains aggregation, DISTINCT, UNION…) — better to use source tables.

Materialized view

A view whose result is precomputed. Queries return rows directly from the underlying table, but on every change the materialized view must be recomputed (faster reads, slower writes).

FROM REPO — Aggregations

Used with GROUP BY sloupec. If GROUP BY is omitted, aggregates compute over the whole SELECT.

  • COUNT(...) — row count; COUNT(*) works.
  • AVG(...)
  • SUM(...)
  • MIN(...)
  • MAX(...)

HAVING… is like WHERE but allows aggregates.

EXAMPLE — typical join + group-by 
-- Top-selling product per category, last 30 days, only categories with >100 orders.
SELECT  c.name        AS category,
        p.name        AS product,
        SUM(oi.qty)   AS units
FROM    order_items oi
JOIN    orders   o   ON o.id = oi.order_id
JOIN    products p   ON p.id = oi.product_id
JOIN    categories c ON c.id = p.category_id
WHERE   o.created_at >= NOW() - INTERVAL '30 days'
GROUP BY c.name, p.name
HAVING  SUM(oi.qty) > 0
ORDER BY units DESC
LIMIT 20;

Relational algebra

FROM REPO

A relation is a subset of the Cartesian product of domains. In a DB this becomes: domains are the data types of columns, and a table (a set of columns) contains only the combinations (rows) present in the relation.

Relational algebra consists of:

  • Set operations (for union, difference, intersection the relations must be compatible — same header).
  • Projection — column selection.
  • Selection — WHERE.
  • Rename — AS.
  • Join / product of relations — JOIN.
  • Grouping & aggregation — GROUP BY, AVG(...)…

… so the individual operations correspond to SQL.

There are queries you can't express in relational algebra — e.g. the transitive closure.

The transitive closure over a relation is obtained by looking at elements: if a is related to b and b to c, the closure contains the relation of a with c.

Integrity constraints

FROM REPO

Part of DDL. They restrict what values a field can take.

E.g. NOT NULL, UNIQUE, FOREIGN KEY ... REFERENCES ..(..), CHECK(price > 0)…

Specified on the column (ideally), as an additional table-level row, or as a separate expression:

ALTER TABLE .. ADD CONSTRAINT ... NOT NULL (id)

ADDED — constraint taxonomy
Entity integrity
Primary key: NOT NULL + UNIQUE. Every row is uniquely identifiable.
Domain integrity
Column data type + CHECK constraints + NOT NULL. Values must come from the allowed domain.
Referential integrity
Foreign keys: FK values must reference an existing row in the referenced table (or be NULL).
User-defined integrity
Anything else expressed via CHECK constraints, triggers, or application logic (e.g. "order total = sum of items").

Three normal forms you may be quizzed on:

  • 1NF — each column atomic (no lists in a single field).
  • 2NF — 1NF + no partial dependency on a composite key.
  • 3NF — 2NF + no transitive dependency (non-key → non-key).

Mnemonic: "The key, the whole key, and nothing but the key — so help me Codd."

Transaction control

FROM REPO

Transactions in an RDBMS have ACID properties:

Atomicity
The transaction's group of statements is treated as a unit; all execute, or none do.
Consistency
After the transaction the DB is in a consistent state; no integrity constraint is violated.
Isolation
The transaction is isolated from other transactions. Levels can be set; depending on the level, the transaction may error (if it tried to modify the same object as another transaction) or use locking mechanisms.
Durability
After the transaction completes, data is persistently stored.

Transactions are confirmed by COMMIT; rolled back by ROLLBACK (to the state before the transaction started or after the last SAVEPOINT).

ADDED — isolation levels & anomalies

The four standard isolation levels and which anomalies they allow:

LevelDirty readNon-repeatable readPhantom read
READ UNCOMMITTED✅ possible✅✅
READ COMMITTED (Postgres default)—✅✅
REPEATABLE READ (MySQL default)——✅ (— in MySQL/Postgres via MVCC)
SERIALIZABLE———

Anomalies defined:

  • Dirty read — read a value another transaction wrote but hasn't committed.
  • Non-repeatable read — same row read twice in one transaction, different values.
  • Phantom read — same range query run twice, different rows appear (someone INSERTed).

Higher isolation = stronger guarantees, more locking, lower throughput. Pick the weakest level that prevents the anomaly you actually care about.

EXAMPLE — transaction in JDBC 
// Java — transferring money atomically
try (Connection c = ds.getConnection()) {
    c.setAutoCommit(false);
    try (PreparedStatement debit  = c.prepareStatement(
            "UPDATE accounts SET balance = balance - ? WHERE id = ?");
         PreparedStatement credit = c.prepareStatement(
            "UPDATE accounts SET balance = balance + ? WHERE id = ?")) {
        debit.setBigDecimal(1, amount);  debit.setLong(2, fromId);
        credit.setBigDecimal(1, amount); credit.setLong(2, toId);
        debit.executeUpdate();
        credit.executeUpdate();
        c.commit();
    } catch (SQLException e) {
        c.rollback();
        throw e;
    }
}

In Spring you'd use @Transactional on the service method and let the proxy handle commit/rollback.

Indexing

FROM REPO

An index accelerates frequent queries on a table. Queries involving the indexed column (or combination) become faster.

CREATE INDEX my_index ON Products (Price);

Indexes can be:

  • Traditional indexes — like in books, references to rows with a given value. Can have multiple levels, various orderings, etc.
  • Hashes — to get a simple value out of large data.
  • B+ trees — each node contains pointers to lower nodes or values (leaf node). Values in leaves are in ascending order, nodes include interval info about their pointers/values, leaves are linked.
  • R trees — like B+ but multi-dimensional; in 2D they're rectangles. Data sits in leaves. A parent encloses its children (in 2D, the bigger rectangle contains its children). Ideally parent rectangles take as little space as possible — the parent acts as an index that reduces the search area ("look inside me!").

Further index dichotomies:

  • Dense — every row is indexed; takes more space, faster lookup.
  • Sparse — only some rows indexed; less space, slower (the specific row must still be located).
INTUITION

Think of an index as a shortcut sorted by the indexed key. Without one, you scan the whole table (O(n)). With a B+ tree, you binary-search the tree (O(log n)). With a hash index, you go straight to the bucket (O(1) average), but you can only do equality lookups, not ranges.

Cost of an index: writes pay. Every INSERT/UPDATE/DELETE must update every index. Don't index speculatively — profile real query workloads (EXPLAIN ANALYZE in Postgres, EXPLAIN in MySQL).

EXAMPLE — EXPLAIN 
EXPLAIN ANALYZE
SELECT * FROM orders WHERE customer_id = 42 AND created_at > NOW() - INTERVAL '7 days';

-- Without index: Seq Scan on orders ... (cost=0..1000 rows=10) actual time=12.3 ms
-- After CREATE INDEX idx_orders_customer_date ON orders(customer_id, created_at):
--   Index Scan ... actual time=0.04 ms — 300× faster.

A composite index on (customer_id, created_at) is left-prefix usable — it can be used for queries on just customer_id too, but not on just created_at.

Hashing

FROM REPO

The goal of hashing is to convert input data of arbitrary length into output of fixed length (a fixed-length string or number) — the hash. The hash shouldn't allow deriving the input back (one-way); for each input we should deterministically (data-only) produce a single hash. We may also need to minimize collisions (two inputs producing the same hash) — though impossible to avoid completely (we map infinite possible inputs to a finite range). Depending on use, similar inputs should have wildly different hashes, so small (intentional or accidental) modifications can be detected. Hash cracking uses rainbow tables — known inputs and their hashes for a given algorithm.

Hashing is used for:

  • Ensuring data integrity (certificates, checksums)
  • Fast data comparison (HashMap)
  • Comparing data while knowing only the hash (storing password hashes, Argon2)

Collision resistance

  • Weak — given input A, we can't quickly find a different input B with the same hash.
  • Strong — we can't quickly find any two distinct inputs with the same hash.

For different purposes we use different algorithms — a balance of speed (for passwords, slowness can be desirable) and security / collision likelihood:

  • MD5 — relatively fast, not secure (collisions can be found quickly on commodity hardware).
  • Family SHA; currently SHA-2 (SHA256, SHA512, SHA-384…) is considered secure.
  • Argon2 — currently recommended for password hashing.
  • A (dumb but fast) hash can be the length of the input, a modulo, the sum of ASCII codes… (called CRC).
ADDED — hashing inside the database

In an RDBMS, hashing is used in three places:

  1. Hash index — CREATE INDEX… USING HASH (Postgres). Equality lookups only.
  2. Hash join — the planner builds an in-memory hash table on the smaller side, probes it with the larger side. O(n + m) vs nested loop's O(n × m).
  3. Sharding / partitioning — hash of the partition key decides which shard/partition a row lives in (Cassandra, Postgres hash partitions).

For password storage specifically:

  • bcrypt — adjustable cost factor; still acceptable.
  • scrypt — memory-hard; resistant to GPU/ASIC attacks.
  • Argon2id — current OWASP recommendation.

Always combine with a per-user salt (stored alongside the hash); optionally a system-wide pepper (kept separately, e.g. in AWS KMS / Secrets Manager).

Putting it together — practical examples

EXAMPLE — full DDL + constraints + index + transaction 
-- 1) Schema
CREATE TABLE customers (
    id           BIGSERIAL PRIMARY KEY,
    email        TEXT NOT NULL UNIQUE,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE orders (
    id           BIGSERIAL PRIMARY KEY,
    customer_id  BIGINT NOT NULL REFERENCES customers(id) ON DELETE RESTRICT,
    total_cents  BIGINT NOT NULL CHECK (total_cents >= 0),
    status       TEXT NOT NULL CHECK (status IN ('PENDING','PAID','REFUNDED','CANCELLED')),
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 2) Index for the most common access pattern
CREATE INDEX idx_orders_customer_created
    ON orders(customer_id, created_at DESC);

-- 3) Transactional write
BEGIN;
INSERT INTO customers (email) VALUES ('a@b.c') RETURNING id;
INSERT INTO orders (customer_id, total_cents, status)
       VALUES ( currval('customers_id_seq'), 1999, 'PENDING')
       RETURNING id;
COMMIT;
