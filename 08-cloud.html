Cloud Computing

IaaS, PaaS, SaaS cloud. Concepts and principles of designing and developing applications for deployment in the cloud. Cloud services in the context of PaaS cloud, data storage services. Introduction to public cloud

FROM LECTURES — PA200 NIST definition
"Cloud computing is a model for enabling ubiquitous, convenient, on-demand network access to a shared pool of configurable computing resources (e.g. networks, servers, storage, applications, and services) that can be rapidly provisioned and released with minimal management effort or service provider interaction." — NIST

Key features (the 5 NIST essentials)

  1. On-demand self-service
  2. Broad network access
  3. Resource pooling
  4. Rapid elasticity
  5. Measured service

Service models

  • Software as a Service (SaaS)
  • Platform as a Service (PaaS)
  • Infrastructure as a Service (IaaS)

Deployment models

  • Public cloud
  • Private cloud
  • Hybrid cloud

INTUITION — when to pick what

The classic decision tree:

  • Strict OS/kernel needs, custom drivers, legacy software? ⇒ IaaS (EC2 / Azure VM).
  • Standard web app, want zero infrastructure? ⇒ PaaS managed hosting (Elastic Beanstalk / App Service).
  • Microservices, polyglot, complex orchestration? ⇒ managed Kubernetes (EKS / AKS / GKE) or container PaaS (ECS / Container Apps / Cloud Run).
  • Event-driven, sporadic, want pay-per-use? ⇒ FaaS (Lambda / Functions / Cloud Functions).
  • Don't even want to write the app? ⇒ SaaS (Salesforce, Workday, Office 365).

FROM LECTURES — Why a dedicated storage service?
  • Strongly recommended to separate user data/files from the application code.
  • App Service / equivalent has a small persistent disk — not for user uploads.
  • A dedicated storage service gives you: higher capacity, defined redundancy, high availability, high throughput & low latency, scalability.

Azure Blob types:

  • Block Blob — entire-block access. Maps to S3 standard objects.
  • Page Blob — random-access (used for VM disks). Maps to EBS in AWS.
  • Files — SMB / NFS shares, can be mounted concurrently across OSes. Maps to EFS (NFS) / FSx for SMB.
  • Data Lake — analytics-friendly object storage. Maps to S3 + Lake Formation.
FROM LECTURES — Storage account = security boundary

Azure Storage Account contains multiple types of stored data; its primary & secondary keys grant full access to all stored resources!

  • Keys must be protected & stored in a trusted environment. Never expose to the client.
  • Can be disabled and replaced by Azure AD (Entra) authentication.
  • Account name is part of the DNS name.
  • Account-level policies can block public access and enforce data protection / retention.

AWS analog — an S3 bucket has bucket policies, IAM, ACLs, and Block Public Access settings. Same idea: scope the trust boundary tightly and prefer IAM roles over static keys.

Designing cloud applications — challenges

FROM LECTURES — Motto
Classical web applications designed for on-premise environment cannot utilize the full potential of the cloud. Result: higher operating costs.

Effective cloud application — deployed to the cloud and does not make wasteful use of allocated resources.

FROM LECTURES — PaaS design challenges
  • Broad offer of platform services — differences from on-prem/IaaS variants.
  • Poor availability and limited validity of design patterns and best practices — most existing patterns target on-prem / IaaS.
  • Quality conflicts — tactics have both positive and negative effects.
  • Multi-tenant environment — performance unpredictably affected by other tenants' workloads ("noisy neighbour").
  • Cross-service communication latency — devs forget LAN latency in distributed cloud calls.
  • Not all cloud services are designed to scale — some have hard throughput limits.
  • Pay-as-you-go billing — unknown workload patterns produce unpredictable operating costs.
  • Undocumented behaviour — some perf/availability issues aren't in the basic docs.
  • Missing data and tools for performance modeling and prediction — current techniques target on-prem.

Quality concerns — Availability

FROM LECTURES — Availability
  • The cloud platform offers service availability of 99.95–99.99%.
  • Be aware of transient errors — implement detection and retry policy.
  • Handle user errors — damaged data, recovery procedures.
  • Data centre outage — design of a cross-DC cloud application requires complex tactics.

Recommendations

  1. Use services with built-in high availability.
  2. Handle transient faults (retry with exponential backoff).
  3. Consider service recovery scenarios.
  4. Design to run in a restricted mode (degraded but functional).

FROM LECTURES — Chaos Engineering
"Chaos Engineering is the discipline of experimenting on a system in order to build confidence in the system's capability to withstand turbulent conditions in production." — principlesofchaos.org

Key principles:

  • Build a hypothesis around steady-state behavior.
  • Vary real-world events.
  • Run experiments in production.
  • Automate experiments to run continuously.
  • Minimize blast radius.

Tools: Azure Chaos Studio, Chaos Monkey (originally Netflix — the OG; on AWS now called AWS Fault Injection Service / FIS).

Quality concerns — Throughput & Response Time

FROM LECTURES — Throughput

The amount of work an application must perform per unit of time — quantified as transactions / operations / requests per second.

Strongly dependent on the throughput of every component involved in request processing. Early identification of the bottleneck is critical.

Difference between average and peak throughput — you must plan for the peak.

Tools (HTTP benchmarking & load testing)

  • Azure Load Testing — fully managed, high-scale.
  • wrk — modern HTTP benchmarking CLI.
  • AWS equivalents: Distributed Load Testing on AWS (CloudFormation solution), or third-party Gatling/JMeter in CodeBuild.

Recommendations

  1. Be familiar with the real throughput of your cloud services — every service has a limit.
  2. Be aware of services with hard throughput limits (e.g. account-level S3 PUT rate, DynamoDB on-demand burst, RDS connection limits).
  3. Minimize the number of service calls.
  4. Use batch processing — can increase throughput up to 50×.
FROM LECTURES — Response Time

The latency an application exhibits in processing a business transaction. Determined by:

  • Communication latency between services.
  • Request processing time at each service.

Recommendations

  1. Minimize the number of service calls.
  2. Merge simple calls — if you're hitting the same resource, do it in one request.
  3. Optimize service calls — load only the necessary data.
  4. Call services in parallel — when you have independent fetches, fire them concurrently.

Quality concerns — Scalability & Elasticity

FROM LECTURES — Scalability vs Elasticity

Scalability — how well a solution works when problem size increases.

Elasticity — the degree to which a system can adapt to workload changes by autonomically provisioning/deprovisioning resources.

Don't confuse them:

  • Performance attributes specify behavior for a static instance of cloud configuration.
  • Elasticity is the ability to change that configuration in response to load.

Scalability recommendations

  1. Design the application for stateless request processing — horizontal scaling requires it.
  2. Distribute the load — don't over-rely on vertical scaling.
  3. Use scalable cloud services.
  4. Be aware of service scalability mechanics.

Elasticity recommendations

  1. Use horizontal scalability.
  2. Collect and process performance metrics for decision-making (auto-scale rules need signal).
  3. Use auto-scale services where available — saves effort.
  4. Allocate resources before the peak passes — warm-up time matters.
ADDED — auto-scaling in AWS
  • EC2 Auto Scaling Group (ASG) — min/desired/max instance count, scaling policies (target tracking on CPU/memory/custom CloudWatch metric, step scaling, scheduled).
  • Application Auto Scaling — for ECS services, DynamoDB capacity, Aurora replicas, etc.
  • HPA / VPA / Cluster Autoscaler in EKS.
  • Lambda — the platform handles concurrency; you only set a max.

"Warm-up time matters" in AWS practice: ASG default warm-up is 5 min after launch; ECS task launch from cold ECR pull can take 30–90s; Lambda cold starts depend on runtime (~100ms Node, ~1s Java without GraalVM).

Quality concerns — Costs & Security

FROM LECTURES — Operation & development costs
  • Operation costs — service is billed on real resource usage; how do you predict that?
  • Development costs — multiple services offer similar features but differ in critical ways:
    • Storage services differ in functionality (SQL vs NoSQL).
    • Differences in scalability.
    • Integration costs depend on functionality, available libraries, tools.
    • Example: Azure Storage vs Azure SQL Database vs Cosmos DB — same kind of "storage," very different cost & capability profiles.
ADDED — security in the cloud

The lecture doesn't have a dedicated security chapter, but cloud security has its own well-defined shape:

Shared responsibility model
The provider is responsible for security of the cloud (the physical infra, hypervisor, managed services' internals). You're responsible for security in the cloud (your data, IAM, network config, code). The boundary shifts with the service model: IaaS ⇒ you do almost everything; PaaS ⇒ less; SaaS ⇒ very little.
Identity (IAM)
The foundation. Roles + policies, least privilege, MFA, no long-lived keys. AWS IAM, Azure AD / Entra, Google IAM. SSO via OIDC/SAML.
Encryption
At rest (KMS-backed) and in transit (TLS). Customer-managed keys (CMK) for compliance, provider-managed for convenience.
Network
VPC + subnets + security groups + NACLs. Private endpoints to keep data off the public Internet. WAF + Shield against DDoS.
Audit & compliance
CloudTrail / Activity Logs — every API call recorded. SOC 2, ISO 27001, HIPAA, PCI — the provider's compliance carries forward into yours.
Posture management
AWS Security Hub / Azure Defender for Cloud / GCP Security Command Center — aggregate findings, score against benchmarks (CIS, NIST).

CI/CD for cloud applications

FROM LECTURES — Azure DevOps (PA200 Lect 9)

The five components of Azure DevOps:

  • Azure Repos — Git hosting. (≈ GitHub / GitLab repos.)
  • Azure Pipelines — CI/CD. (≈ GitLab CI / Jenkins / GitHub Actions / CodePipeline.)
  • Azure Boards — project tracking. (≈ Jira.)
  • Azure Test Plans — manual + exploratory testing.
  • Azure Artifacts — package management. (≈ Nexus / Artifactory / GitHub Packages.)

Why DevOps? Better feedback from users and production; breaks the wall between devs and IT ops; reduces release errors; documents the release process; gives you visibility into what the application is doing (monitoring).

FROM LECTURES — Pipeline structure

Build pipeline

  1. Pull source code.
  2. Restore dependencies (NuGet, Maven, npm).
  3. Compile.
  4. Run tests.
  5. Publish ≠ deploy — produce a build artifact (package ready for deploy or install).

Release pipeline

  • Input: the build artifact.
  • One release may deploy to multiple environments (Dev, Staging, Production) with different configurations.
  • Secrets are injected at release time, not stored in Git.
  • Entry conditions are configurable — some envs deploy automatically, others require approval.

Provisioning environments

  • For large projects: Infrastructure as Code — ARM templates, Bicep (Azure-native), Terraform (multi-cloud).
  • Or run apps in a standardized environment — Kubernetes, Helm.
  • For small projects: a script you run when provisioning a new env (overkill to automate).

YAML pipelines

  • Pipeline as a YAML file in Git — versioned with code; branches/merges work normally.
  • Triggers: commit to a branch, completion of another pipeline, schedule, manual.
  • Secrets warning — never commit them; inject them via the UI / Key Vault.
  • Builds run on either hosted agents (free minutes per month) or your own (VM / Docker container).

Case studies from the lectures

FROM LECTURES — US Elections (Azure CTO Mark Russinovich, TechEd 2014)

System for presenting US election results. Architecture:

  • Frontend layer — Azure Web Role load balancer + N web-role VMs.
  • Storage layer — Azure SQL DB (Results DB + Main DB).
  • Processing layer — Worker Role VMs continuously processing results.
  • Staging layer — Azure Blob Storage containers, precinct partitions.

Expected load: 6M page views in 1h peak hour = 1,667 page views/sec; each user request hits the DB 10× ⇒ 16,667 DB calls/sec.

Problem: Azure SQL throughput limited to ~1000 req/s.

Solution: Add an in-memory cache layer (worker servers hosting an in-memory cache) — throughput of storage layer increased by an order of magnitude. Result: jumped from 0.22% requests served at peak (with DB only) to 100% (with cache).

AWS analog of the architectural fix: put ElastiCache (Redis) in front of RDS; configure read replicas for further read scale; if even that's not enough, consider DAX in front of DynamoDB or a CQRS pattern.

FROM LECTURES — Smart Thermostats

HVAC thermostats → building router → Azure web role (synchronous HTTP) → Azure DB. Mobile UI on another web role. Email and weather as external services.

Initial test failed with 35,000 connected thermostats. Goal: 100,000–150,000.

Issues:

  1. Synchronous HTTP handler → switched to Service Bus Queue + Worker Role for async writes.
  2. Row-level DB updates → switched to batch updates.
  3. Database tuning.
  4. Queue scalability issues → resolved by partitioning.

AWS equivalent: thermostats → API Gateway → SQS → Lambda batch processor → DynamoDB / RDS in batch writes. Or IoT Core MQTT → Kinesis → Lambda.

FROM LECTURES — Amazon product page rendering

Question from PA200: Do you know how Amazon renders a product page?

Answer: a product page with personalized recommendations is pre-rendered as a fragment and stored in S3. The page transmitted to the user is a simple composition of pre-rendered fragments.

Why: lets read latency stay near-zero at any scale; cost of rendering is amortized.

Practical example — your DevOps stack

ADDED — end-to-end cloud-native app on AWS

A typical Java/Spring service deployed on AWS, hitting every Q8 sub-topic:

  1. Service model choice — ECS Fargate (PaaS-leaning containers) so you don't manage EC2. Maps to "Azure Container Apps" in PA200 terms.
  2. Storage:
    • RDS PostgreSQL Multi-AZ — "Azure SQL Database with built-in HA."

  3. Networking — ALB in public subnets, ECS in private. CloudFront in front of S3 + ALB.
  4. Availability:
    • Multi-AZ ASGs / ECS services.
    • Health checks & circuit breakers in the app (Resilience4j).
    • Retry with jitter for transient errors.
    • Cross-region replication on critical S3 buckets ("RA-GRS").
  5. Scalability:
    • ECS service auto-scaling on CPU + custom CloudWatch metric (e.g. queue depth).
    • RDS read replicas for read-heavy traffic.
    • S3 + CloudFront cope with any read scale already.
  6. Performance & cost:
    • Batch SQS messages where possible (50× throughput tip from PA200).
    • Cache aggressively (Redis, CloudFront).
    • Right-size ECS tasks; use Savings Plans / Reserved Instances for steady baseload; on-demand for peaks; spot for non-critical jobs.
  7. CI/CD — GitLab CI:
    • Build & test job runs in a Docker-in-Docker job.
    • Container pushed to ECR.
    • Terraform plan against staging; manual approval; apply.
    • ECS rolling deploy; CodeDeploy for blue/green if it's a critical service.
  8. Observability — CloudWatch metrics + alarms; CloudWatch Logs (or OpenSearch); X-Ray traces; Grafana dashboards.
  9. Chaos engineering — AWS FIS scheduled monthly: kill a task, drop AZ network, throttle RDS.
EXAM TIP — pick your soundbite

When asked "what's different about designing for the cloud?", reach for one of these:

  • Transient failures are normal. Retry with backoff is not optional.
  • Stateless first. Horizontal scale only works if requests don't depend on local state.
  • Pay-per-use changes economics. Caching, batching, and "off-peak" patterns matter financially, not just for performance.
  • The bottleneck is the slowest service in the call chain. Know your service quotas (the lecture's #1 throughput recommendation).
  • Shared responsibility. The vendor secures the infra; you still secure your app, data, identity, network config.

Answering the question — point by point

Each phrase from the exam prompt, with a tight answer you could give aloud in 30–60 seconds.

IaaS, PaaS, SaaS cloud
Three service models distinguished by where the vendor's responsibility ends and yours begins:
  • IaaS (Infrastructure as a Service) — vendor manages physical HW, virtualization, networking; you manage OS upward. Examples: EC2 + EBS / Azure VM + Managed Disk / Compute Engine + Persistent Disk.
  • PaaS (Platform as a Service) — vendor also manages OS, runtime, middleware; you bring code + data. Examples: Elastic Beanstalk / Azure App Service / App Engine. Includes managed containers (ECS, AKS, Container Apps, Cloud Run) and serverless / FaaS (Lambda, Azure Functions, Cloud Functions).
  • SaaS (Software as a Service) — vendor manages everything; you just use the application. Gmail, Salesforce, Office 365.
NIST's 5 essential characteristics: on-demand self-service, broad network access, resource pooling, rapid elasticity, measured service. Deployment models: public, private, hybrid (and increasingly multi-cloud).
Concepts and principles of designing and developing applications for deployment in the cloud
Classical on-prem apps don't use the cloud well — higher cost, no elasticity. A cloud-native app is designed around the cloud's properties: stateless services (so they horizontally scale), externalized config & secrets (12-factor app), idempotent operations (so retries are safe), immutable infrastructure (replace, don't patch), asynchronous messaging for decoupling. Build for transient failures as normal events (retry with exponential backoff). Minimize cross-service calls and use batch processing where possible (PA200: batching can yield 50× throughput). Account for service quotas. Track real workload patterns for cost predictability.
Cloud services in the context of PaaS cloud
The PaaS layer covers compute, data, integration, ML/AI, observability. Compute: managed web/app hosting (App Service / Elastic Beanstalk), managed containers / Kubernetes (AKS / EKS / GKE / ECS / Container Apps / Cloud Run), FaaS (Lambda / Azure Functions / Cloud Functions). Integration: API gateways, message brokers (Service Bus / SQS / Pub/Sub, Kafka MSK), workflow orchestration (Step Functions, Logic Apps). AI: managed models, embeddings, vision, NLP services. Monitoring: Application Insights / CloudWatch / Cloud Operations. Each PaaS service trades control for ops cost savings.
Data storage services
Pick storage by access pattern, not by hype:
  • Object / blob — S3 / Azure Storage / Cloud Storage. Highly scalable, durable, eventually consistent on overwrites; tiers Hot/Cool/Archive trade access cost for storage cost. Redundancy: LRS / ZRS / GRS / RA-GRS.
  • Block — EBS / Managed Disk / Persistent Disk. Attached to compute, high IOPS.
  • File shares — EFS / Azure Files / Filestore. SMB/NFS, mountable by multiple instances.
  • Relational managed — RDS/Aurora / Azure SQL / Cloud SQL. Automatic backups, failover, read replicas.
  • NoSQL managed — DynamoDB, DocumentDB, Neptune, Keyspaces / Cosmos DB / Bigtable, Firestore.
  • Data warehouse / analytics — Redshift, Athena / Synapse, Databricks / BigQuery.
  • Cache — ElastiCache / Cache for Redis / Memorystore.
A storage account / S3 bucket is a security boundary — protect access keys, prefer IAM roles, block public access by default.
Quality assurance — performance
Throughput = ops/sec the system can handle. Bottleneck is the slowest component in the call chain — know your service quotas. Recommendations: minimize service calls, batch them, optimize each (load only needed data), call in parallel where independent. Caching layers (CloudFront, Redis) are the #1 high-leverage move. Measure with HTTP benchmarking tools (Azure Load Testing, AWS Distributed Load Testing, wrk, Gatling, JMeter). Watch response time = communication latency + processing time; merge calls that hit the same resource.
Quality assurance — scalability
Scalability = ability to handle larger load by adding resources. Elasticity = ability to automatically adjust resources to current workload.
  • Design for stateless request processing — required for horizontal scaling.
  • Distribute load — don't lean on vertical scaling (ceilings + cost).
  • Use scalable cloud services and know their mechanics (some have hard quotas).
  • Use auto-scale with metrics (CPU, request rate, queue depth) — but account for warm-up time. Allocate before the peak hits.
  • Patterns: read replicas, sharding, SQS in front of consumers, CloudFront for static assets.
Quality assurance — availability
Cloud SLA tiers: 99.95–99.99% per service. Three 9s = 8h 45m / year of downtime; four 9s = 53 min; five 9s = 5 min. SLA of composite system = product of component SLAs — three 99.95% services in series ≈ 99.85%.
  • Use services with built-in HA.
  • Handle transient faults with retry + exponential backoff + jitter.
  • Design recovery scenarios; consider running in degraded mode under partial failure.
  • Multi-AZ first; multi-region for the highest tiers.
  • Practice Chaos Engineering (Chaos Monkey, AWS FIS, Azure Chaos Studio) — minimize blast radius, vary real events, automate.
Quality assurance — security
Shared responsibility model — the vendor secures of the cloud (physical, hypervisor); you secure in the cloud (your data, IAM, network, code). The boundary shifts with the service model.
  • Identity (IAM) — least privilege, MFA, roles over long-lived keys, SSO via OIDC/SAML.
  • Encryption — at rest (KMS-backed), in transit (TLS). Customer-managed keys for compliance.
  • Network — VPC + private subnets + security groups + private endpoints; WAF + Shield for public surfaces.
  • Audit — CloudTrail / Activity Log records every API call. Posture management aggregates findings (Security Hub, Defender for Cloud).
  • Compliance — provider's SOC2/ISO 27001/HIPAA/PCI certifications carry forward into your audit story.

Java/Spring service on AWS: ECS Fargate (PaaS containers, like Azure Container Apps); RDS Postgres Multi-AZ for transactional data; S3 for blobs (lifecycled to Glacier); ElastiCache Redis in front of RDS (US Elections case study from PA200 — added cache, throughput grew an order of magnitude); ALB in public subnets, ECS in private; CloudFront in front of S3 and ALB; auto-scaling on CPU + custom queue-depth metric; multi-AZ deployment; daily backup + cross-region replication for the highest-tier data; CI/CD via GitLab → ECR → Terraform → blue/green ECS; CloudWatch alarms + GuardDuty + Security Hub; monthly AWS FIS chaos test that kills a task and drops an AZ.
