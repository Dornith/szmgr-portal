Foundational concepts

FROM LECTURES — PA018 generic security goals

Generic security goals — protection against:

  • access to information by unauthorized parties
  • modification of data by unauthorized parties
  • unaccountable modification/deletion/creation of data by unauthorized parties
  • withholding data or resources from authorized parties
  • false denial of a party's involvement in a given action

Typical security requirements:

Authentication
Originator's identity assured.
Integrity
Information received as originated.
Confidentiality
Information available only to authorized parties.
Availability
Data & resources available when needed.
Non-repudiation
Party cannot deny communication (origin, receipt, delivery).
Access Control
Resources controlled by authorized parties.

The first three (Confidentiality, Integrity, Availability) form the CIA triad — the canonical exam answer when asked "what does information security try to protect?"

FROM LECTURES — Security is not just prevention

Three pillars:

  1. Prevention (protection)
  2. Detection
  3. Reaction

You can't prevent everything; you must also be able to detect compromise and react quickly. Modern security operations are dominated by detection & reaction (SIEM, SOAR, IR) rather than prevention alone.

INTUITION — defense in depth

The mental model: layers of fences. Network firewall, host firewall, app-level WAF, identity-based access, encrypted data, monitoring. The attacker has to bypass every layer; you only need one to catch them. Each layer is imperfect; together they're robust.

Cyber threats & threat management

FROM LECTURES — PA211 cyber threat terminology
Cyber threat
Circumstance or event with potential adversarial impact on operations, assets, individuals, or organizations. A threat actor is a source of malicious activity. Examples: SQL injection, DoS, elevation of privileges, unauthorized access, disclosure/modification of information.
Cyber threat management
A process that detects threats and prevents attacks using internal data (packets, logs, IP flows, scans) and external data (threat intelligence, knowledge bases). Coupled with vulnerability and asset management.
Cyber Threat Intelligence (CTI)
"Threat information that has been aggregated, transformed, analyzed, interpreted, or enriched to provide context for decision-making processes" (NIST). Evidence-based knowledge about hazards to assets. Contains: context, mechanisms, indicators, implications.

CTI sources:

  • OSINT — threat intelligence platforms.
  • Device log files.
  • Network traffic.
  • Human intelligence from incident resolution (alerts, warnings, advisories).
FROM LECTURES — Indicators of Compromise (IoC)

A piece of information that objectively describes an intrusion. Three flavors:

  • Atomic — can't be broken down. Examples: IP addresses, email addresses, vulnerability identifiers.
  • Computed — derived from incident data. Examples: file hashes, regular expressions.
  • Behavioral — collection of computed + atomic indicators, often quantified. Examples: unusual outbound traffic, user activity anomalies.
FROM LECTURES — TTPs (Tactics, Techniques, Procedures)

Describe the behavior of a threat actor.

  • Tactic — highest-level description (e.g. "privilege escalation").
  • Technique — more detailed behavior in the context of a tactic ("vulnerability exploitation").
  • Procedure — highly detailed steps in the context of a technique ("submit crafted input to vulnerable app").

Example — network scanning:

  • Tactic: reconnaissance.
  • Technique: network scanning.
  • Procedure: nmap -sP <ip_address>/<mask>.

Canonical knowledge base: MITRE ATT&CK — a matrix of tactics (columns) and techniques (rows), based on real-world observations. e.g. tactic TA0001 Initial Access contains technique T1190 Exploit Public-Facing Application, T1566.001 Spearphishing Attachment.

FROM LECTURES — Pyramid of Pain

"The relationship between the types of indicators you might use to detect an adversary's activities and how much pain it will cause them when you deny those indicators." Higher in the pyramid = more painful for the attacker to change.

TTPs ▲ Tough! (forces attacker to retrain) Tools │ Network/Host │ Artifacts │ Domain Names │ IP Addresses │ Hash Values ▼ Trivial (attacker rotates daily)

Detecting an attacker by their TTPs is far more durable than by IP/hash — those change instantly.

FROM LECTURES — Standards & enumerations for threats
  • CAPEC (Common Attack Pattern Enumeration and Classification) — hierarchical enumeration of attack categories. e.g. CAPEC-125 Flooding → CAPEC-482 TCP Flood, CAPEC-486 UDP Flood.
  • MITRE ATT&CK — adversarial tactics & techniques.
  • STIX (Structured Threat Information eXpression) — format for CTI; graph of objects (attack pattern, campaign, malware, vulnerability) and relationships.
  • TAXII (Trusted Automated Exchange of Intelligence Information) — HTTPS-based exchange protocol for CTI; any TAXII server must support STIX.
FROM LECTURES — Threat hunting, SIEM, SOAR
Cyber threat hunting
Proactive security search that reveals malicious activity which evaded existing tools; identifies and categorizes potential threats before an attack. Targets Advanced Persistent Threats (APTs). Example: searching logs for whoami, hostname, ipconfig — signs of internal reconnaissance.
SIEM (Security Information and Event Management)
Threat detection and security incident management. Collection and analysis of security events (near-real-time and historical). Core capabilities: log/event collection, multi-source analysis, dashboards/reporting. Examples: Splunk Enterprise Security, IBM QRadar, AT&T USM, Elastic Security, Apache Metron, Security Onion.
SOAR (Security Orchestration, Automation, and Response)
Related to playbooks. Streamlines security ops in three areas: threat & vulnerability mgmt, incident response, security ops automation.

Vulnerability management

FROM LECTURES — Weakness vs Vulnerability vs Patch
Weakness
Flaws, faults, bugs, errors in SW or HW — caused by design, architecture, or implementation. Results in vulnerabilities. Examples: Stack-based Buffer Overflow (CWE-121), Single-Factor Authentication (CWE-308), SQL Injection (CWE-89).
Vulnerability
Concrete instance of weakness in a specific asset; exploited or triggered by a threat source. Examples: Shellshock (CVE-2014-6271), Log4Shell (CVE-2021-44228), Heartbleed (CVE-2014-0160).
Vulnerability patch
A "fix" for the identified problem provided to users. Sometimes published on vendor's website; not always optimal compared to the next product release.
FROM LECTURES — Vulnerability management lifecycle

A process of discovering, analyzing, and mitigating vulnerabilities. Often includes tracking status of each. Vulnerability scanning is only one activity.

  1. Discover — based on asset discovery. Large-scale vulnerability assessment; agent-based (Pakiti); active monitoring (Nessus, OpenVAS); passive monitoring. All use CPE to determine CVEs.
  2. Assess
  3. Analyze
  4. Fix
  5. Measure

Tools: Nessus, OpenVAS, Nexpose (scanners). Full-lifecycle solutions: Qualys, Rapid7 InsightVM, Tenable.sc, F-Secure Radar.

FROM LECTURES — Enumerations & data sources
SCAP (Security Content Automation Protocol)
Synthesis of standards enabling automated vulnerability management. Includes enumerations (CVE, CPE, CCE), scoring systems (CVSS, CCSS), languages (OVAL).
CVE (Common Vulnerabilities and Exposures)
Public list of vulnerabilities. >225,000 entries. Each has an identifier (CVE-YYYY-NNNN), description (no standardized format), references. CVEs assigned by CVE Numbering Authorities (CNAs) within their scope; top-level CNAs are MITRE and CISA ICS. Other CNAs: Google, RedHat, etc.
CWE (Common Weakness Enumeration)
Hierarchical enumeration of common weaknesses in SW/HW. Example: CWE-74 (Improper Neutralization / Injection) has children CWE-77 (Command Injection), CWE-81 (XML Injection).
CVSS (Common Vulnerability Scoring System)
Scoring system evaluating vulnerability properties. Two versions used in NVD: v3.1 and v2. v4 published end-2023.
  • Impact metrics — confidentiality, integrity, availability (CIA).
  • Exploitability metrics (v3.1) — attack vector, attack complexity, privileges required, user interaction.
  • v4 adds — attack requirements (race conditions), passive vs active user interaction. More suitable for IoT/OT.
Criticism — CVSS wasn't designed for prioritization; many high-CVSS vulnerabilities have no public exploit, while less severe but exploitable ones should be prioritized first.
NVD (National Vulnerability Database)
U.S. government repository built on CVE. Adds: impact/severity (CVSS), product info (CPE configurations), CWE reference.
CPE (Common Platform Enumeration)
Identifies a product/version/edition. Example: cpe:2.3:o:linux:linux_kernel:*:*:*:*:*:*:*:*.

Penetration testing

FROM LECTURES — Definition & primer

"A method for gaining assurance in the security of an IT system by attempting to breach some or all of that system's security, using the same tools and techniques as an adversary might." — NCSC.

Synonyms: pentesting, ethical hacking, security assessment.

  • Why? To find security holes before an adversary; assess criticality and risk; report them; provide remediation ideas.
  • Who? Internal or external. Best answer for a CISO: both — internal team for continuous testing, external for unbiased assessments.
  • How often? Depends on the target. Best practice: periodically, driven by product development (not by budget cycles).
FROM LECTURES — The Offensive Security Pyramid
Red Teaming / ▲ Whole system incl. users & defenders. Adversary Emulation │ Few limits. Vulnerability research. │ Goal: test processes, detections, IR. Penetration Testing │ Part of system or app. │ Substantial limits. Find & exploit known. │ Goal: test the technical implementation. Vulnerability Assessment ▼ Part of system or app. Strong limits. Find all known & assess true positives. Easy to automate, scalable.

Bottom = scalable, automated, low-impact. Top = creative, full-scope, expensive. Most engagements live in the middle (penetration testing).

FROM LECTURES — Pen-testing taxonomy
  • By team — Internal vs External.
  • By target — Web, Infrastructure, Mobile, Physical, etc.
  • By access:
    • Black-box — no prior info.
    • Grey-box — some info (design, topology).
    • White-box — full info (source code, technical details).

Duality of a pen-test — technical plane (testing implementation, report to developers) vs social plane (main customer is executive board/management; good communication is essential).

FROM LECTURES — Penetration test lifecycle
  1. Preparation — defining goals and rules.
    • Goals — prioritize most valuable / sensitive parts; filter unqualified, impractical or low-value client requirements.
    • Scope — precise target. Violating the scope is the fastest way to lose contracts and credibility. Example: Czech Police Presidium hired a 3rd party to test their systems; the Ministry of Interior was unaware and accused the tester of intruding on their systems — criminal case followed.
    • Rules of Engagement — time constraints, allowed techniques (social engineering?), special requirements for production, handling sensitive data, reporting rules / recovery plans / responsibility sharing for accidents, NDA.
    • Planning — receive documentation, choose methodology, create test schedule.
  2. Testing:
    1. Reconnaissance — explore system as a user, find documentation discrepancies, scan target, OSINT.
    2. Threat modeling & vulnerability identification — think like an attacker; find ways to achieve goals; try to get the system into an unexpected state.
    3. Exploitation — create PoC exploit. Don't use immediately — could be damaging; consider all scenarios.
    4. Post-exploitation — use new capabilities to dig deeper; iteratively.
  3. Reporting — informing the customer about the results.
FROM LECTURES — Methodologies

Guidelines for different scenarios:

  • OWASP Web Security Testing Guide (and Excel checklist).
  • Open-Source Security Testing Methodology Manual (OSSTMM).
  • Penetration Testing Execution Standard (PTES).
  • NIST 800-115.
  • MITRE ATT&CK — not a methodology per se, but summarizes well-known TTPs.

Required skills: well-versed in CVEs/CWEs; exploitation experience; standard-tech knowledge (so you know what you might disrupt); soft skills; overview of privacy laws; ability to write scripts and understand code (hard-core programming not needed).

Collaboration tools: CoreImpact, CobaltStrike (built-in collab), PwnDoc (open-source collaborative reports), PenteRep.

EXAM TIP

"Two-cents" advice from the lecture — worth quoting in an exam:

  • If you break something, don't delay reporting it to the client.
  • Take notes of anything valuable.
  • Screenshots help, but be careful with info leakage.
  • Use a throwaway environment (sandbox) when possible.

System hardening

FROM LECTURES — Definition
"With system hardening, IT administrators take action to reduce the potential attack surface, leaving fewer opportunities for hackers to exploit your systems. System hardening means doing everything possible to find and fix security vulnerabilities, whether in hardware, firmware, software, applications, passwords, or processes." — Intel
"A process intended to eliminate a means of attack by patching vulnerabilities and turning off nonessential services." — NIST SP 800-152
FROM LECTURES — "Ten Commandments" of system hardening

(Intersection of CIS Controls, NIST 800-123, NSA Network Infrastructure Security Guide.)

  1. Maintain up-to-date software and operating systems.
  2. Configure AAA (Authentication, Authorization, Accounting) and Access Control Mechanisms.
  3. Apply the principle of least privilege.
  4. Password management (strong, unique, not default).
  5. Implement encryption and data backup policies.
  6. Enable malware defense — endpoint, perimeter, app firewalls, IDS/IPS/ADS.
  7. Disable unused ports / services / protocols / accounts / applications.
  8. Enable log management.
  9. Continuous monitoring and vulnerability management.
  10. Perform security audits regularly.

Simplification — in general you need specific rules per component to provide sufficient security.

FROM LECTURES — Sources of hardening guidelines
  1. Vendor guidelines — e.g. Microsoft Security Baselines, AWS Best Practices.
  2. Industry standards — CIS, NIST, ISO 27k, NSA, NÚKIB (Czech).
  3. Other — proactive (security templates) and reactive (monitoring/scanning).

Authorities

NÚKIB / NCISA
Czech National Cyber and Information Security Agency. Publishes reports, legislation, analyses, good-practice guides. Compulsory for admins of important info systems (MUNI), critical infrastructure (hospitals); recommended for the rest. Anchored in Act 181/2014 and Cybersecurity Decree 82/2018.
ISO
International Organization for Standardization. ISO 27k — set of standards for ISMS (Information Security Management System). 27000 (Overview) is free; the rest involve audits & certification (~3000 Kč each).
NIST
U.S. National Institute of Standards and Technology. Two big things:
  • NIST CSF (Cybersecurity Framework) — industry standards & best practices for managing cyber risk. NIST 800-53 and 800-171 implement controls.
  • NIST SP (Special Publications) — 200+ deep-dives covering identity, access control, response, etc.
CIS
Center for Internet Security. Non-profit. Two products:
  • CIS Benchmarks — per-OS / per-app / per-device hardening guides; industry standard (HIPAA, PCI DSS, NIST regard them).
  • CIS Critical Security Controls (CSC) — general recommended practices.
Both are free to download.

CIS Security Levels:

  • Level 1 — necessary basic security; doesn't interfere with functionality.
  • Level 2 — defense in depth; may restrict functionality.
  • STIG (replaced Level 3) — DoD Security Technical Implementation Guides.

ISO 27001 vs NIST CSF: complementary — "you've completed 50% of NIST CSF when you've finished ISO 27001; if you implement NIST CSF, you're 80% to ISO 27001." ISO involves audits and certification (not free); NIST CSF is voluntary (free).

FROM LECTURES — Other standards you might cite
  • PCI DSS — Payment Card Industry Data Security Standard; rules for credit-card world.
  • HIPAA — U.S. federal law protecting health information.
  • SOC — auditing procedure ensuring service providers handle data securely (AICPA).
  • GDPR — EU regulation on personal data.

Access control mechanisms

FROM LECTURES — Discretionary Access Control (DAC)

Owner of a resource decides who else can access it.

Linux file permissions

  • Each file/directory belongs to a user and a group.
  • Each has three basic permissions: r (read), w (write), x (execute for files / traverse for dirs).
  • Permissions apply separately to user / group / others. Numerical: r=4, w=2, x=1. So 755 = rwxr-xr-x.

SUID / SGID

  • SUID (4000) — effective user ID becomes the owner of the executable. /usr/bin/sudo is the canonical example (owned by root, SUID set, so any user gets root privileges just for this binary).
  • SGID (2000) — effective group ID becomes the group of the executable.
  • Security concern — any vuln in a SUID binary ⇒ privilege escalation. CVE-2021-3156 (sudo heap-based buffer overflow) is a real example. Hunt them: sudo find / -type f \( -perm -4000 -o -perm -2000 \) -exec ls -l {} \;
  • Mitigation: mount partitions with nosuid or noexec.

Process IDs

  • Effective ID — the one checked during permission verification.
  • Real ID — the user who started the process.
  • Saved ID — used when a SUID program needs to do unprivileged work.

umask

User file-creation mode mask. umask 0022 → new files get 644, dirs 755. Important for secure file creation.

FROM LECTURES — Access Control Lists (ACL)

More granular than user/group/other — individual permissions for specific users and groups. Supported by Ext2/3/4, ReiserFS, BtrFS, XFS.

Commands: getfacl (read), setfacl (set).

$ setfacl -m g:intern2:rw example       # grant rw to group intern2
$ getfacl example
# file: example
user::rw-
group::---
group:intern2:rw-
mask::rw-
other::---

Mask ACL entry — the maximum access any ACL entry (except file-owner and other) can grant.

Default ACL — applied to a directory; each new object in it inherits the ACL. Works recursively.

Shared directory pattern

$ chmod 3775 example/     # SGID + sticky bit
# Each new object inherits the same group;
# users can't delete each other's files (sticky bit).
FROM LECTURES — Mandatory Access Control (MAC)

Limits abilities of processes to: access/create/delete files, execute other processes, communicate over network, allocate resources. The system enforces the policy — users can't override it.

Two main Linux mechanisms:

AppArmor

  • Debian, Ubuntu, SUSE.
  • Profiles limit processes based on their paths.
  • Only processes with loaded profiles are protected.
  • Three modes: Enforced (block + log), Complain (only log, useful for profile dev), Unconfined (no profile).
  • Profiles in /etc/apparmor.d/.
  • Status: sudo aa-status. Generate new: aa-genprof <path>. Reapply after change: apparmor_parser -r <path>.
  • Docker uses AppArmor to limit containers by default (the docker-default profile).

SELinux

  • Red Hat Enterprise Linux family.
  • Assigns a security context to files, directories, processes (user, role, type).
  • Needs filesystem support to store the context.
  • ls -Z, ps -Z reveal contexts.
  • getenforce / setenforce — Enforcing vs Permissive.
  • Contexts manipulated by chcon, restorecon, semanage.
  • Can limit access to specific ports.
INTUITION — DAC vs MAC

DAC = "my file, my rules". The owner of a resource controls who else can use it. Default in Linux file permissions.

MAC = "the system's rules, regardless of who owns it". Even root can't override an SELinux policy without explicitly disabling it. Used in environments where you don't fully trust the owner of every file (multi-tenant systems, containers, classified data).

Network & transport security mechanisms

FROM REPO — protocols (dev_3_bezpecny_kod)

Reused from the secure-code question because they're as much infrastructure as code:

TLS / SSL
Cryptographic protocol between transport and application. Authenticates parties via certificates and challenge-response (mandatory for server, optional for client by default). Ensures integrity & authenticity via Message Authentication Code (MAC) and confidentiality via symmetric encryption. Initial handshake uses asymmetric crypto to exchange a symmetric session key; the rest of the traffic uses that symmetric key. SSL is the deprecated predecessor; use TLS 1.2+.
IPSec
On the network layer; added to IPv4, mandatory in IPv6. Per IP datagram: sender authentication (header + data hashed with a shared secret in the AH header), data integrity, confidentiality (symmetric encryption in ESP), replay protection (MAC + sequence number). Modes: transport (end-to-end, no NAT-friendly) and tunnel (entire datagram becomes payload, new IP header).
SSH
Remote login + tunneling. Symmetric encryption after handshake; mutual authentication via key pairs. Standard for managing UNIX boxes; widely used for sftp, port forwarding, jump hosts.
SAML / OIDC
SAML — XML-based; old enterprise SSO. OIDC (OpenID Connect on top of OAuth 2.0) — modern. Both enable identity-federated access to many apps with one identity provider (Okta, Azure AD, AWS IAM Identity Center).
ADDED — network-layer defenses you'll be expected to know
  • Firewalls
    • Stateless / packet-filter — decides based on 5-tuple alone.
    • Stateful — tracks connection state, allows return packets.
    • NGFW (Next-Generation Firewall) — adds DPI, app awareness, IDS/IPS, TLS inspection.
    • WAF (Web Application Firewall) — HTTP-level, blocks OWASP-class attacks. AWS WAF, Cloudflare, F5.
  • IDS / IPS — Intrusion Detection / Prevention System (Snort, Suricata, Zeek for IDS).
  • Network segmentation — VLAN at L2, subnets + security groups + NACLs in cloud VPCs. Containment when a host is compromised.
  • Zero Trust — "never trust, always verify." Authenticate every request, even from inside the network. BeyondCorp (Google), Tailscale, Cloudflare Access.
  • VPN — IPsec or WireGuard or TLS-based; encrypts traffic across untrusted networks.
  • DDoS mitigation — CDN (Cloudflare, CloudFront), AWS Shield, scrubbing centers, anycast.
  • DNS hardening — DNSSEC (signed records), DNS-over-HTTPS / DNS-over-TLS (encrypted lookups).
  • BGP security — RPKI (Resource Public Key Infrastructure) to prevent route hijacks.

Putting it together — design of secure systems

ADDED — design principles examiners want you to name
Defense in depth
Multiple independent layers. No single control is the last line of defense.
Least privilege
Each process / user gets only the access strictly required. Default-deny.
Fail securely
If something fails (e.g. auth service is down), default to denying access, not allowing it.
Separation of duties
No single individual can perform critical actions alone. "Two-person rule."
Secure defaults
Out-of-the-box config should be secure; opting in to insecurity, not out of it.
Economy of mechanism
Keep security designs as simple as possible (Saltzer & Schroeder, 1975). Complexity = attack surface.
Complete mediation
Every access to every resource is checked, not cached and reused.
Open design
Security shouldn't depend on the obscurity of the design (Kerckhoffs's principle for crypto).
Psychological acceptability
Security shouldn't make legitimate use significantly harder — or users will work around it.

Practical example — end-to-end

ADDED — securing a typical Java/AWS stack

Imagine an internal Java/Spring service deployed on ECS, fronted by ALB, backed by RDS — how do you actually harden it?

  1. Network
    • Private subnets for ECS & RDS; only ALB in public subnets.
    • Security groups: ALB allows 443 from 0.0.0.0/0; ECS allows 8080 only from the ALB SG; RDS allows 5432 only from ECS SG.
    • NACLs as the second perimeter line.
    • VPC Flow Logs → CloudWatch for forensics.
    • AWS WAF on the ALB with OWASP Core Rule Set.
    • Shield Standard (free DDoS); Shield Advanced for high-risk apps.
  2. Identity / AAA
    • IAM roles with least-privilege policies for the ECS task; no long-lived AWS keys.
    • Application auth: OIDC via Cognito / Okta; MFA mandatory.
    • RDS auth: IAM DB authentication or short-lived credentials from Secrets Manager.
  3. Encryption
    • TLS 1.3 at ALB; ACM-managed certs auto-renewed.
    • RDS encryption at rest with KMS; S3 buckets encrypted (KMS, default SSE-S3 at minimum); EBS encryption on.
    • Secrets in AWS Secrets Manager; never in env vars or code.
  4. Hardening
    • Container images built FROM a minimal base (distroless, Alpine, or Amazon Linux 2023); patched in CI.
    • Non-root user inside the container.
    • Read-only root filesystem where possible.
    • CIS Benchmark for EKS (if Kubernetes).
  5. CI/CD
    • SAST (SonarQube) + SCA (Dependabot) + container scan (Trivy) + IaC scan (checkov, tfsec) gate the merge.
    • Signed commits; signed container images (Cosign).
    • Pipeline runs in an isolated runner; minimal AWS IAM role.
  6. Detection & response
    • GuardDuty enabled (network + DNS + cloud trail anomaly detection).
    • CloudTrail to S3 + Athena; access logs from ALB; app logs structured JSON.
    • Security Hub aggregates findings; routed to Slack / PagerDuty.
    • Runbooks for typical incidents; tabletop exercises quarterly.
  7. Audit
    • Annual external pentest (OSSTMM/PTES-aligned).
    • Continuous vuln scans (Inspector + Trivy).
    • Quarterly patch & CVE review.

Answering the question — point by point

Each phrase from the exam prompt, with a tight answer you could give aloud in 30–60 seconds.

Aspects affecting system security at the level of its infrastructure
Three layers, all needed (defense in depth):
  • Network layer — firewalls (stateful, NGFW, WAF), VPN/IPSec, segmentation (VLAN/VPC/security groups), TLS, DNSSEC, BGP/RPKI, DDoS mitigation.
  • Host layer — OS hardening (CIS Benchmarks), patching, EDR, host firewall, MAC (SELinux/AppArmor), driver signing.
  • Application layer — secure coding, dependency scanning, WAF rules.
Cross-cutting: IAM (least privilege, MFA, no long-lived keys), monitoring & logging (SIEM), incident response, backup & recovery. The CIA triad is the goal: Confidentiality, Integrity, Availability. Plus prevention vs detection vs reaction — you need all three.
Network security
Protecting data in motion and limiting attacker movement.
  • Encryption — TLS between transport and application; IPSec at the network layer (mandatory in IPv6); SSH for remote login; VPN/WireGuard tunnels.
  • Segmentation — VLAN, subnets, security groups, NACLs; Zero Trust ("never trust, always verify") replaces perimeter trust.
  • Filtering — packet, stateful, NGFW; WAF for HTTP-level attacks (OWASP Core Rule Set).
  • Detection — IDS/IPS (Snort, Suricata, Zeek); flow logs.
  • DDoS — CDN scrubbing (Cloudflare, AWS Shield), anycast.
  • Authentication — Kerberos for SSO, SAML/OIDC for federated identity.
Cybersecurity threats
A cyber threat = circumstance or event with potential adversarial impact on operations/assets/individuals (NIST). Threat actors range from script kiddies → organized crime (ransomware-as-a-service, $1.2T global industry) → nation-states (Stuxnet, Ukraine grid). Threats are managed through Cyber Threat Intelligence (CTI) — evidence-based knowledge, sources include OSINT, logs, traffic, IR. Standards: MITRE ATT&CK (tactics/techniques matrix), CAPEC (attack patterns), STIX (CTI format), TAXII (exchange protocol). Indicators of Compromise (IoC) come in three flavours: atomic (IP, hash), computed (regex, file hash), behavioral (anomaly). Pyramid of Pain — TTPs are hardest for attackers to change; IPs/hashes are trivial.
Examples of attacks
Canonical ones to memorize:
  • Stuxnet (2010) — state-on-infrastructure; ICS/SCADA sabotage of Iranian centrifuges.
  • Mirai (2016) — IoT botnet for DDoS; took down DynDNS.
  • SolarWinds (2020) — supply-chain; trojanized Orion update reached ~18 000 customers.
  • Colonial Pipeline (2021) — ransomware via leaked VPN creds without MFA; US fuel shutdown.
  • Log4Shell (CVE-2021-44228) — RCE via JNDI lookup in logged input; massive blast radius.
  • Heartbleed (CVE-2014-0160) — OpenSSL leak; arbitrary process memory readable.
By layer: DDoS, MitM, ARP/DNS spoofing, BGP hijacking at network; privilege escalation, kernel exploits at host; SQLi, XSS, CSRF, SSRF, IDOR, RCE at application; credential stuffing, MFA bypass at identity; dependency confusion, typosquatting at supply chain. Verizon DBIR: ~90% of breaches happen at the web application layer.
Design of security mechanisms
Saltzer & Schroeder principles + standard frameworks. Principles examiners want:
  • Defense in depth — multiple independent layers.
  • Least privilege — each principal gets only what's needed; default-deny.
  • Fail securely — if a check fails, deny.
  • Separation of duties — no single individual can perform a critical action alone.
  • Secure defaults — opt into insecurity, not out of it.
  • Economy of mechanism — keep it simple.
  • Complete mediation — check every access, no caching of authorization.
  • Open design — security shouldn't depend on secrecy of design (Kerckhoffs).
  • Psychological acceptability — if users find it hard, they'll work around it.
Hardening guidelines come from CIS Benchmarks, NIST CSF + 800-series SPs, ISO 27001/27002, NÚKIB (Czech). Access control via DAC (file owner decides), MAC (system enforces — SELinux/AppArmor), RBAC, ABAC.
Vulnerability detection
Process of discovering → assessing → analyzing → fixing → measuring vulnerabilities (the lifecycle). Tools:
  • Vulnerability scanners — Nessus, OpenVAS, Nexpose, AWS Inspector, Qualys.
  • SAST (static analysis) — SonarQube, Semgrep, Snyk Code.
  • DAST (dynamic analysis) — OWASP ZAP, Burp.
  • SCA (dependency scanning) — Dependabot, Snyk Open Source, OWASP Dependency-Check.
  • Container scanning — Trivy, Grype.
  • IaC scanning — tfsec, checkov.
  • Fuzzing — generates random/mutated inputs to find crashes (AFL, libFuzzer).
Enumerations to know by name: CVE (specific vulnerabilities, e.g. CVE-2021-44228), CWE (weakness categories like CWE-89 = SQLi), CVSS (severity scoring v3.1/v4 — impact + exploitability), CPE (platform/product enumeration), NVD (NIST's CVE database with CVSS scores), SCAP (the standard that ties them together).
Penetration testing
Authorized simulation of an attacker to find security holes before a real adversary. Sits in the OffSec pyramid between automated vulnerability assessment (broad, scalable) and red teaming (full-system, creative). Internal team (continuous testing, knows the system) + external team (unbiased, used for compliance) is the ideal. Three access levels: black-box (no info), grey-box (some info), white-box (full info + source). Lifecycle: Preparation (goals, scope, rules of engagement, NDA) → Testing (reconnaissance → threat modeling / vulnerability identification → exploitation → post-exploitation) → Reporting. Methodologies: OWASP Web Security Testing Guide, OSSTMM, PTES, NIST 800-115. Rule of thumb: violating the scope is the fastest way to lose a contract.
Analysis of security attacks
After an attack: incident response + forensics. The IR lifecycle (NIST 800-61): Preparation → Detection & Analysis → Containment, Eradication, Recovery → Post-incident activity (lessons learned). Tools and concepts:
  • SIEM (Security Information & Event Management) — Splunk, IBM QRadar, Elastic Security — collects logs, correlates events, alerts.
  • SOAR (Orchestration, Automation, Response) — runs playbooks; auto-quarantines hosts.
  • Threat hunting — proactive search for attackers that evaded automated detection (looking for whoami, lateral movement, persistence).
  • Forensics — disk imaging, memory analysis, log timeline, chain of custody.
  • Blue team / Red team / Purple team — defend / attack / cooperate to improve detection.
  • Honeypotting — deliberately exposed bait to study attacker behaviour.
Practical examples for all of the above
Hardening a Java/Spring service on AWS: private subnets + tight security groups + WAF on the ALB; Shield Standard against DDoS. IAM roles with least privilege; no long-lived AWS keys; secrets in AWS Secrets Manager; KMS-encrypted EBS/RDS/S3. CIS-benchmark base image, non-root container, read-only root FS. CI pipeline runs Trivy + Snyk + Semgrep + tfsec; merges require 2 reviewers; container images Cosign-signed. Runtime: GuardDuty + Security Hub aggregate findings; CloudTrail logs everything; PagerDuty on alarms. Quarterly external pentest aligned with PTES; bug bounty program; annual ISO 27001 audit.
