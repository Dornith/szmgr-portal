Basic concepts & principles

FROM REPO

A distributed system consists of components (computers) connected by a communication network. Distributed systems solve problems (computations / request handling) by cooperation among individual components (each does something). This makes the system easier to scale (strengthen the troubled subsystem).

Architectures are described in Q1; here's a summary:

Monolithic architecture
Contains everything the system needs; only vertical scaling is possible; poor reliability (one crash crashes everything).
Tiered architecture (not to be confused with layered)
Individual tiers can be distributed, parallelized, swapped (via APIs). Client can be thin/thick depending on provided functionality; classic tiers are client, server, database.
Hexagonal / Microkernel / Component-based
The base app provides minimal functionality; the rest is delivered via plug-in components communicating over predefined APIs. Components can be plugged in at runtime. Alternatively: wrap a legacy system we can't afford to rewrite as a component accessed through our compatible interface. Developing component systems is harder (defining the interface is tricky), but yields better adaptability/reuse. E.g. VSCode extensions; Jakarta Enterprise Edition's Java Beans.
Pipeline architecture
Sequential processing; each component is responsible for a small input→output transformation (do one thing well).
Service-Oriented Architecture (SOA)
(see dedicated section)
Microservice architecture
High cohesion, low coupling between services; system is a large number of small services. Fast inter-service communication matters (gRPC). Services don't share a DB.
FROM REPO — RPC, queues, brokers

Remote Procedure Call allows executing a pre-exposed procedure between processes (even on a remote machine) as if calling it directly in code. The procedure may be implemented in a different language. An interface definition is part of it, from which callable functions/structs for the calling language are generated. The de-facto standard is gRPC. For fullstack TypeScript apps, tRPC is popular. In PA053 the course discussed CORBA (focused on Java but also supporting other languages; not just RPC but an architecture for inter-object communication in a distributed environment), but it's largely unused in new systems due to complexity / better alternatives — replaced by simpler SOAP and REST, or RPC (gRPC, on HTTP/2, binary-serializing via Protocol Buffers).

For communication in DS, besides RPC, we use message queues and event brokers that allow publish-subscribe communication, or one-of-many consumption, and can persistently store messages (handy for transactional processing, reliability against outages). Distinguish:

  • Queue — classic queue; we expect one consumer per message.
  • Topic — message goes to all listeners.

e.g. Apache Kafka, RabbitMQ.

Alternatively for communication: REST, or (for low-level control and perf) raw socket communication.

FROM REPO — Cloud

Advantage: use a platform/infra as a service without having to operate it / make a costly initial investment. Compute power can be (auto)scaled based on load. Physical resources can be shared, lowering cost and allowing compute demand distribution (peaks in different apps at different times can be served by one machine). Data centres can be chosen by proximity to customers.

FROM REPO — Grid computing

Computing very demanding tasks across many resources (e.g. volunteer-based Folding@home). A resource can be CPU, storage, special device…

FROM REPO — Batch vs Stream processing

In batch processing we distribute via individual jobs; job scheduling matters (a queue may suffice).

Stream: e.g. Apache Kafka.

FROM REPO — MapReduce

Data is transformed via MAP (1:1) and REDUCE (N:1). Mappers are trivially parallel (same or different ops); reducers a bit trickier — parallelize different ops. e.g. Apache Hadoop.

ADDED — distributed systems theory you may be asked
CAP theorem (Brewer)
In presence of a network Partition, a distributed data store can only guarantee one of Consistency (every read sees the latest write) or Availability (every request gets a non-error response) — not both. In normal operation you can have CA; the choice CP vs AP only matters during a partition.
PACELC
Refinement: if Partition, choose A or C; Else (no partition), choose between Latency and Consistency.
FLP impossibility
In an asynchronous system with even one faulty process, no deterministic consensus algorithm exists. Practical algorithms (Paxos, Raft) add timing assumptions or randomness.
Two Generals Problem
Two armies can't reliably coordinate over an unreliable channel even with ACKs of ACKs. The reason exactly-once delivery is impossible in the general case — we settle for at-least-once + idempotency, or at-most-once.
Eight fallacies of distributed computing
Things people wrongly assume: (1) network is reliable, (2) latency is zero, (3) bandwidth is infinite, (4) network is secure, (5) topology doesn't change, (6) there is one administrator, (7) transport cost is zero, (8) the network is homogeneous. Memorise this list — it shows up.

Centralized vs distributed

FROM REPO

The main difference: centralized architecture concentrates data and logic in one place; distributed disperses logic across multiple independent components (possibly on separate machines) that communicate.

Disadvantages of centralized & how to overcome

  • No horizontal scaling ⇒ scale vertically.
  • A part fails ⇒ the whole fails ⇒ redundancy, hot/cold spares.
  • Low flexibility, high coupling ⇒ emphasize code quality.

Disadvantages of distributed & how to overcome

  • Overall complexity, harder admin.
  • Need more / more complex communication, harder sync, latency-sensitive ⇒ use message queues, gRPC, eventual consistency.

Versus centralized, distributed systems:

  • Don't have ACID transactions but BASE:
    • BAsically available — a part not working doesn't make the whole unusable; the rest functions. e.g. Netflix's search may be broken but everything else runs fine. Every query gets some answer.
    • Soft state — changes in the system can happen even without incoming queries — the system propagates data to reach consistency.
    • Eventually consistent — data may not be consistent immediately after a query response but will be after a while.
  • Failure of a part doesn't kill the whole.
  • More flexible to modify thanks to low coupling.
INTUITION — ACID vs BASE

Translation: a relational DB transaction says "I'll wait until I'm sure of the right answer." A BASE-style distributed system says "Here's an answer I'm pretty sure about; I'll fix it later if I'm wrong." Banking still picks ACID. Twitter, Instagram, and most read-heavy social apps pick BASE.

Replication & data sharing

FROM REPO

Distributed systems use replication for various reasons. In distributed databases (Apache Cassandra) / filesystems (Apache Hadoop) this is for security / availability / outage prevention; in general we use replication to ensure faster responses. A central DB used for sharing data can become the bottleneck → use a distributed DB that handles replication internally, or multiple DBs that may be simpler (MongoDB), because by distributing data we drop ACID and operate with BASE. Caching (Redis) also creates a form of replication. With replication we must handle data invalidation after changes (timeout, or CQRS).

Replication is desirable for content delivery networks (CDN), where we keep static assets (web, images) as close to the user as possible for fast loading.

NoSQL DBs typically have mechanisms for automatic replication/distribution between nodes. Sharding can be used (break data; each node owns its domain) for data distribution, and/or master-slave replication for read scaling (apps with frequent reads) and outage prevention (master crashes, one slave becomes the new master) — the master pushes write updates to the slaves.

For data sharing (events), Apache Kafka can be used — a platform for streaming data stored in logs. For sharing info about services in a DS, you can use Apache ZooKeeper.

ADDED — consistency models you might be asked to compare
  • Strong / linearizable — reads always see the most recent committed write. Feels like a single machine.
  • Sequential consistency — all operations appear in some global order all clients agree on, but not necessarily the real-time order.
  • Causal consistency — writes causally related (one happened-before another) are seen in order by all; concurrent writes may be seen in different orders.
  • Eventual consistency — if no new writes occur, all replicas eventually converge.
  • Read-your-writes / monotonic reads — session-scoped weaker guarantees.

Replication patterns:

  • Single-leader (a.k.a. primary/replica): one node accepts writes, replicates to others. Simple; the leader is a write bottleneck and SPOF until failover.
  • Multi-leader: multiple nodes accept writes (e.g. per-region). Needs conflict resolution.
  • Leaderless / Dynamo-style (Cassandra, Riak): clients write to any N replicas; reads use quorum (R + W > N for strong consistency).

Service-Oriented Architecture (SOA)

FROM REPO

A hybrid between microservices and monolith, a fairly pragmatic architecture for scalable systems. The system is made of mutually independent, independently deployable, and mutually communicating services (typically 4–12) that contain cohesive units of functionality (e.g. users, product management, orders, payments, mail…). Each service can be scaled independently. A facade layer can sit over the services to provide a unified API. Services often share a database, but it can be broken into multiple independent ones. Harder to develop, but the whole system is more maintainable and scalable thanks to modularization.

Modern view: microservices are SOA done with lessons learned (smaller services, lighter integration, polyglot, automation-heavy).

Web services

FROM REPO

Components enabling communication and interaction via standardized protocols and formats. Based on Service-Oriented Architecture. Web services abstract their functionality behind a web API. A definition language formally describes the schema/interface of a service; client code can be generated for various languages to ease use. The schema can be generated from source code via annotations.

Historically: web services on SOAP (Simple Object Access Protocol) + XML, defined via WSDL (Web Service Definition Language).

Now: REST (Representational State Transfer; not a protocol but an architectural style) + JSON (other formats possible), defined via OpenAPI Specification; or GraphQL (+ JSON) with GraphQL Schema and its query language. GraphQL uses one entrypoint (+1 playground) and allows precise specification of desired data (down to field level), solving over-fetching (one query returns extra) and under-fetching (one query lacks data, forcing multiple).

SOAP is transport-independent; REST uses HTTP. REST is simpler, faster, more efficient. SOAP allows targeting one message to many recipients, transit through intermediaries (who can process the header; the body is for the recipient). SOAP supports exchanging structured, typed XML data; the SOAP header (optional) can carry metadata, QoS, security info, SAML data, session identifier (cookie)…; the SOAP envelope is the root XML element (namespace = protocol version, encoding); the SOAP body contains the actual message. REST allows linkage (via hyperlinks) and can take you off our system entirely.

REST views the web as resources addressable by URL that return a representation (HTML, XML, PNG, JSON…). Receiving data transitions the client to a state which can be transformed by accessing another resource. Stateless — every message contains everything needed to interpret it (correctly, the message shouldn't contain a cookie but e.g. a JWT). Queries are cacheable.

ADDED — REST maturity (Richardson Model)
  1. Level 0 — "the swamp of POX": one URL, one verb (POST), XML/JSON tunneled inside. Essentially RPC over HTTP.
  2. Level 1 — Resources: multiple URLs (/users/42), but still mostly POST.
  3. Level 2 — HTTP verbs: GET reads, POST creates, PUT updates, DELETE deletes. Most so-called REST APIs stop here.
  4. Level 3 — HATEOAS (Hypermedia as the engine of application state): the response contains links to related actions, so clients can navigate dynamically. Rare in practice.
EXAMPLE — same operation, three styles 
// REST
GET  /users/42/orders?status=pending HTTP/1.1
Authorization: Bearer …

// GraphQL
POST /graphql
{
  user(id: 42) {
    orders(status: PENDING) { id total }
  }
}

// gRPC (over HTTP/2, binary)
// .proto:
// rpc GetUserOrders(GetUserOrdersRequest) returns (OrderList);
// client call:
GetUserOrders(user_id=42, status=PENDING)

Choose REST for public APIs, GraphQL for client-controlled fetching (mobile, dashboard apps), gRPC for high-perf internal service-to-service.

ADDED — communication patterns in DS
Synchronous request/response
REST, gRPC. Simple, low-latency, tightly coupled to availability of the callee.
Asynchronous messaging — queue
RabbitMQ, SQS. Producer doesn't wait. Decouples availability. Each message processed by exactly one consumer.
Pub/Sub — topic
Kafka, SNS, MQTT. One publisher, many subscribers each get a copy.
Event sourcing
State is derived by replaying an append-only event log. Strong audit trail; rebuilding views is easy.
CQRS — Command/Query Responsibility Segregation
Separate write model from read model. Often pairs with event sourcing.
Saga pattern
Distributed transaction as a sequence of local transactions with compensating actions on failure. Used because 2-phase commit is unavailable across microservices.

Practical examples

EXAMPLE — splitting a monolith into services

Suppose you have a monolithic e-commerce app (Java/Spring). You decide to extract three services:

  1. Catalog Service — products, categories. REST API; reads dominate; replica DB.
  2. Order Service — orders, line items. REST + emits events on order state changes to Kafka.
  3. Payment Service — subscribes to order.placed events; charges via Stripe; publishes payment.completed or payment.failed.

The order saga: Order Service creates an order in PENDING, publishes order.placed. Payment Service charges and publishes a result. Order Service consumes the result and either marks the order PAID or runs the compensation (cancel reservation, notify customer). No 2PC, no shared DB — consistency by message correlation + idempotent handlers.

Deployment: each service is a container in ECS / EKS; talks to others via service discovery (AWS Cloud Map / k8s service); shared infra (Kafka MSK, Redis Elasticache) provisioned by Terraform modules; observability via OpenTelemetry → CloudWatch / Grafana; alerts on lag in Kafka consumer offsets and on Saga compensations / second.

EXAM TIP — anti-patterns
  • Distributed monolith — many services, all must be deployed together. Worst of both worlds.
  • Shared database between microservices — couples them via the schema. Every service should own its data.
  • Chatty interfaces — one user action triggers 50 hops between services. Use aggregation (BFF, API Gateway) or coarser-grained APIs.
  • Synchronous fan-out — A calls B calls C calls D synchronously. The latency adds; reliability multiplies (in the wrong direction).

Answering the question — point by point

Each phrase from the exam prompt, with a tight answer you could give aloud in 30–60 seconds.

Basic concepts, principles
A distributed system = multiple components on networked machines cooperating to solve a problem. Each does part of the work, so the system scales by reinforcing weak subsystems. Key building blocks: RPC (call a procedure as if local — gRPC over HTTP/2 with Protocol Buffers), message queues / event brokers (Kafka, RabbitMQ — async, persistent, decouple producers from consumers), service discovery, load balancing. Operational concerns dominate: latency, partial failure, eventual consistency.
Difference between centralized and distributed system architecture
Centralized concentrates data and logic in one place; distributed disperses logic into multiple components that communicate over a network. Centralized systems are simpler, can use full ACID transactions, but they don't scale horizontally and a single failure brings everything down. Distributed systems scale horizontally, fail partially, but pay in complexity and consistency. The major architectural styles fall on a spectrum: monolith → tiered → SOA → microservices, increasing in distribution and independence.
Disadvantages of both and how to overcome them
Centralized cons: no horizontal scaling → scale vertically (bigger hardware) but you hit a ceiling; failure of part = failure of whole → add redundancy / hot spares; tight coupling reduces flexibility → focus on code quality (SOLID). Distributed cons: overall complexity; harder admin; needs more / more complex communication, synchronization, and handling of latency → use message queues + gRPC for fast comms, accept eventual consistency, design for partial failure with retries / circuit breakers / sagas. Also expect to give up ACID and live with BASE (basically available, soft state, eventually consistent).
Replication
Copying data to multiple nodes for availability (survive failures), read scalability (more replicas absorb more reads), and locality (CDN serves users from their region). Patterns: single-leader / master-slave (one accepts writes; simple; leader is SPOF until failover), multi-leader (multiple regional writers; needs conflict resolution), leaderless / Dynamo-style (any replica accepts writes; uses quorums R+W>N for strong consistency). Always need an invalidation strategy (TTL, CQRS events).
Data sharing
Mechanisms to share data between services or instances. Options: shared database (simple, but couples services via schema — anti-pattern in microservices), distributed database (Cassandra, Spanner — handles replication internally), polyglot persistence (each service picks its own DB), messaging (Kafka publishes events, consumers build their own read models — CQRS / event sourcing), caching layer (Redis), CDN (static assets). Apache ZooKeeper for sharing service-discovery / coordination data.
Service-oriented architecture (SOA)
A pragmatic middle ground between monolith and microservices: the system is built from 4–12 independently deployable, communicating services with cohesive responsibilities (users, orders, payments, mail…). Each service can be scaled independently. A facade layer can give clients a unified API. Services often share a database (unlike microservices). Allows full ACID transactions in many cases. Good for domain-driven design without microservices overhead.
Web services
Components communicating via standardized web protocols. Three main styles:
  • SOAP + XML — old enterprise standard; transport-independent; defined by WSDL; rich SOAP headers carry metadata, security, sessions. Strict typing.
  • REST + JSON — current default; not a protocol but a style on top of HTTP; defined by OpenAPI; stateless; cacheable; resources identified by URLs, manipulated via HTTP verbs.
  • GraphQL — single endpoint, client describes exactly what fields it wants → solves over-/under-fetching; defined by GraphQL Schema.
For low-latency internal calls: gRPC (binary, HTTP/2, code-generation from .proto).
Examples of existing technologies and their use
  • RPC / API gateways — gRPC for internal services; REST/GraphQL for public APIs.
  • Messaging — Apache Kafka (event streaming, persisted log), RabbitMQ (classic queue), AWS SQS/SNS/EventBridge.
  • Distributed DBs / caches — Apache Cassandra (column-family), MongoDB (document), Redis (cache).
  • Stream processing — Kafka Streams, Apache Flink.
  • Batch & MapReduce — Apache Hadoop, Apache Spark.
  • Service discovery / coordination — ZooKeeper, etcd, Consul; cloud-native: AWS Cloud Map, Kubernetes Service.
  • Orchestration — Kubernetes, ECS.
  • Grid / volunteer computing — Folding@home.
Practical examples for all of the above
Splitting an e-commerce monolith into three services: Catalog (REST, read-heavy, replica DB), Order (REST + emits order.placed events to Kafka), Payment (subscribes to order.placed, charges via Stripe, publishes payment.completed/failed). No 2PC: the order saga pattern handles consistency via correlation IDs and compensating actions. Deployed as containers on ECS; service discovery via AWS Cloud Map; observability via OpenTelemetry → CloudWatch. Anti-patterns to avoid: distributed monolith (services must deploy together), shared DB, chatty synchronous fan-out.
