Part A — UNIX

Boot & kernel start (PV065)

FROM LECTURES — Boot sequence (PV065 Ch.5)
  1. Firmware — stored in ROM. On PC: BIOS. Tests hardware, loads the OS from external media, often provides a command line (PROM monitor), serial console.
  2. Primary bootloader — in the boot block of the disk; fixed length; loads the secondary bootloader. On PC: master boot record (incl. partition table).
  3. Secondary bootloader — loads the kernel and passes it parameters. Some provide a CLI; some can read filesystems (boot arbitrary file); uses firmware to load the kernel.
  4. UEFI — a tiny OS by itself: recognises partitions, has a filesystem (VFAT), executable format, EFI shell, EFI environment variables, secure boot. Linux: grub.efi, linux.efi.
  5. Kernel parameters — system console, root disk, device-driver parameters, other params passed to user space. bootparam(7).
FROM LECTURES — Kernel initialization (PV065 Ch.5)
  1. Virtual memory — as soon as possible.
  2. Console init (sometimes two-phase: early_printk()).
  3. CPU init.
  4. Bus init (autoconfig devices).
  5. Device init.
  6. Create process 0 (idle task / swapper / scheduler).
  7. Start kernel threads (kflushd, kswapd…).
  8. Init the other CPUs and start their idle processes.
  9. Mount the root filesystem.
  10. Start process 1 — usually /sbin/init.
  11. From here on: user space.

Initial RAM disk (initrd / initramfs)

  • Loaded into memory by the secondary bootloader together with the kernel.
  • The kernel doesn't have to contain any drivers other than for console + the FS that lives on the ramdisk.
  • Modules are linked in; the ramdisk gets unmounted later.
  • Then the boot continues by mounting the real root FS and starting init.

Boot messages: dmesg(8), also stored in /var/log/dmesg or /var/log/boot.msg.

FROM LECTURES — Kernel architecture (PV065 Ch.5)
Monolithic kernel
One file on disk. All used drivers live inside the kernel. Often no autodetection. Memory equally accessible to all kernel parts. One bad part (a buggy driver) can crash another. Classic Linux example.
Microkernel
"What doesn't have to be in the kernel goes outside." Processes (servers) for VM management, device control, disks, etc. Well-defined operating conditions (theoretically — in practice DMA, SMI, IOMMU complicate things). Message passing has lower throughput, higher latency. Examples: CMU Mach, OSF Mach, L4, Minix, Windows NT HAL, QNX, VxWorks. (Historical aside: "Linux is obsolete" — the 1992 Tanenbaum vs. Torvalds debate.)
Modular monolithic (Linux today)
Linux is monolithic but supports loadable kernel modules (insmod, modprobe) — best of both.
INTUITION — monolithic vs micro

Monolithic: one giant shared address space; functions calls between subsystems. Fast, but a bug anywhere can crash everything. Microkernel: kernel only handles IPC and scheduling; drivers and filesystems are userspace servers. Robust, but message passing is slower than a function call. Real-world OSes are hybrid (Linux = modular monolith; macOS XNU = monolithic on top of a Mach microkernel base; Windows NT is conceptually hybrid).

Memory management

FROM LECTURES — Kernel memory model (PV065 Ch.5)

The kernel uses virtual memory, the same abstraction it provides to processes. Each process sees its own address space — text, data, BSS, heap, stack — backed by physical pages via the MMU's page tables and the TLB.

Course motto (Rik van Riel): "Virtual memory is like a game you can't win; however, without VM there's truly nothing to lose."

ADDED — VM concepts the exam may probe
  • Paging — address space split into pages (typically 4 KB on x86). Each page maps to a physical frame or is on disk (swap).
  • Page table — per-process structure storing the mapping. Multi-level (x86_64 uses 4 levels).
  • TLB (Translation Lookaside Buffer) — cache of recent VA→PA translations.
  • Page fault — CPU access to a page not in memory: the kernel maps it in (demand paging) or swaps it from disk.
  • Copy-on-write — after fork(), parent & child share physical pages; the first write triggers a copy. Makes fork+exec cheap.
  • Memory-mapped files (mmap) — file pages mapped into a process's address space; read/write becomes a memory access. Powers shared memory between processes and large-file processing.
  • ASLR (Address Space Layout Randomization) — security feature; randomizes positions of stack, heap, libraries to mitigate exploits.
  • Swap — backing store on disk for pages evicted from RAM under pressure.
  • OOM killer — when memory is exhausted Linux picks a process to kill based on a heuristic (oom_score).

Device management

FROM LECTURES — Devices (PV065 Ch.5)
  • UNIX v7 — static block / character device tables (bdevsw[], cdevsw[]).
  • Linux — block / character / SCSI / network devices, dynamic tables.
  • Device services — open, read, write, control operations, private device data.
  • Runtime reconfiguration — hot-plug / hot-unplug (USB etc.).
ADDED — Linux device model
  • Devices visible in /dev as block (random-access, e.g. /dev/sda) or character (sequential, e.g. /dev/tty).
  • udev (in /dev) — userspace daemon that creates device nodes on hot-plug events; rules in /etc/udev/rules.d.
  • sysfs — /sys exposes the kernel device tree as files.
  • procfs — /proc exposes kernel and process info as files (/proc/cpuinfo, /proc/[pid]/status).
  • Driver = a kernel module that registers the device with the relevant subsystem (block, char, net, USB, PCI…).

Processes

FROM LECTURES — Processes (PV065 Ch.6)

A process has:

  • Memory image — text, data, BSS, heap, stack.
  • PID — process identifier.
  • UID / GID — real, effective, saved (set-uid/set-gid programs use these).
  • File descriptors — open files, sockets…
  • Signal mask, working directory, environment…

Process is created by fork() — duplicates the calling process; child gets a new PID, otherwise shares everything (memory via copy-on-write). Then exec() replaces the process image with a new program.

Process termination: exit(); parent waits with wait() / waitpid() and gets the exit status. Otherwise the child becomes a zombie. If the parent dies first, the child is reparented to init (PID 1), which reaps it.

ADDED — process states & scheduling

Linux process states (visible in ps):

  • R — running or runnable.
  • S — interruptible sleep (waiting for an event).
  • D — uninterruptible sleep (usually I/O).
  • Z — zombie (finished, waiting to be reaped).
  • T — stopped (SIGSTOP, debugger).

Schedulers in Linux: CFS (Completely Fair Scheduler), since 6.6 partially replaced by EEVDF. Real-time policies: SCHED_FIFO, SCHED_RR (when guaranteed timing matters).

Files & I/O

FROM LECTURES — Files (PV065 Ch.7-8)

Everything is a file — regular files, directories, devices, sockets, pipes.

File descriptor

  • A small non-negative integer; an index into the process's file descriptor table.
  • Standard: 0 = stdin, 1 = stdout, 2 = stderr.
  • Syscalls: open, read, write, close, lseek, ioctl…

i-node (inode)

  • Per-file metadata structure: owner, group, permissions, size, timestamps (atime, mtime, ctime), direct/indirect block pointers.
  • The directory entry maps a name to an inode number.
  • Hard link — another directory entry pointing to the same inode (same data, ref count incremented).
  • Symbolic link — a tiny file whose content is a path string.

Permissions

  • rwx for user / group / other.
  • Special: SUID (4000), SGID (2000), sticky bit (1000). Sticky on /tmp prevents users from deleting each other's files.
  • ACLs (POSIX ACL) for finer control — getfacl, setfacl. (See Q7 for full coverage.)
FROM LECTURES — Advanced I/O (PV065 Ch.10)
File locking
fcntl advisory locks, flock, mandatory locks (rare).
Scatter-gather I/O
readv / writev — read/write multiple buffers with one syscall.
Memory-mapped files
mmap — file pages become memory; great for large files and IPC.
I/O multiplexing
select, poll — old, O(n) scan. epoll (Linux) / kqueue (BSD) — O(1) events. io_uring — modern Linux async I/O.

IPC, signals, pipes

FROM LECTURES — IPC (PV065 Ch.9 & Ch.12)

Pipes

  • Unnamed pipe — pipe() returns two FDs (read end, write end); usually used between parent and child after fork().
  • Named pipe (FIFO) — mkfifo; file in the filesystem; arbitrary processes can connect.
  • Used by the shell: ls | grep foo | wc -l — three processes connected by pipes.

Signals

  • Async notifications to a process. kill -SIGINT <pid>, signal() / sigaction() to install handlers.
  • Important ones: SIGINT (Ctrl-C), SIGTERM (please stop), SIGKILL (can't be caught, instant kill), SIGHUP (terminal hung up / reload config), SIGSEGV (segfault), SIGCHLD (child changed state).

System V / POSIX IPC

  • Shared memory — multiple processes map the same region; fastest IPC.
  • Message queues — typed messages.
  • Semaphores — synchronization primitives.

Sockets

  • UNIX domain sockets (local) and TCP/UDP sockets (network).
  • BSD sockets API: socket, bind, listen, accept, connect, send, recv.

Threads

FROM LECTURES — Threads (PV065 Ch.11)

Threads share the same address space but have separate stacks. POSIX threads (pthreads): pthread_create, pthread_join, pthread_mutex_*, pthread_cond_*.

Synchronization primitives: mutex, condition variable, read-write lock, barrier. Watch out for race conditions and deadlocks.

Filesystem layout & init (PV077)

FROM LECTURES — FHS & key directories (PV077 Ch.3)
  • / — root; contains everything needed to boot.
  • /bin, /sbin — essential binaries (sbin = administrative).
  • /usr/bin, /usr/sbin — non-essential binaries.
  • /lib, /usr/lib — shared libraries.
  • /etc — configuration.
  • /var — variable data (logs, spool, caches).
  • /tmp — temporary (sticky bit).
  • /home — user home directories.
  • /proc, /sys — virtual: process & kernel state.
  • /dev — device nodes.
  • /boot — kernel, initramfs, bootloader files.
  • /opt, /srv, /mnt, /media — optional packages, served data, mount points.

Common filesystems: ext4 (Linux default), XFS (high-perf, RHEL default), Btrfs & ZFS (modern, with snapshots, COW).

FROM LECTURES — init systems (PV077 Ch.4)
  • SysV init — old standard. /etc/inittab, runlevels 0-6 (0=halt, 1=single-user, 3=multi-user TUI, 5=GUI, 6=reboot). Scripts in /etc/init.d; links from /etc/rcN.d.
  • systemd — modern. PID 1; manages units (service, socket, timer, mount, target). Parallel startup; dependencies declared. Commands: systemctl start/stop/enable/disable/status <unit>; journalctl for logs.
  • Other: upstart (Ubuntu, deprecated), OpenRC (Gentoo), runit, s6.

Users & groups

FROM LECTURES — Users & groups (PV077 Ch.5)
  • UID / GID — kernel's view of user/group identity.
  • Username — used for login and logging.
  • Primary database: /etc/passwd, /etc/group.

/etc/passwd format

ayanami:x:1999:2014:Rei Ayanami:/home/ayanami:/bin/bash
└──────┴┴────┴────┴────────────┴───────────────┴─────────
 user pw  UID  GID  GCOS         home            shell
  • Username — key.
  • Password — encrypted (x means stored in shadow).
  • UID — not necessarily unique (you can have multiple usernames mapping to UID 0).
  • GID — real/effective GID after login.
  • GCOS — comment, usually full name. Some systems use structured fields (room, phone) for finger(1). "GCOS" = General Electric Comprehensive Operating System — Dennis Ritchie said this field was "where to stash info for the $IDENT card. Not elegant."
  • Home directory.
  • Login shell.

Password encryption

  • Originally 25-round DES with 2-char salt. Library function crypt(3).
  • Modern: MD5, SHA-1, SHA-512, bcrypt, Argon2 — all with salt. Format: $6$<salt>$<hash> = SHA-512.

Shadow passwords

  • Standard UNIX passwords were exposed to brute force (John the Ripper) and dictionary attacks (crack(8)).
  • Solution: /etc/shadow — passwords + aging info stored in a root-only file. Requires set-uid/gid on tools.
  • BSD uses /etc/master.passwd.

/etc/group format

nerv:x:2014:ayanami,asuka,shinji
  • Group name — for logging and newgrp(1).
  • Password — usually unused; might be in /etc/gshadow.
  • GID — system identifier.
  • Member list — comma-separated usernames. Primary group is implicit; user need not be listed there.

User management commands

  • vipw(8) — safe editing (locks).
  • useradd, userdel, usermod, groupadd, groupdel, groupmod — batch add/remove; creates home dir, allocates free UID, copies /etc/skel.
  • chfn(8) — change user info (GCOS).
  • passwd — change own password (or any with root).
  • pwconv(8) — convert passwords to shadow.

systemd dynamic users

UID/GID allocation rules: systemd.io/UIDS-GIDS. Declarative creation: /usr/lib/sysusers.d/*.conf, applied by systemd-sysusers(8).

PAM & NSSwitch

Two layers worth knowing:

  • PAM (Pluggable Authentication Modules) — /etc/pam.d/* stacks of modules for auth, account, password, session. Lets you plug in LDAP, Kerberos, two-factor, … without changing login programs.
  • NSSwitch (Name Service Switch) — /etc/nsswitch.conf defines where to look up user names, hostnames, etc. (files / DNS / LDAP / NIS).

Local subsystems

FROM LECTURES — Subsystems (PV077 Ch.8)
  • syslog / journald — system logging. Facilities (auth, cron, daemon, kern, mail…) × severities (emerg, alert, crit, err, warn, notice, info, debug). Config: /etc/rsyslog.conf; journald uses binary journal in /var/log/journal.
  • cron — scheduled jobs. /etc/crontab, /etc/cron.{daily,hourly,weekly,monthly}, per-user crontabs via crontab -e. Modern replacement: systemd timers.
  • Printers — CUPS (Common UNIX Printing System).
  • Disk quotas — per-user / per-group limits on disk usage. quotaon, edquota.

Virtualization & containers

FROM LECTURES — Virtualization (PV077 Ch.9)
  • Full virtualization — KVM (Linux kernel hypervisor), QEMU, VMware, Hyper-V.
  • Paravirtualization — Xen.
  • Containers — LXC, Docker, Podman. Built on kernel features:
    • namespaces — isolate PID, network, mount, UTS, IPC, user.
    • cgroups — resource limits (CPU, memory, I/O).
    • seccomp, capabilities — restrict syscalls and privileges.

UNIX networking (PV077)

FROM LECTURES — TCP/IP (PV077 Ch.10-11)

Covered in detail in Q4 Networks. UNIX-specific bits:

  • BSD sockets API — socket(), bind(), listen(), accept(), connect(), send()/recv(), close().
  • Network config — ip addr, ip route (modern), legacy ifconfig, route. Per-distro config in /etc/network/interfaces (Debian), /etc/sysconfig/network-scripts/ (RHEL), NetworkManager, systemd-networkd.
  • ARP / ND — address resolution. arp -n, ip neigh.
  • Address allocation — static, DHCP (dhclient, NetworkManager), SLAAC (IPv6).
  • Routing — static (ip route add); dynamic via daemons (BIRD, FRR — these speak BGP/OSPF/RIP).
FROM LECTURES — Network services
DNS
Hierarchical name resolution. BIND is the canonical UNIX implementation. Record types: A, AAAA, CNAME, MX, NS, SOA, TXT, PTR (reverse). dig, nslookup, host for queries.
RPC, NFS, NIS/YP
Sun RPC (Remote Procedure Call) with XDR (External Data Representation). portmapper (rpcbind) maps RPC programs to ports. NFS (Network File System) — share filesystem over network. NIS/YP (Network Information Service / Yellow Pages) — centralized user/group/host info; largely superseded by LDAP today.
LDAP
Lightweight Directory Access Protocol. Tree of entries (DNs). Used as centralized user/group store integrating with PAM/NSSwitch. OpenLDAP is the standard implementation. Active Directory is Microsoft's LDAP-based directory.
Kerberos
Network authentication protocol. Tickets instead of passwords on the wire. KDC (Key Distribution Center) issues TGTs and service tickets. Symmetric crypto. Foundation of single sign-on in UNIX/Windows environments.
Email
Message format: RFC 5322 (was 2822, originally 822). Protocols: SMTP (sending, port 25/465/587), POP3 (download, 110/995), IMAP (sync, 143/993). UNIX MTAs: Sendmail, Postfix, Exim.
Firewalls
Linux: netfilter in-kernel, iptables (old userspace tool), nftables (modern replacement, unified rule syntax). firewalld, ufw are higher-level wrappers.

UNIX dev environment (PV065)

FROM LECTURES — Toolchain (PV065 Ch.2-4)
  • Standards — ANSI C, POSIX.1, Single UNIX Specification (SUS). Code following SUS should be portable across UNIXes.
  • C program startup & termination — main(argc, argv, envp); exit(), _exit(), atexit(); argument and environment access.
  • Memory — malloc/free/calloc/realloc; long jumps with setjmp/longjmp.
  • Header files & libraries — static (.a), shared (.so), dynamic linking via ld.so & LD_LIBRARY_PATH.
  • Build tools — gcc, clang, make, cmake, autoconf family.
  • Debugging — gdb (interactive), strace (syscalls), ltrace (lib calls), valgrind (memory errors), perf (profiling).

ADDED — what the exam wants you to know about Windows

The slide outline only lists topics; the exam still expects substance. Here's a precise summary of each:

Windows OS architecture

ADDED — NT architecture

Modern Windows (10, 11, Server) descends from Windows NT (1993). High-level layers:

  1. Hardware
  2. HAL (Hardware Abstraction Layer) — isolates the kernel from CPU/board specifics.
  3. Kernel — scheduling, interrupts, low-level synchronization.
  4. Executive — object manager, memory manager, I/O manager, process manager, security reference monitor.
  5. Subsystems — Win32, POSIX (legacy), WSL (Windows Subsystem for Linux).
  6. User mode — applications + system services (Services, Session Manager, Local Security Authority).

"Everything is an object" — files, processes, threads, registry keys, named pipes — all are NT objects accessed via handles (the Windows analog of UNIX file descriptors).

Basic configuration

ADDED — Windows config surfaces
  • Settings app (modern, touch-friendly).
  • Control Panel (legacy, still present).
  • Registry — hierarchical config DB. Hives:
    • HKEY_LOCAL_MACHINE (HKLM) — machine-wide.
    • HKEY_CURRENT_USER (HKCU) — current user.
    • HKEY_USERS (HKU) — all loaded users.
    • HKEY_CLASSES_ROOT (HKCR) — file associations.
    • HKEY_CURRENT_CONFIG (HKCC) — current hardware profile.
    Edit with regedit.exe or PowerShell.
  • Group Policy (gpedit.msc) — local; in AD environments, centrally managed.
  • Services (services.msc) — background daemons; analogous to UNIX systemd units.
  • Task Scheduler — UI for scheduled tasks (analog of cron).
  • WMI (Windows Management Instrumentation) — introspection API used by management tools.

User management

ADDED — Windows user model
  • SID (Security Identifier) — the actual identity (e.g. S-1-5-21-…-1001); usernames are display labels mapped to SIDs.
  • Local vs domain accounts:
    • Local — defined in the SAM database on the local machine.
    • Domain — defined in Active Directory; same identity across all domain machines.
    • Microsoft account — cloud-managed (Outlook/Live).
  • Built-in groups — Administrators, Users, Guests, Power Users (legacy).
  • UAC (User Account Control) — even admins run with standard-user token by default; elevation requested per-action with the consent prompt.
  • User profiles — per-user directory tree (C:\Users\<name>). Roaming profiles in AD environments. Default profile in C:\Users\Default.
  • Tools: net user, net localgroup (CLI), lusrmgr.msc (GUI), PowerShell cmdlets New-LocalUser, Add-LocalGroupMember.

NTFS file system

ADDED — NTFS features

The default Windows filesystem since NT 3.1 (1993).

  • Journaling — metadata changes written to a log first, replayed after a crash. Same idea as ext3/ext4 journaling.
  • ACLs (discretionary) — per-file/folder. Far more granular than UNIX rwx: separate read/write/execute/delete/take-ownership/change-permission entries per principal. Inherited from parent folder by default. Manage via Properties → Security or icacls.
  • Alternate Data Streams (ADS) — multiple data streams per file (file.txt:hidden.txt). Used by the OS (e.g. Zone.Identifier marks files downloaded from the internet) and historically by malware.
  • Hard links (mklink /H) — multiple directory entries to one MFT record.
  • Junctions / Symlinks (mklink /J, mklink) — redirection points.
  • Sparse files — large files with mostly zero content take minimal disk space.
  • Compression — per-file/folder transparent NTFS compression.
  • EFS (Encrypting File System) — per-file encryption tied to user account.
  • BitLocker — full-volume encryption; uses TPM for key sealing; BitLocker To Go for removable drives.
  • Quotas — per-user storage limits.
  • VSS (Volume Shadow Copy Service) — snapshot mechanism powering "Previous Versions" and many backup tools.

Successor in heavy storage roles: ReFS (Resilient File System) — for Storage Spaces, Hyper-V hosts.

HW devices & drivers

ADDED — Windows drivers
  • Device Manager (devmgmt.msc) — tree of installed devices.
  • Driver models:
    • WDM (Windows Driver Model) — older, kernel-mode.
    • WDF (Windows Driver Framework) — modern; KMDF (kernel-mode) and UMDF (user-mode).
  • Driver signing — Windows requires drivers to be signed (since 64-bit Vista); enforcement can be reduced for testing but not in normal use.
  • PnP (Plug-and-Play) — automatic device detection & driver loading.
  • Drivers via Windows Update — default delivery channel.
  • BSoD (Blue Screen of Death) — kernel-mode driver crashes are the most common cause.

Networking (Windows)

ADDED — Windows networking
  • Network profiles — Public, Private, Domain. Different firewall defaults per profile.
  • Windows Defender Firewall — built-in. Manage via Settings, wf.msc (Advanced), or netsh advfirewall / PowerShell Get-NetFirewallRule.
  • Tools — ipconfig, ping, tracert, netstat, nslookup, Test-NetConnection (PS), Get-NetTCPConnection.
  • Remote access:
    • RDP (Remote Desktop Protocol) — mstsc.exe; port 3389.
    • WinRM / PowerShell Remoting — Enter-PSSession, Invoke-Command over WS-Management.
    • SSH — built into Windows 10+ (both client and OpenSSH server optional feature).

Scripting

ADDED — Windows scripting options
  • CMD / batch (.bat, .cmd) — legacy; tiny syntax, terrible quoting.
  • PowerShell — the modern shell. Cmdlets follow Verb-Noun convention (Get-Process, Stop-Service, New-Item). Pipelines pass objects, not text — fundamentally different from UNIX shells.
  • VBScript / WSH — legacy; on its way out.
  • Python, WSL bash — both fully usable on modern Windows.
EXAMPLE — PowerShell flavor 
# Get top 5 processes by CPU usage
Get-Process | Sort-Object CPU -Descending | Select-Object -First 5

# Create a local user and add to Administrators
$pw = Read-Host -AsSecureString -Prompt "Password"
New-LocalUser -Name "andrej" -Password $pw -FullName "Andrej"
Add-LocalGroupMember -Group "Administrators" -Member "andrej"

# Schedule a daily task (no interactive editor like crontab)
$action  = New-ScheduledTaskAction -Execute "powershell.exe" `
           -Argument "-File C:\scripts\backup.ps1"
$trigger = New-ScheduledTaskTrigger -Daily -At 3am
Register-ScheduledTask -TaskName "NightlyBackup" `
       -Action $action -Trigger $trigger -RunLevel Highest

# Find files larger than 100 MB in C:\Users
Get-ChildItem C:\Users -Recurse -File -ErrorAction SilentlyContinue |
  Where-Object { $_.Length -gt 100MB } |
  Sort-Object Length -Descending |
  Select-Object FullName, @{N='MB';E={[int]($_.Length/1MB)}}

The pipe carries .NET objects; Where-Object filters them by property; Select-Object projects fields. No text parsing required.

Backup & recovery

ADDED — Windows recovery tools
  • File History — per-user file backups to an external drive (versions kept).
  • Windows Backup — system image backup (legacy "Backup and Restore (Windows 7)").
  • System Restore — rolls back system files & registry to a restore point.
  • Recovery Environment (WinRE) — pre-boot menu offering Startup Repair, System Restore, Reset PC, Command Prompt.
  • Reset this PC — reinstall keeping or removing personal files.
  • Volume Shadow Copies — "Previous Versions" tab.

Practical examples

EXAMPLE — minimal C process under UNIX 
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>

int main(void) {
    pid_t pid = fork();
    if (pid < 0) { perror("fork"); exit(1); }
    if (pid == 0) {                          // child
        execlp("ls", "ls", "-la", NULL);     // replaces image
        perror("exec");                      // only reached on error
        _exit(127);
    } else {                                 // parent
        int status;
        waitpid(pid, &status, 0);
        printf("child exited with %d\n", WEXITSTATUS(status));
    }
    return 0;
}

This is fork+exec+wait in 15 lines. Every shell you've ever used does this thousands of times a day.

INTUITION — answering "compare UNIX and Windows"

Three angles examiners like:

  1. Philosophy — UNIX: "everything is a file," small composable tools, text streams. Windows: "everything is an object," large integrated subsystems, object pipelines (PowerShell).
  2. Identity & auth — UNIX has UID/GID + PAM + /etc/passwd. Windows has SID + Local Security Authority + SAM/AD.
  3. Permissions — UNIX has rwx with POSIX ACL extension. NTFS has fine-grained ACEs natively.

Answering the question — point by point

Each phrase from the exam prompt, with a tight answer you could give aloud in 30–60 seconds.

UNIX operating system, kernel operation
UNIX is a multi-user, multitasking OS family (Linux, *BSD, macOS, Solaris) built around the philosophy "everything is a file". The kernel runs in privileged mode, manages CPU, memory, devices, filesystems, and processes, and exposes its services via system calls. Boot sequence: firmware (BIOS/UEFI) → primary bootloader (MBR/EFI) → secondary bootloader (GRUB) → kernel + initrd loaded → kernel init (VM, console, CPU, buses, devices, kernel threads, mount root, exec /sbin/init) → user space. Architectures: monolithic (Linux: all drivers in kernel + dynamic modules), microkernel (Minix, L4: drivers in user space, message passing).
Memory and device management by the kernel
Memory: virtual memory via the MMU + page tables; demand paging; copy-on-write (so fork is cheap); swap to disk under pressure; TLB caches translations; ASLR randomizes addresses for security; mmap maps file pages directly into the address space; OOM killer reaps processes when memory is exhausted. Process address space layout: text → data → BSS → heap (grows up) ↔ stack (grows down). Devices: Linux exposes them in /dev as block (random-access, e.g. /dev/sda) or character (sequential, e.g. /dev/tty). Drivers register as kernel modules (insmod, modprobe), with operations open/read/write/ioctl. Hot-plug is handled by udev in user space, which creates the device node on event. /sys exposes the kernel device tree.
Basic configuration (UNIX)
System config lives under /etc as text files. Service management used to be SysV init (/etc/inittab, runlevels, scripts in /etc/init.d); today it's systemd (units in /etc/systemd/system, parallel startup, dependency graph, commands systemctl start/stop/enable/status, logs via journalctl). Network config varies by distro (/etc/network/interfaces on Debian, NetworkManager / systemd-networkd elsewhere). The Filesystem Hierarchy Standard defines where things live: /bin, /sbin, /usr, /var, /etc, /home, /tmp, /opt, /boot, /proc, /sys, /dev.
User management (UNIX)
Users identified by UID, groups by GID. Primary databases: /etc/passwd (username, UID, GID, GCOS comment, home, shell), /etc/shadow (encrypted password + aging info; root-only), /etc/group (group name, GID, members). Password hashing — historically 25-round DES; today SHA-512 ($6$), bcrypt, Argon2, all with salt. PAM (Pluggable Authentication Modules) lets you plug in LDAP/Kerberos/MFA without changing login programs; NSSwitch picks the source for name lookups (files, DNS, LDAP). Commands: useradd, userdel, usermod, passwd, vipw, chfn. Modern systemd: declarative users in /usr/lib/sysusers.d.
Network services (UNIX)
Built around BSD sockets API (socket / bind / listen / accept / connect / send / recv). Common services:
  • SSH — encrypted remote login (port 22); key-pair auth via ssh-keygen.
  • DNS — hierarchical name resolution; BIND is canonical; record types A/AAAA/CNAME/MX/NS/SOA/TXT/PTR.
  • NFS — Network File System; built on Sun RPC + XDR + portmapper.
  • LDAP — directory protocol for centralized user / group data; integrates with PAM/NSSwitch; replaces NIS.
  • Kerberos — ticket-based network authentication; KDC issues TGTs; SSO across services.
  • Email — SMTP (Postfix, Sendmail), POP3, IMAP.
  • Firewall — netfilter in kernel, controlled by iptables (legacy) or nftables (modern); higher-level firewalld, ufw.
  • Web — nginx, Apache.
Principles of development and development environment in UNIX
POSIX-portable C programming. Standards: ANSI C, POSIX.1, Single UNIX Specification — code that follows SUS is portable across UNIX-like systems. Development toolchain:
  • Compiler — gcc or clang; linker ld; dynamic linker ld.so with LD_LIBRARY_PATH.
  • Build — make, cmake, autoconf.
  • Libraries — static (.a) vs shared (.so).
  • Debugger — gdb; tracing strace (syscalls), ltrace (library calls); profiling perf; memory errors valgrind.
  • Shells — bash/zsh; pipes carry text.
Program lifecycle: main(argc, argv, envp); exit() vs _exit(); atexit(); long jumps via setjmp/longjmp; memory via malloc/free.
Working with processes
A process owns memory, file descriptors, signal mask, working dir, environment, real/effective/saved UID/GID. Created by fork() (duplicates calling process — copy-on-write), then exec() replaces image with a new program. Parent waits with wait()/waitpid(); otherwise child is a zombie. Orphan whose parent died gets reparented to init (PID 1). Lifecycle states (seen in ps): R (run/runnable), S (interruptible sleep), D (uninterruptible — I/O), Z (zombie), T (stopped). Signals (SIGINT, SIGTERM, SIGKILL, SIGHUP, SIGCHLD, SIGSEGV) for async notification — install handlers via sigaction. Threads via pthreads share address space + own stack.
Working with files
"Everything is a file" — regular files, directories, devices, sockets, pipes are all accessed via file descriptors. Each file has an inode storing metadata (owner, group, permissions, size, timestamps, block pointers); directory entries map names to inode numbers. Hard link = another directory entry sharing the inode; symbolic link = tiny file whose content is a path string. Permissions: rwx for user/group/other (numerical: 4/2/1). Special bits: SUID (4000) sets effective user ID to the file's owner on exec — security-critical (sudo's mechanism), SGID (2000), sticky bit (1000) on directories (e.g. /tmp) prevents users from deleting each other's files. POSIX ACLs via getfacl/setfacl for finer control.
Input/output operations
Syscalls: open, read, write, close, lseek, ioctl. Standard file descriptors: 0 stdin, 1 stdout, 2 stderr. Advanced techniques:
  • File locking — fcntl (advisory), flock.
  • Scatter-gather — readv / writev handle multiple buffers in one syscall.
  • Memory mapping — mmap maps file pages into address space; used for large files and shared-memory IPC.
  • I/O multiplexing — select / poll (O(n) scan), epoll (Linux, O(1) events), kqueue (BSD), io_uring (modern async).
IPC mechanisms: pipes (unnamed for parent/child, FIFOs in filesystem), signals, System V/POSIX shared memory + semaphores + message queues, sockets (UNIX domain or TCP/UDP).
MS Windows operating system
NT-based since 1993. Layered architecture: HW → HAL (Hardware Abstraction Layer) → Kernel (scheduling, interrupts) → Executive (object manager, memory, I/O, process, security reference monitor) → Subsystems (Win32, WSL, legacy POSIX) → user mode (apps, system services). Philosophy: "everything is an object" accessed via handles (the Win analog of UNIX file descriptors). Multi-user, preemptive, supports SMP.
Windows basic configuration
Multiple config surfaces:
  • Settings app (modern UI) and legacy Control Panel.
  • Registry — hierarchical config DB; hives HKEY_LOCAL_MACHINE, HKEY_CURRENT_USER, HKEY_USERS, HKEY_CLASSES_ROOT, HKEY_CURRENT_CONFIG. Edited via regedit or PowerShell.
  • Group Policy (gpedit.msc) — local; centrally managed in AD.
  • Services (services.msc) — background daemons (UNIX systemd analog).
  • Task Scheduler — UI for scheduled jobs (cron analog).
  • WMI — Windows Management Instrumentation; introspection API used by management tools.
  • Event Viewer — logging.
Windows user management
Identity = SID (Security Identifier, e.g. S-1-5-21-…-1001); usernames are display labels mapped to SIDs. Account types:
  • Local — stored in the SAM database on the local machine.
  • Domain — stored in Active Directory; same identity across all domain machines.
  • Microsoft account — cloud-managed.
Built-in groups: Administrators, Users, Guests. UAC (User Account Control) — even admins run with a standard-user token by default; elevation prompts per action. User profiles live under C:\Users\<name>; roaming profiles for AD. Tools: net user, net localgroup, lusrmgr.msc, PowerShell New-LocalUser, Add-LocalGroupMember.
NTFS file system
Default Windows filesystem since NT 3.1. Features:
  • Journaling — metadata changes written to a log first; replayed after a crash.
  • ACLs (discretionary) — per-file/folder; far more granular than UNIX rwx (separate read / write / execute / delete / take-ownership / change-permission entries per principal). Inherited from parent. Manage via Properties → Security or icacls.
  • Alternate Data Streams (ADS) — multiple data streams per file (file.txt:Zone.Identifier marks downloads from the internet).
  • Hard links (mklink /H), junctions/symlinks (mklink /J, mklink).
  • Sparse files, compression, quotas.
  • EFS (Encrypting File System) — per-file encryption tied to user account.
  • BitLocker — full-volume encryption using TPM; BitLocker To Go for removable drives.
  • Volume Shadow Copy Service — snapshots ("Previous Versions"), foundation for backup tools.
Successor for heavy storage: ReFS (Resilient File System) — Storage Spaces, Hyper-V hosts.
HW devices and drivers (Windows)
Devices visible in Device Manager (devmgmt.msc) as a tree of installed hardware. Driver models:
  • WDM (Windows Driver Model) — older, kernel-mode.
  • WDF (Windows Driver Framework) — modern; KMDF (kernel-mode) and UMDF (user-mode for less risky drivers).
Driver signing is enforced — since 64-bit Vista, drivers must be signed; enforcement can be reduced for testing but not in normal use. PnP (Plug-and-Play) handles automatic detection + driver loading. Windows Update is the default delivery channel for drivers. BSoD (Blue Screen of Death) is most often caused by buggy kernel-mode drivers.
Basic scripting
Modern Windows automation = PowerShell. Cmdlets follow Verb-Noun (Get-Process, Stop-Service, New-Item, Set-ItemProperty). Pipelines pass .NET objects, not text — fundamentally different from UNIX shells. Filter and project with Where-Object and Select-Object. Examples: list top 5 CPU processes (Get-Process | Sort-Object CPU -Descending | Select-Object -First 5); create scheduled task; remote execution via Invoke-Command / Enter-PSSession over WS-Management. Legacy: CMD / batch (.bat, .cmd) — tiny syntax, terrible quoting. VBScript / WSH — deprecated. WSL brings bash to Windows for cross-tooling.
Practical examples for all of the above
UNIX side: write a daemon in C using fork() + exec(); create a systemd unit; manage users via useradd and AD via SSSD + Kerberos; expose an NFS share from /etc/exports; deploy a TLS-fronted nginx with a Let's Encrypt cert; firewall it with nftables; use strace -p <pid> to see what a hung process is doing; journalctl -u myservice -f for logs. Windows side: install Windows 11 in Hyper-V; create a domain user via New-LocalUser + add to Administrators; lock down NTFS permissions on a shared folder via icacls; enable BitLocker on the system drive with TPM; allow inbound RDP on the Domain profile but block on Public via New-NetFirewallRule; schedule a nightly PowerShell backup via Task Scheduler; export an Event Viewer query for failed logon events.
