Concepts, principles, architectures

FROM REPO
A computer network is a group of computers and devices connected by communication channels, enabling user communication and resource sharing (information, files, data, SW, HW).

An ideal network is transparent (the user doesn't notice they're communicating through a network), with unlimited bandwidth, lossless, no latency, preserving packet order. In reality networks have all these problems and limitations.

Basic network types:

  • Connection-oriented — capacity is allocated/reserved for a communication and not used by anyone else (e.g. old wired telephone). QoS is easy to guarantee; in QoS terminology this is integrated services.
  • Connection-less (packet-switched) — capacity is shared by all communicating parties; messages are split and wrapped into packets. QoS is hard (e.g. the Internet). In QoS terms: differentiated services.

To unify rules and means of communication, we use standardised protocols.

Architectures: peer-to-peer vs client–server (clients only talk to the server; service discovery is via client config; client initiates the connection).

Routing happens via routers' routing tables: based on a packet's address (and possibly type), the table determines the next hop — hop-by-hop principle. The next destination is determined by the longest matching CIDR prefix. Routing tables are usually updated by distributed algorithms. Routers can also filter (stray packets, to enforce QoS) and possibly classify (for pay-as-you-go models). Routing is not always by shortest path — it can consider current load/state.

ISO/OSI and TCP/IP models

ISO/OSI

FROM REPO

Seven layers; each layer responsible for specific functionality; a layer only talks to its neighbours; each layer is an abstraction.

  1. Physical — wire, bits, signals…
  2. Data link — physical addressing (MAC).
  3. Network — logical addressing within the network; routing.
  4. Transport — communication between processes.
  5. Session — user sessions.
  6. Presentation — data representation.
  7. Application — the network application.

In practice TCP/IP, which collapses some OSI layers, became the de-facto standard.

INTUITION — mnemonic

Bottom-up: Please Do Not Throw Sausage Pizza Away (Physical, Data link, Network, Transport, Session, Presentation, Application).

Top-down: All People Seem To Need Data Processing.

Each layer adds a header to data going down (encapsulation) and removes it going up (decapsulation). The mental model: each layer talks to its peer on the other side, oblivious to the layers underneath.

TCP/IP

FROM REPO

Four layers:

Application

  • Provides services to users (web, mail…).
  • Uses application protocols (HTTP, SMTP, DNS, FTP…) which are part of applications; each defines syntax, semantics, message types and exchange rules.
  • Distinctions:
    • peer-to-peer vs client–server
    • pull (client initiates) vs push (server initiates)

Transport — TCP, UDP

  • Takes data, transforms it into segments.
  • Delivers segments to the target application; communication between processes.
  • Addressing via port (16-bit, 0–65535).
  • May provide end-to-end reliability, connection (segments numbered, order matters, delivery acknowledged).
  • May provide connection control, QoS.
  • Logical communication channel; illusion of direct communication.

Network (internet) layer

  • IP (Internet Protocol).
  • Takes segments, transforms them into packets (= datagrams).
  • Transports packets between communicating nodes (across different LANs) — effectively forming a WAN.
  • Addresses every device on the Internet via IP (IPv4 32 bits, IPv6 128 bits).
  • Routes packets — depends on network load and topology.
    • Internet topology is hard to determine and changes dynamically.
    • Each router handles delivery to its nearest neighbours, trying to bring the packet closer to the (presumed) destination, based on its routing table.
    • Routing tables maintained manually (small nets) or automatically via distributed algorithms (more below).

Network access (network interface) layer

Often split into:

  • Data-link
    • Takes packets, transforms them into frames with source/destination addresses (so devices know who's on the medium).
    • Provides addressing via physical/MAC addresses.
    • Ensures reliability of the physical layer (error detection & possible correction via redundancy: parity bit, Hamming code).
    • Flow control.
    • Coordinates access of multiple devices to a shared medium (channels, reservations, randomness, …) — the MAC protocol.
    • Networks can be connected at this layer into topologies (bus, star, ring).
  • Physical
    • Interface as frames of bits.
    • Provides access to the transmission medium.
    • Internally transforms bits into signals; provides synchronization, multiplexing (combining several signals/data streams into one on a shared medium — time/frequency/wavelength-division multiplexing), demultiplexing…
    • Medium: wired/optical cable, air (radio/IR signals…).

IP protocol

FROM REPO
  • Ensures delivery of IP datagrams (data sliced into pieces with an envelope) across the Internet, host-to-host (also through intermediaries, i.e. routers); the network is connection-less, packet-switched.
  • Best-effort service; no delivery guarantee.

More in the IPv4 and IPv6 sections below.

Transport protocols — UDP, TCP

UDP — User Datagram Protocol

FROM REPO
  • Simple; connectionless, best-effort (the app handles reliability if needed).
  • Key feature: simplicity ⇒ minimal overhead, speed.
  • Used for simple request-reply (DNS), real-time transmission (livestream), multicast.

Header contains: source port, destination port, total length, checksum.

TCP — Transmission Control Protocol

FROM REPO
  • Reliable, connection-oriented service; preserves order.
  • Works with byte streams.
  • Connection must be established — 3-way handshake (SYN, SYN+ACK, ACK).
  • Communication is end-to-end recognisable only; routers don't see it as a connection.
  • No multicast.

Header contains: source port, destination port, sequence number (within the stream), ACK number (next expected byte; acknowledges receipt), header length, flags (ACK, RST, FIN…), window size (for flow control), checksum, options.

TCP varies the data sent over the course of a connection so the receiver (Flow Control) or the network (Congestion Control) isn't overwhelmed. The mechanism: window size.

  • Flow Control — the receiver, in its ACK, sends how much room remains in its buffer (e.g. 500 bytes), so the sender knows it can't send more. When the receiver's buffer is full, it advertises window size 0; the sender waits for the next ACK indicating space.
  • Congestion Control — at start, the window grows exponentially until a threshold; then linearly until a packet is lost. On loss, drop to the threshold and continue linearly. Variants differ: Tahoe drops to minimum window after loss; Reno follows the described scheme.
  • The actual cap is the minimum of these two windows.
INTUITION — TCP vs UDP

TCP is a phone call: dial, confirm, talk in order, ensure everything was heard, hang up. UDP is a postcard: write, drop in mailbox, hope it arrives, never know.

Use TCP when correctness matters more than latency (HTTP, file transfer, SQL). Use UDP when latency matters more than completeness (video calls, games, DNS, NTP).

Routing protocols

FROM REPO

Routing tables are maintained manually (small nets) or automatically via distributed algorithms.

Distance Vector
"I tell my neighbours everything I know." Protocol: RIP. Neighbouring routers periodically (or on change) exchange routing tables containing hop distances to various destinations (the distance vector). Principle: Bellman-Ford. Used for small networks without redundancy. Variants: IGRP, EIGRP.
Link State
"I tell everyone about my neighbours." Routers exchange neighbour-state info; the full network topology is kept; each router computes its own routing table. Routes use Dijkstra. Protocol: OSPF (Open Shortest Path First). Metric is cost derived from bandwidth; lower is better. More robust (each computes its own table). Used for large networks. Variant: IS-IS.
Path Vector
Distance vector but instead of just costs, full path descriptions are exchanged. Protocol: BGP (Border Gateway Protocol); allows routing policies; uses CIDR for routing efficiency. Used for routing between autonomous systems.

You can use interior routing (RIP, OSPF) within your domain (autonomous system) and exterior routing (EGP, BGP-4) between domains.

EXAM TIP

Examiners love asking "give me an example protocol for distance vector / link state / path vector." Memorise: RIP → DV, OSPF → LS, BGP → PV. Acronym for protocol families: RIP for small interior nets, OSPF for big interior, BGP between providers.

IPv4 features

FROM REPO

Protocol enabling host-to-host communication.

  • 32 bits, 0.0.0.0–255.255.255.255.
  • Address types:
    • Unicast — 1-to-1.
    • Broadcast — to all on LAN.
    • Multicast — to those subscribed to a multicast address.

Header contains:

  • Version
  • Header length
  • Type of service (for QoS)
  • Total datagram length
  • Identification, flags, offset (used for fragmentation when transport tech can't carry the size)
  • TTL (decremented every hop; dropped at 0; eliminates stray packets)
  • Protocol — specifies the higher-layer protocol (ICMP, IGMP, TCP, UDP, OSPF…)
  • Header checksum (not the body; recalculation would be expensive; recomputed each hop due to TTL change)
  • Source & destination address
  • Options — for testing/debugging; optional (header length field allows this)

IPv4 in general:

  • Works with:
    • ICMP — network status info (e.g. echo request/reply); reports delivery errors to sender (unreachable, TTL exceeded).
    • IGMP — multicast group management; (un)subscribe.
    • ARP, RARP — translate IP to MAC and back.
  • Allows multicast.

Originally IPv4 addresses were split into classes:

Due to address exhaustion, network masks (CIDR) began to be used.

127.0.0.1 is loopback.

Advanced IPv6 features

FROM REPO
  • Solves IPv4 exhaustion with 128-bit length.
  • 0000:0000:0000:0000:0000:0000:0000:0000 – FFFF:FFFF:FFFF:FFFF:FFFF:FFFF:FFFF:FFFF.
  • Shortcuts:
    • Drop leading zeros: 1050:0000:0000:0000:0005:0600:300c:326b ⇒ 1050:0:0:0:5:600:300c:326b.
    • Drop at most one sequence of zeros: 1050::5:600:300c:326b.
  • vs IPv4:
    • IPSec is mandatory.
    • Longer address ⇒ more devices.
    • Simpler header (no checksum, options, fragmentation) with optional extension headers.
    • Flow labeling and priorities.
    • Security, encryption, authentication, integrity built-in.
    • Mobility — every device is "at home" somewhere with a Home Agent. Away from home, the device tells the Home Agent its foreign address; the Home Agent forwards messages or sets up a direct tunnel.
    • Device auto-configuration.
    • Has anycast — like multicast but delivery to just one member suffices.
    • Fragmentation during delivery is removed — the sender must handle it. If too large, the datagram is dropped and an ICMP message indicates max link size (the Path MTU Discovery process — send, get ICMP, fragment, retry).
    • Checksum removed (handled at lower/higher layer).
    • Broadcast replaced by specific multicast groups (e.g. all-routers on LAN).

Header (40 bytes):

  • Version
  • Priority (and a field indicating if delay was observed along the way)
  • Flow label (packets of one flow must be processed the same; routing choice can be cached)
  • Data length
  • Next header — extension (auth, options…) or e.g. TCP header for data
  • Hop limit
  • Source/destination addresses

ICMPv6 extends ICMP's role to also cover what IGMP and ARP did.

  • The original ARP is replaced by Neighbour Discovery Protocol (NDP), which handles:
    • IPv6 address autoconfiguration (no DHCP needed) based on network prefix (from routers) + MAC/locally unique ID.
    • Address collision detection (send to self; if reply, duplicate).
    • Determining MAC addresses on the same link; change tracking.
    • Determining neighbours that can forward.
    • Reachability monitoring.
    • Achieved via multicast groups.

(Sub-)nets defined via CIDR notation (all in the net share the same prefix).

Transition from IPv4 to IPv6

  • Legacy systems must be updated.
  • More complex IPv6 processing.
  • The ideal today is dual-stack applications; alternatively use encapsulation (tunneling IPv6 in IPv4) or NAT.

Peer-to-peer (P2P) networks

FROM REPO

A system is built from many identical (and equal-level) modules — peers — that communicate with each other. Each peer acts as both client (sends requests) and server (responds) at the same time. In P2P you don't need to know the full network topology.

vs Client–Server:

  • Better scalability (though C-S with load balancing does fine too) — capacity grows with users.
  • Decentralisation — good for availability (one peer failing doesn't kill the system).
    • Can be a data-consistency problem.
  • More volatile topology.
  • Compute capacity sharing.
  • Self-organisation.
  • C-S is simpler / more established / easier to manage.
  • In C-S the server is a clear authority for security and data authenticity; harder to ensure in P2P.

Examples:

  • Distributed computing: Folding@Home
  • File sharing: BitTorrent
  • Apache Cassandra (DB)
  • Blockchain (public: cryptocurrencies; private: e.g. Hyperledger)

Layers:

  • Application
  • Middleware — abstracts the overlay, provides access to peer services/resources, access control, locating & maintaining service resource peers.
  • Base overlay — layer atop the physical network (usually TCP/UDP) where peers are treated as hops (physically distant peers may be P2P neighbours and vice versa); peer discovery, message forwarding, network maintenance.

Peer discovery:

  • Static / manual — a preconfigured list of possible peers (IP, port) to try; not suited to dynamic systems; for big systems limit to a few long-running stable peers.
  • Centralised registry of peers — maintains a list of active peers; communication continues directly after discovery. The registry is effectively a server, a single point of failure. Healthcheck-aware (peers can crash silently; on-shutdown deregistration isn't enough).

Service discovery via flooding queries (unstructured nets; each peer answers for its data/services; TTL to prevent overload; can use DFS/BFS/IDS/heuristics/random walks based on most-promising…), or a registry (structured nets; central registry, distributed hash tables, skip lists…), or selected peers as registries (hybrid nets).

The overlay topology (how peers are interconnected) determines the P2P network's overall performance. We avoid linear formations, splits.

  • Random mesh — after discovery, connect to several peers (chosen by latency — favours physically close ones).
  • Layers — peers organize into layers by services/connections; the top layer has the most reliable peers with forwarding capacity; on each layer a peer connects to several peers in adjacent layers and forwards to/from. Tree-like; ensure one failure can't split the net. Useful for video streaming — a peer can duplicate a downward-bound message and save bandwidth higher up.
  • Mesh / Grid — peers in a grid (possibly multi-dimensional; edges can interconnect). Adding/removing peers can be a problem; rows/columns may differ in length. Peer coordinates can encode the services they offer.

Ad-hoc / sensor networks

FROM REPO — Radio networks in general
  • Distributed control over a shared medium.
  • May need to handle eavesdropping.
  • Potentially high error rate, collisions… ⇒ use connection-oriented protocols which may have reservation/scheduling mechanisms for QoS.
  • Transmission signalling (to prevent collisions) needs a separate channel, or must happen before the transmission.
  • Absence of interference at the sender doesn't necessarily mean absence at the receiver.

Ad-hoc

FROM REPO
  • Networks working without prior existing infrastructure, using the network capabilities of individuals ⇒ distributed.
  • Each participant also acts as a host.
  • Unlike P2P:
    • Far greater topology dynamism, participant mobility.
    • Participants are themselves responsible for access to the transmission medium (often air → possibly high error rate, interference).
    ⇒ specialised protocols are needed.

Examples:

  • Communication between autonomous cars.
  • Creating a communication infrastructure in emergency/rescue situations.
  • Tracker tags using the network capabilities of mobile phones could count as mini-ad-hoc.

Routing can be:

  • Address-based — each participant must have a unique address; the problem is finding them in the network. Advantage: supports uni/multi/broadcast.
  • Data-centric — messages don't carry a destination address but a data identifier. Recipients announce interest in data; matching data is forwarded. No need for unique addresses.

Routing further:

  • Proactive (periodic exchange of state, topology info kept) vs reactive (flooding needed); depends on desired speed, load, energy needs…
  • Hop-by-hop vs full topology knowledge.
  • Flat vs hierarchical (superiors may have added responsibilities, use specialised algorithms…).

Sensor networks

FROM REPO

End devices typically contain:

  • Processor and memory.
  • Communication module (radio).
  • Battery (possibly with energy-harvesting HW, e.g. solar).
  • The sensor itself (light, motion…).

Specifics:

  • Emphasis on energy efficiency — battery is small, energy intake limited (shared with ad-hoc).
  • Limited compute/memory capacity.
  • Limited bandwidth.
  • Possible unreliability.
  • Need to solve unique addressing (typically not built in).
  • Possible emphasis on self-organisation, multi-hop within the network.
  • Temporary device sleeping when no activity, to save energy.

Examples:

  • Tracking natural phenomena, behaviour/movement of (endangered) animals.
  • Fire, earthquake, wave detection.
  • Agricultural automation; drought, pest detection…
  • Traffic management.
  • Building HVAC automation.
  • Air-quality measurement.
  • Object security.
ADDED — high-speed networks (the missing subtopic)

The repo doesn't have a dedicated section, but the question mentions high-speed networks. Briefly:

  • Gigabit / 10 / 40 / 100 / 400 GbE — Ethernet's ladder; 400 GbE is mainstream in data centres.
  • InfiniBand — lower-latency than Ethernet; common in HPC clusters.
  • Optical networks — DWDM (Dense Wavelength Division Multiplexing) puts many wavelengths on one fibre; backbone of the Internet.
  • 5G — not just faster mobile; supports low-latency slices for IoT and vehicle networks.
  • RDMA / RoCE — Remote Direct Memory Access over Ethernet; CPU-bypass data transfer used in DBs and ML training clusters.
  • QUIC / HTTP/3 — transport-level high-speed for the public Internet; multi-stream, 0-RTT handshake when resuming.

The recurring theme: reduce per-byte overhead (hardware offload, kernel bypass, fewer round-trips) and parallelise (many wavelengths, many streams).

Computer networks & multimedia

FROM REPO

Multimedia = data composed of different media (text, sound, video, images) integrated together — e.g. videoconferencing (video + sound), TV/stream (video + sound + text/subtitles). Some media are analog and need conversion. Compression is desirable to reduce volume, but adds overhead unacceptable in some real-time transmissions. Depending on the application, errors may or may not matter.

Delay

Time to transmit from source to destination (humans notice latency > 100–200 ms).

  • Processing delay — sender/receiver overhead; depends on system speed/load.
  • Transmission delay — time to push all bits into the medium (depends on size).
  • Propagation delay — medium traversal time; tech + distance.
  • Routing / queuing delay — network load & routing speed.

Jitter — variation in delay across packets.

For minimal overhead and max speed (at the cost of errors and outages) real-time apps use UDP; receivers can estimate missing data; senders can use error-detection/correction via redundancy (or retransmission).

Interleaving — e.g. video: shuffle adjacent frames so they aren't neighbours. A dropped packet affects more parts but only slightly, instead of a noticeable failure in one part. Adds latency.

For real-time transmissions use multicast instead of many unicasts (less congestion). If the network doesn't support multicast, deploy a reflector — the sender sends one stream; the reflector replicates to each recipient via unicast.

Media can be transmitted discretely (file, message) or continuously (stream).

  • Text — HTTP, SMTP, FTP; relatively low bandwidth; delay and error tolerance depend on app. Compression via Huffman / Shannon-Fano (more in Q3 Databases); GZIP (contains Huffman) on the web; small errors are problematic.
  • Audio — basically analog; digitization via sampling (value over time) and quantization (mapping to e.g. integers). Bandwidth depends on quality and compression; small errors tolerable.
  • Graphics — usually no speed requirements (unless tragic).
  • Video — usually bandwidth-hungry; otherwise like audio.

Notes from the repo

FROM REPO
Protocol
A set of syntactic (what messages look like) and semantic (what messages mean) rules for communication between systems/devices/communicators.
CIDR
Besides the IP we keep a netmask (e.g. 147.209.5.0/24 says only the first 24 bits are relevant). Allows subnetting and more efficient routing — if many addresses share a prefix and route, we can say "the prefix goes our way."
Piggybacking
If we want to send data and acknowledge receipt at the same time, combine both into one packet.
NAT
Address translation at the network interface. E.g. one public IP for a whole network; for proper routing back, the interface translates between public and internal addresses.
SSL, TLS
Cryptographic protocols (TLS replaces SSL) between transport and application; ensure authenticity and encryption of data between communicators. First, parties agree on supported algorithms; then symmetric-key exchange happens via asymmetric cryptography (server cert; client cert may be required). After that, symmetric encryption is used for communication.
Multiprotocol Label Switching (MPLS)
For expected/known flows, define paths; assign labels; route by label only (faster; multiple paths possible for backup or load balancing).
Flow control vs congestion control
Flow control is for the receiver; congestion control is for the network.
Authenticity
Data is from the correct sender.
Integrity
Data wasn't changed in transit.
Encryption
Data isn't readable/understandable to third parties.

Practical examples

EXAMPLE — what happens when you type https://example.com in a browser
  1. DNS (UDP/TCP port 53) resolves example.com → IP.
  2. TCP 3-way handshake to that IP, port 443.
  3. TLS handshake negotiates ciphers, exchanges certs, derives a symmetric key.
  4. HTTP GET goes over the encrypted TCP stream.
  5. Server replies with the HTML; browser parses, fires more requests (CSS, JS, images) often multiplexed on HTTP/2 over one TCP connection (or HTTP/3 over QUIC over UDP).

Where each layer lives in this:

  • Application: HTTP, TLS, DNS
  • Transport: TCP (or UDP/QUIC for HTTP/3)
  • Network: IP routing — many hops via BGP-driven decisions between ISPs
  • Data link / Physical: Ethernet or Wi-Fi at each hop
